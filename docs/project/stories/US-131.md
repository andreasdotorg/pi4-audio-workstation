## US-131: Parallel Local-Demo Instances for E2E Testing

**As** the developer running E2E tests,
**I want** multiple local-demo instances to run simultaneously without port
conflicts,
**so that** parallel test sessions and CI jobs don't block each other on
fixed port assignments.

**Status:** BACKLOG
**Priority:** Low
**Related:** US-085 (GM-managed lifecycle — dynamic port allocation),
F-262 (E2E CI failures)

### Context

The current local-demo stack (`scripts/local-demo.sh`) uses fixed ports for
its services (GM 4002, signal-gen 4001, pcm-bridge 9100, web-ui 8080). This
means only one E2E test session can run at a time. If a second instance
starts, it fails on port conflicts.

This becomes easier to solve once US-085 lands — GM managing service
lifecycles enables dynamic port allocation (GM assigns ports at spawn time
and passes them to child processes). Without GM lifecycle management, each
service hardcodes its port.

### Acceptance Criteria

1. Two local-demo instances can run simultaneously on the same machine
   without port conflicts.
2. Each instance uses dynamically allocated ports (no hardcoded port numbers
   in the test runner).
3. E2E test harness discovers service ports from the instance it started
   (not from global defaults).
4. Cleanup of one instance does not affect the other.
5. Documentation in `docs/guide/howto/development.md` updated with parallel
   usage instructions.

### Risks

1. **Depends on US-085:** Dynamic port allocation is significantly easier
   with GM-managed lifecycles. Without it, each service needs its own port
   override mechanism.
2. **PipeWire isolation:** Multiple PipeWire instances on the same machine
   may conflict on the PipeWire socket. May need `PIPEWIRE_REMOTE` or
   separate `XDG_RUNTIME_DIR` per instance.
3. **Resource contention:** Two full local-demo stacks double CPU/memory
   usage. May not be practical on CI runners with limited resources.
