## US-137: Software Version Information in Web UI

**As** the system operator troubleshooting an issue or verifying a deployment,
**I want** to see the software version (git revision and tag) in the web UI,
**so that** I can confirm which build is running and communicate it when
seeking support.

**Status:** draft
**Priority:** Low (operational convenience, diagnostics aid)
**Effort:** 1-2 worker days
**Origin:** Owner directive (session 14)
**Depends on:** none
**Blocks:** none (US-138 SBOM download link is a nice-to-have add-on)
**Cross-references:** US-136 (modular config — may reorganize tab structure)

### Background

The web UI currently has no way to show which software version is running.
When troubleshooting remotely or verifying a deployment, the operator must SSH
in and check git or Nix metadata. A simple version display solves this.

The owner specified this as a "starting point" — more fields may be added
later. The AC is strictly limited to git revision + tag for this story. Future
fields (PipeWire version, kernel version, build date, hardware model, etc.)
are separate stories.

### Acceptance Criteria

**1. Build-time version injection:**
- [ ] AC #1: The git revision (full 40-char SHA) and tag (if present) are
      injected at Nix build time as constants. The flake's `self.rev` (or
      `self.shortRev`) provides the revision. **Dirty tree handling:**
      `self.rev` is `null` when the tree has uncommitted changes — the
      derivation must use `self.rev or "dirty"` / `self.shortRev or "dirty"`.
      The tag is NOT available from the flake — it must be passed from CI as
      a build-time env var (e.g., `GITHUB_REF_NAME`). For local builds,
      default to `null`/`"untagged"`.
- [ ] AC #2: A `version.json` file is generated at build time and included
      in the deployed web UI package. Format:
      ```json
      {
        "git_revision": "cde909a1b2c3d4e5f6...",
        "git_short_rev": "cde909a",
        "git_tag": "v0.4.0",
        "dirty": false
      }
      ```
      If no tag exists, `git_tag` is `null`. If dirty, `dirty` is `true`.

**2. API endpoint:**
- [ ] AC #3: `GET /api/about` returns the version info as JSON. Served from
      the build-time `version.json` — no runtime git operations.

**3. Web UI display:**
- [ ] AC #4: Version info displayed as a section within the System tab (not
      a new top-level tab). Section title: "SOFTWARE". Key-value grid
      matching the existing System tab style (FILTER CHAIN, SCHEDULING,
      MEMORY sections).
- [ ] AC #5: Fields displayed:
      - **Version:** tag if present (e.g., `v0.4.0`), or `dev` in muted
        style if untagged. If dirty, append `(dirty)` in amber.
      - **Git Revision:** short rev (7 chars), displayed as a clickable link
        to `https://github.com/andreasdotorg/mugge/commit/<full-rev>`. Full
        hash in title tooltip.
- [ ] AC #6: The section is extensible — additional key-value pairs can be
      added in future stories without layout changes.

**4. Development / local-demo mode:**
- [ ] AC #7: In development mode (no Nix build), `version.json` is absent.
      The `/api/about` endpoint handles `FileNotFoundError` gracefully and
      returns `git_rev: "dev"`, `git_tag: "local"`, `build_date: null`. The
      web UI shows "dev (local)" instead of erroring. FastAPI startup logs a
      single INFO line: "Running in development mode — no version.json found".
      No placeholder file generation required — the absence of the file IS
      the dev mode signal.

### Definition of Done

- [ ] All acceptance criteria met
- [ ] Architect review: Nix build-time injection, `version.json` generation
- [ ] UX review: section layout, link behavior, dirty indicator
- [ ] QE review: tests for `/api/about` endpoint, development mode fallback
- [ ] No regressions in existing E2E tests

### Technical Notes

- Nix injection: In the web UI derivation, use `writeTextFile` or
  `substituteAll` to generate `version.json` from `self.rev`,
  `self.shortRev`, and tag info. The flake's `self` attribute provides the
  revision. Example Nix pattern:
  ```nix
  pi4audioVersion = {
    rev = self.rev or "dirty";
    shortRev = self.shortRev or "dirty";
    # Tag must come from CI env var (GITHUB_REF_NAME) — flake has no tag awareness
  };
  ```
- The `/api/about` endpoint reads `version.json` once at startup (cached in
  memory). No file I/O on each request.
- If US-136 (modular config) lands first, the SOFTWARE section becomes a
  sub-tab of System (`System > About`). If US-137 lands first, it's a scroll
  section in the current System view — easily promoted to a sub-tab later.
- AE recommended including PipeWire version as static build-time info. This
  is a good future addition but out of scope for this story.

### Security Notes

- The git revision reveals which code is running. An attacker could check for
  known vulnerabilities in that specific commit. This is acceptable for the
  venue threat model (local network, US-110 auth required). The About section
  is behind whatever auth is active (currently none per F-037; will be behind
  US-110 passkey auth when deployed).
