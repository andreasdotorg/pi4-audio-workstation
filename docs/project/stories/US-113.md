## US-113: First-Boot Active Config and FoH Passthrough Profile

**As** the system operator at a venue,
**I want** the system to produce clean audio output immediately after
deployment without any speaker or room configuration,
**so that** I can use front-of-house speakers by simply connecting cables
and powering on, deferring crossover and room correction setup to when I
need my own speakers.

**Status:** deferred/parked (session 6 — owner directive: US-075 is active priority)
**Priority:** High (prerequisite for safe speaker profile activation)
**Effort:** 2-3 worker days (estimate)
**Origin:** Owner directive (session 6) + architect consultation
**Depends on:** US-059 (GraphManager — DONE), US-089 (speaker config management)
**Blocks:** US-012 (E2E room correction automation), venue workflow stories (US-099-104)
**Decisions:** D-062 (first-boot / active config architecture), D-010 (speaker profiles — amended),
D-051 (dynamic filter-chain config — amended), D-053 (profile switching safety — amended)

### Background

D-062 establishes a two-layer config model (speaker profile + venue config)
with symlink-based coefficient activation. This story implements the core
infrastructure: the active config mechanism, the built-in FoH passthrough
profile, and the mute-default safety gate for own-speaker profiles.

### Acceptance Criteria

**1. Active config symlink mechanism:**
- [ ] `/etc/pi4audio/coeffs` is a symlink to the active venue's coefficient
      directory (e.g., `/etc/pi4audio/venues/<venue>/coeffs/`)
- [ ] `active.yml` in `/etc/pi4audio/` declares the active speaker profile
      and venue configuration
- [ ] Activation operation: update `active.yml`, `ln -sfn` coefficients
      symlink, regenerate PW filter-chain `.conf`
- [ ] No file copying — symlink ensures coefficients and config cannot desync
- [ ] Active config persists across reboots (symlink + `active.yml` survive)

**2. Built-in FoH passthrough profile (owner decision: 8ch):**
- [ ] FoH passthrough is a built-in speaker profile (not a separate mode)
- [ ] **8 channels passthrough** from app to output — all USBStreamer output
      channels pass through with Dirac (identity) coefficients
- [ ] Audio passes through unmodified on all 8 channels
- [ ] Fresh deploy defaults to FoH passthrough profile active
- [ ] **Mandatory universal audio gate** — no sound comes out until the
      operator explicitly opens the gate (see AC #3)

**3. Universal audio gate (owner directive: mandatory before ANY sound):**
- [ ] ALL profiles (FoH passthrough AND own-speaker) start muted (`Mult=0.0`)
- [ ] No audio flows until operator explicitly opens the gate
- [ ] Gate opening uses cosine ramp-up (smooth fade-in, no transients)
- [ ] Own-speaker profiles additionally require gain staging confirmation
      before gate can be opened
- [ ] Thermal limiter must be active before gate opens (if US-092 is implemented)
- [ ] UI shows clear indication that audio is gated and how to open it

**4. Three authoritative sources, two derived artifacts:**
- [ ] Speaker profile YAML files stored in `/etc/pi4audio/profiles/`
- [ ] Venue measurement data stored in `/etc/pi4audio/venues/<venue>/`
- [ ] `active.yml` is the single pointer to active profile + venue
- [ ] Derived: `/etc/pi4audio/coeffs` symlink (from active.yml)
- [ ] Derived: PW filter-chain `.conf` (from active profile topology)

**5. File permission model (SEC-062-04):**
- [ ] Document permission model for `/etc/pi4audio/` directory tree
- [ ] Audio service user has read access to profiles, venues, coefficients
- [ ] Write access to `active.yml` and coefficients symlink restricted to
      activation tool only (not audio service user)
- [ ] Symlink target validated via `_SAFE_NAME` + `realpath()` — must
      resolve within `/etc/pi4audio/venues/` (SEC-062-02)

**6. Local-demo alignment:**
- [ ] Local-demo starts with FoH passthrough profile (same as fresh deploy)
- [ ] No Dirac generation at local-demo startup (removed — mock boundary)
- [ ] Local-demo exercises same activation path as production

### Architect Review Notes (session 6, approved)

1. **Dirac coefficients are build-time artifacts.** FoH passthrough Dirac
   impulse WAVs must be generated at build time and shipped with the profile
   — not generated at runtime. This aligns with the mock boundary fix
   (removing Dirac generation from local-demo startup).

2. **Mult defaults are profile-dependent.** The config generator must write
   the correct `Mult` value per profile type into the `.conf` file:
   `Mult=1.0` for FoH passthrough, `Mult=0.0` for own-speaker profiles.
   This is not a runtime decision — it is baked into the generated config.

3. **FoH passthrough channel count — SUPERSEDED by owner decision.**
   Architect originally recommended 4ch production topology. AE recommended
   2ch (subs muted). **Owner decided: 8ch passthrough** from app to output,
   all USBStreamer channels, with mandatory universal audio gate.

### Architect Revision (session 6, post-owner decision)

Based on owner's 8ch + universal gate decision:
- **8ch passthrough**: All USBStreamer output channels pass through with
  Dirac coefficients. No crossover, no channel restriction.
- **Universal audio gate**: ALL profiles start muted (Mult=0.0). Gate
  opening uses **cosine ramp-up** (smooth fade-in avoids transients on
  speakers/amps). This replaces the previous split model where FoH was
  Mult=1.0 and only own-speaker was Mult=0.0.
- **Cosine ramp-up**: `gain(t) = 0.5 * (1 - cos(pi * t / T))` where T
  is ramp duration. Smooth first and second derivatives avoid speaker
  cone excursion transients.

### Audio Engineer Review (session 6, APPROVED)

**Concern RESOLVED by owner decision:** AE raised D-031 risk with 4ch Dirac
on sub channels. Owner decision supersedes both architect (4ch) and AE (2ch):
**8ch passthrough with mandatory universal audio gate.** The gate (Mult=0.0
default, cosine ramp-up) addresses the AE's safety concern — no unprotected
signal reaches any output until the operator explicitly opens the gate.

- Gain staging gate: AE recommends a checklist before unmute:
  1. Speaker profile active and loaded
  2. Crossover confirmed (FIR coefficients generated)
  3. Mandatory HPF validated (D-031 subsonic protection present in coefficients)
  4. Operator confirms target gain values
- Two-layer config model: **Approved, nothing missing.**

### Security Review (session 6, APPROVED — 4 findings, none blocking)

### Risks

1. **Symlink on Pi filesystem** — `/etc/pi4audio/` must be writable by the
   audio service user. NixOS (US-072) may need special handling for mutable
   state in `/etc/`.
2. **Filter-chain reload during activation** — D-053 safety flow applies.
   Profile switch causes ~500ms audio dropout.
3. **FoH passthrough channel count** — RESOLVED by owner decision: 8ch
   passthrough (all USBStreamer channels) with mandatory universal audio gate.
