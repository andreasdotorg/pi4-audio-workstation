## US-132: Network Management

**As** the system operator setting up the audio workstation at a venue,
**I want** the system to manage its own network connectivity — Ethernet,
WiFi client, WiFi AP mode with fallback — configurable via the web UI,
**so that** I can connect the system to any venue network or create my own
network when none is available, without command-line configuration.

**Status:** draft (epic — decompose before implementation)
**Priority:** High (operational necessity for venue deployment)
**Effort:** 14-21 worker days total (4 sub-stories)
**Origin:** Owner directive (session 14)
**Depends on:** US-110 (passkey auth — HARD DEPENDENCY for web UI config),
US-072 (NixOS build)
**Blocks:** US-034 (offline venue operation — US-132 provides the network
infrastructure)
**Cross-references:** A28 (venue network self-sufficiency), US-000a
(firewall), D-017 (withdrawn — replaced by US-034 + US-132)

### Background

Currently the Pi has a simple network config: Ethernet DHCP + nftables
firewall + avahi mDNS. WiFi is unused (brcmfmac blacklisted for boot
cleanliness). There is no WiFi client mode, no AP mode, no web UI for
network configuration.

The owner needs the system to be self-sufficient at venues that may have no
network infrastructure (A28). This requires WiFi AP mode as a fallback and
a web UI for network configuration so the operator doesn't need SSH access
for basic network setup.

### Architecture (from Architect consultation)

**NixOS stack:** `networking.useNetworkd = true` + `services.hostapd` (AP) +
`networking.wireless.iwd.enable = true` (client). NOT NetworkManager
(overkill for headless embedded system).

**Config bridge:** Web UI writes `/var/lib/pi4audio/network-config.json`.
NixOS module reads this at build time. `nixos-rebuild switch` applies
persistent changes. `iwctl` (iwd CLI) handles temporary "try this network"
actions without a rebuild.

**Fallback mechanism:** systemd timer checks wlan0 IP 30s after boot. If no
IP, stops iwd, starts hostapd + DHCP server. Standard systemd primitives.

### Security Requirements (from Security consultation)

1. **Default AP: WPA3-SAE/WPA2-PSK, NEVER open.** Generated password at
   first boot (16+ chars, cryptographically random). Displayed via SSH MOTD
   and `mugge-network show-password` CLI.
2. **Client isolation:** `ap_isolate=1` in hostapd + nftables
   client-to-client DROP. Non-negotiable.
3. **Config injection prevention:** All network config generated from
   validated structs. SSID: `^[a-zA-Z0-9_-]{1,32}$`. Passwords: 8-63
   chars, no control characters. IPs parsed and validated, not strings.
4. **Confirm-and-apply with auto-rollback:** Network changes applied with
   60-second rollback timer. If new config breaks connectivity, auto-revert.
5. **US-110 hard dependency:** All network config endpoints require passkey
   authentication. Unauthenticated network config = trivial DoS.
6. **nftables rewrite:** Interface zones (eth0=trusted, wlan0=untrusted),
   FORWARD chain, NAT for masquerading. Replaces current flat INPUT-only
   model.
7. **WireGuard keys:** mode 0600, never in logs/API. Public key only in UI.
8. **IPv6:** RFC 7217 stable privacy addressing.

### Audio Safety (from AE consultation)

- AP mode is better for RT than client mode (no scanning jitter)
- Network config changes do NOT require audio gate close — but log a warning
  when initiated with gate open
- WiFi scanning (client mode) can cause latency spikes on PREEMPT_RT. Note:
  brcmfmac driver on Pi 4 is the worst offender during active scanning
- Network changes that require reboot trigger USBStreamer transient risk —
  warn before restart actions

### Decomposition (all advisors agree)

This epic MUST be decomposed. Sub-stories:

#### US-132a: Ethernet + WiFi Client Mode (~3-5 days)
- Enable `networking.useNetworkd = true` and `networking.wireless.iwd`
- Unblacklist brcmfmac WiFi driver
- Ethernet: DHCP + IPv6 autoconfig (current behavior, formalized)
- WiFi client: iwd-managed, NixOS declarative config for known networks
- No web UI (SSH/CLI only)
- No AP mode
- **Depends on:** US-072 (NixOS build)

#### US-132b: AP Mode + Fallback (~3-5 days)
- hostapd configuration for WiFi AP mode
- DHCP server (systemd-networkd DHCPServer or kea) on AP interface
- NAT masquerading for AP clients with upstream via Ethernet
- Fallback timer: 30s after boot, if no wlan0 IP -> start AP
- nftables rewrite: interface zones, FORWARD chain, client isolation
- Generated WPA3/WPA2 password at first boot
- **Depends on:** US-132a

#### US-132c: Web UI Network Configuration (~5-8 days)
- Network status card (connection type, IP, signal strength)
- WiFi: manual SSID+password entry (primary), scan with disconnect warning
  (secondary)
- AP fallback toggle with timeout configuration
- Ethernet: DHCP default, static IP behind "Advanced"
- Confirm-and-apply with 60s auto-rollback timer
- Writes `network-config.json`, triggers `nixos-rebuild switch`
- Safety warning banner when in DJ/Live mode
- **Depends on:** US-110 (auth), US-132a, US-132b

#### US-132d: WireGuard VPN (separate story, ~2-3 days)
- WireGuard key generation and peer configuration
- Routing policy for VPN traffic
- Web UI for WireGuard config (behind US-110 auth)
- Private key storage: `/var/lib/pi4audio/wireguard/`, mode 0600
- **Depends on:** US-132a, US-110

### Acceptance Criteria (epic-level)

- [ ] AC #1: **First boot on pristine system:** Ethernet acquires IP via DHCP
      + IPv6 autoconfig. WiFi starts in AP mode with SSID "mugge" and
      generated WPA3/WPA2 password. AP clients get DHCP + IPv6.
- [ ] AC #2: **WiFi client mode:** System can join a venue WiFi network via
      SSID + password. Connection managed by iwd.
- [ ] AC #3: **AP mode:** WiFi runs as access point. Clients get DHCP + IPv6.
      Client isolation enforced. AP clients can reach mugge web UI only.
- [ ] AC #4: **Fallback mechanism:** If no configured network is available
      within 60 seconds, the system activates AP mode with the configured
      fallback SSID and password. Hysteresis prevents flapping (minimum 5
      minutes in AP mode before retrying client networks).
- [ ] AC #5: **Web UI network config:** Accessible from the web UI (behind
      US-110 auth). Shows current network status, allows WiFi configuration,
      AP fallback settings, Ethernet settings.
- [ ] AC #6: **Network changes with auto-rollback:** All network config
      changes applied with a confirmation timer. If new config breaks
      connectivity, auto-revert after 60 seconds.
- [ ] AC #7: **Firewall:** nftables updated with interface zones, FORWARD
      chain for AP routing, NAT masquerading. Default policy: AP clients
      can reach web UI only. Client-to-client traffic blocked.
- [ ] AC #8: **mDNS:** `mugge.local` resolves on all active interfaces
      (Ethernet, WiFi client, WiFi AP).
- [ ] AC #9: **No audio disruption from network transitions.** WiFi mode
      changes may cause brief xruns (documented). No audio gate close
      required for network changes.

### Bonus (owner-specified, lower priority)

- [ ] AC #10: **WireGuard support.** VPN tunnel with web UI configuration.
      Masquerading/routing for AP clients via WireGuard tunnel.
- [ ] AC #11: **AP mode with default route via wired.** When Ethernet is
      connected and AP mode is active, AP clients get Internet access via
      Ethernet uplink (NAT masquerading).

### Known Limitations (from AD challenge)

1. **AP+STA mode is fragile.** The Pi 4's brcmfmac driver supports
   simultaneous AP and client mode, but both share one radio on the SAME
   channel. If venue WiFi is on channel 6, the AP is forced to channel 6.
   DFS channel hopping drops the AP. This is a hardware limitation.
2. **Chicken-and-egg:** Changing WiFi while connected over WiFi can
   disconnect the operator. The auto-rollback timer (AC #6) mitigates this.
3. **WiFi scanning causes RT jitter.** Active scanning on brcmfmac causes
   20-100ms dwell times per channel. Brief xruns possible during scan.
   AP mode (no scanning) is preferred for performance.
