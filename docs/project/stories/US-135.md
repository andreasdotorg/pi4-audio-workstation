## US-135: Room Correction Only Mode

**As** the system operator with full-range speakers or an external crossover,
**I want** to apply room correction FIR filters without crossover filtering,
**so that** I can use mugge purely for room EQ when I don't need the built-in
crossover and speaker management features.

**Status:** draft
**Priority:** Medium (extends usability to a broader set of speaker
configurations)
**Effort:** 2-3 worker days
**Origin:** Owner directive (session 14)
**Depends on:** US-090 (FIR filter generation), US-089 (speaker config
management)
**Blocks:** none
**Cross-references:** D-009 (cut-only correction), D-063 (audio gate),
US-115 (8ch convolver)

### Background

The current system generates combined minimum-phase FIR filters that include
both crossover slopes and room correction in a single filter per output
channel. This is optimal for the owner's multi-way speaker systems but does
not serve users who:
- Have full-range speakers (no crossover needed)
- Use an external active crossover (hardware or another DSP)
- Want mugge only for room EQ

The profile system already supports this via `filter_type: fullrange` — the
`workshop-xrk10f-fullrange.yml` profile demonstrates the pattern. This story
formalizes the "correction only" workflow, fixes a code issue, and adds a web
UI entry point.

### What This Is NOT

This is NOT a "beginner mode." Room correction without crossover is a
legitimate use case for experienced operators with external crossovers. The UI
should present it as a profile type alongside the full crossover+correction
mode, not as a simplified alternative.

### Acceptance Criteria

**1. Profile type support:**
- [ ] AC #1: Speaker profiles with `filter_type: fullrange` produce
      correction-only FIR filters — no highpass/lowpass crossover slopes baked
      into the FIR.
- [ ] AC #2: **Fix default-80Hz fallback.** When `filter_type == "fullrange"`
      and no crossover frequencies are specified,
      `_generate_channel_crossover()` returns a Dirac (identity / no
      filtering) instead of defaulting to 80 Hz crossover. This is currently
      a bug — the XRK-10F profile works around it by other means.

**2. Configurable subsonic protection (default ON, can be disabled):**
- [ ] AC #3: Room correction FIR generation includes a configurable subsonic
      HPF. Default: enabled, 20 Hz, steep (6th order equivalent). The HPF
      frequency is configurable per venue (range: 10-40 Hz).
- [ ] AC #4: The operator can disable the subsonic HPF via the venue config
      (`subsonic_hpf: false`). When disabled, the web UI displays a
      persistent amber warning: "Subsonic protection disabled — room
      correction may boost infrasonic content. Ensure external protection is
      in place." The warning is visible wherever the venue config is
      displayed and cannot be dismissed while HPF is disabled.

**Note:** The AE originally recommended mandatory HPF. The owner overrode
this (decision recorded in `decisions.md`): when the operator uses external
crossovers or full-range speakers, mugge is not responsible for driver
protection. The default-on + warning approach ensures informed consent.

**3. D-009 compliance:**
- [ ] AC #5: Correction-only mode obeys the D-009 cut-only constraint. All
      FIR filter gains are <= -0.5 dB at every frequency. No boost, even for
      room correction. This is the same safety rule as crossover mode.

**4. Audio gate and safety:**
- [ ] AC #6: D-063 universal audio gate applies to correction-only profiles.
      Gate starts closed (Mult=0.0), requires explicit open with cosine
      ramp-up. Same safety flow as crossover profiles.
- [ ] AC #7: Gain staging confirmation is required before opening the gate
      (same as crossover profiles).

**5. Web UI integration:**
- [ ] AC #8: "Room Correction Only" available as a profile template/preset
      in the speaker config page (US-089). Selecting it creates a profile
      with `filter_type: fullrange`, no crossover section, and the default
      subsonic HPF.
- [ ] AC #9: When a `fullrange` profile is active, the Config view hides
      crossover-related sections (crossover frequency, sub delay, driver
      protection LPF). Per-channel gains, subsonic HPF, and room correction
      controls remain visible.
- [ ] AC #10: Profile mode selector clearly labels the two modes:
      "Full System" (crossover + room correction) and "Room Correction Only"
      (room EQ per channel, no crossover). Switching mode does NOT destroy
      existing filter data.

**6. Convolver topology:**
- [ ] AC #11: No convolver topology change required. Unused channels (e.g.,
      sub channels in a stereo fullrange setup) load Dirac at zero gain
      (muted). The 8ch convolver runs regardless — correction-only channels
      load correction FIR instead of crossover+correction FIR.

**7. Documentation:**
- [ ] AC #12: Document the "correction only" profile pattern in
      SETUP-MANUAL.md with a worked example (e.g., stereo full-range
      speakers, correction only, no subs).

### Technical Notes

- The fix is primarily in `generate_profile_filters.py`:
  `_generate_channel_crossover()` returns Dirac when
  `filter_type == "fullrange"` and `crossover_freqs` is empty.
- `_normalize_crossover_freqs()` already returns `[]` when no crossover is
  specified — the issue is the fallback to 80 Hz in
  `_generate_channel_crossover()`.
- Subsonic HPF uses the existing `mandatory_hpf_hz` mechanism from the driver
  identity schema, but the enable/disable control is per-venue
  (`subsonic_hpf: true/false` in venue config). Default: true.
- The XRK-10F profile (`workshop-xrk10f-fullrange.yml`) is the existing
  proof-of-concept. This story generalizes and formalizes it.

### Risks

1. **Room correction pipeline dependency.** The correction FIR generation
   depends on measurement data (US-097, US-098). Without measurements,
   correction-only mode can only produce the subsonic HPF (no room
   correction). This is acceptable — the profile is ready for correction
   when measurements are available.
2. **User expectation of boost.** D-009 limits correction to cut-only. Users
   expecting boost for room dips may be disappointed. Document this as a
   conscious safety trade-off.
