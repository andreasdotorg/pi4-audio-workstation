## US-140: Deterministic Reconciler Settlement

**As** the developer running E2E tests and mode-switching operations,
**I want** the GM `set_mode` RPC to return only after the reconciler has
fully settled the link topology,
**so that** subsequent topology queries reflect the new mode without
sleep-based polling or race conditions.

**Status:** REVIEW (PR #27)
**Priority:** High
**Related:** US-059 (GraphManager core), F-262 (E2E CI failures),
F-263 (rapid mode switch link count drops), F-272 (sleep-based settlement)

### Context

The GM reconciler destroys old-mode links and creates new-mode links in
separate passes. Before this story, callers used `time.sleep()` and polling
loops to wait for the reconciler to finish ‚Äî a pattern prone to races and
flaky test failures (F-262, F-263). Six call sites in the codebase relied
on sleep-based waits.

US-140 adds an epoch-based `await_settled` RPC to the GM. The server tracks
`reconcile_epoch` (incremented each reconciler pass) and `settled_epoch`
(set when `desired == actual` link counts stabilize). Callers pass the epoch
returned by `set_mode` and block until `settled_epoch >= since_epoch` or
timeout, eliminating all sleep-based races.

### Acceptance Criteria

   (`OkWithEpoch` variant).
3. [x] `await_settled` RPC command blocks until `settled_epoch >= since_epoch`

   epoch field.

   `await_settled`.

### Gaps

- **Site 5 (unmute):** Sleep-based wait not yet replaced. Tracked for
  follow-up.
- **Site 6 (web UI JS):** The JavaScript frontend still uses a polling
  approach for mode-switch confirmation. Converting this requires a JS
  `await_settled` client and is deferred to Phase 2.

### Risks

1. **10s timeout:** The `await_settled` RPC uses a 10s timeout. If the
   reconciler genuinely takes longer (e.g., under extreme PipeWire load),
   the call returns an error rather than waiting indefinitely.
2. **Single-threaded epoch access:** Epochs use `Rc<Cell<u64>>` (not
   `Arc<AtomicU64>`) because all access is on the PW main loop thread.
   If the architecture changes to multi-threaded, epochs need atomic access.
