## US-114: Minimal Kernel Config for Audio Workstation

**As** the developer building NixOS SD card images,
**I want** a trimmed kernel defconfig that includes only hardware and
subsystems needed for the Pi 4B audio workstation,
**so that** kernel build time is reduced, the SD card image is smaller,
and the attack surface is minimized.

**Status:** in-progress (IMPLEMENT — code done, heading to TEST)
**Priority:** Medium (owner-directed, session 9)
**Effort:** 1-2 worker days (estimate, plus ongoing maintenance)
**Origin:** Owner observation (session 6) — SD card build compiling DVB
frontends, hwmon, device-mapper VDO, etc.
**Depends on:** US-072 (NixOS build — must have working baseline first)
**Decisions:** D-013 (PREEMPT_RT mandatory)

### Background

During the US-072 SD card image build (session 6), the owner observed the
kernel compiling subsystems that are completely unnecessary for the audio
workstation: DVB frontend drivers, hardware monitoring for absent sensors,
device-mapper VDO, media tuners, etc. A full kernel build takes 3+ hours
on the linux-builder. A trimmed config would reduce this significantly.

### What to include (minimum required)

- PREEMPT_RT scheduler (D-013 mandatory)
- V3D GPU driver (D-022 hardware GL)
- USB audio (snd-usb-audio for USBStreamer)
- USB HID (Hercules DJ controller, APCmini mk2, Nektar SE25)
- Networking (Ethernet, WiFi for venue connectivity)
- SD card / MMC subsystem
- Basic filesystems (ext4, vfat for boot, tmpfs)
- BCM2835 watchdog
- GPIO / I2C (Pi HAT support, future)

### What to strip

- DVB / media tuner frontends
- Unused network drivers (enterprise NICs, InfiniBand, etc.)
- Hardware monitoring for absent hardware (hwmon drivers)
- Device-mapper extras (VDO, RAID, etc.)
- Unused filesystems (XFS, Btrfs, CIFS, NFS client, etc.)
- Bluetooth host controller (Hercules is USB-MIDI only per D-019)
- Unused input drivers (touchscreen, tablet, etc.)
- Sound drivers other than USB audio

### Trade-off

**Pro:** Faster builds (potentially 30-50% reduction), smaller image,
smaller attack surface, less kernel memory usage.

**Con:** Maintenance burden — custom defconfig needs updating with each
kernel upgrade. Risk of missing a required subsystem discovered later.
Mitigation: start from `bcm2711_defconfig` and subtract, not from scratch.

### Acceptance Criteria

- [ ] Custom defconfig derived from `bcm2711_defconfig` with unnecessary
      subsystems disabled
- [ ] Kernel boots on Pi 4B with PREEMPT_RT
- [ ] All required hardware works: USBStreamer, UMIK-1, V3D GL, USB HID
      controllers, Ethernet, WiFi, SD card
- [ ] PipeWire + filter-chain convolver functional
- [ ] Mixxx runs with hardware GL
- [ ] Build time measured and compared to full config
- [ ] Image size measured and compared to full config
- [ ] Defconfig documented: what was removed and why
- [ ] Upgrade procedure documented: how to rebase on new kernel versions

### Risks

1. **Missing subsystem** — a required driver is disabled and only discovered
   at a venue. Mitigation: thorough hardware testing before first gig.
2. **Kernel upgrade friction** — each kernel update requires reviewing the
   defconfig delta. Mitigation: keep changes minimal, document rationale.
