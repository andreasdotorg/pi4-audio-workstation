## US-123: Deterministic Boot State

**As** the system operator powering on the workstation,
**I want** the system to always boot to a known safe state with my last-used venue
recoverable in one tap,
**so that** I can verify setup before enabling audio output, and recover quickly
from crashes or reboots without reconfiguring from scratch.

**Status:** draft
**Priority:** High — safety-critical boot behavior, crash recovery
**Effort:** 1-2 worker days
**Origin:** Owner decision (session 9) + advisory team boot state analysis
**Depends on:** US-113 (first-boot active config — all phases committed), US-059 (GraphManager — DONE)
**Blocks:** US-124 (first-boot experience), US-125 (explicit mode arming)
**Decisions:** D-062 (first-boot / active config), D-063 (8ch convolver + audio gate)

### Background

After a reboot (planned or crash), the system must come up in a deterministic safe
state: no audio output, known mode, known quantum. The operator's last-used venue
configuration is persisted so crash recovery is a one-tap restore, not a full
reconfiguration. The audio gate is NEVER persisted — it always starts closed.

Advisory team input:
- **AE:** USBStreamer DAC transients are hardware-level — the audio gate cannot protect
  against them. "Amps off during boot" must be documented in safety manual.
- **Architect:** No race condition between mode selection and gate opening — GM
  processes RPCs single-threaded.
- **AD:** Gate visibility is high priority — current indicator only in Config tab is
  insufficient (addressed by US-126).
- **UX:** No wizard needed — existing venue + gate flow is the boot workflow.

### Acceptance Criteria

- [ ] **AC #1: GM starts in Standby mode.** NixOS service default changed from `dj`
      to `standby`. GM `initial_mode` config is `standby`. On startup, GM is in
      Standby with no active routing topology.

- [ ] **AC #2: All gain nodes at Mult=0.0 (gate closed) — two independent layers.**
      (a) PW filter-chain `.conf` defaults all gain Mult params to 0.0 — this is
      the static, file-level safety net. (b) GM in-memory gate state starts closed
      (Mult=0.0 applied via `pw-cli` on startup). Both layers must agree: even if
      one fails, the other prevents audio output.

- [ ] **AC #3: Gate state NEVER persisted to disk.** The audio gate is always closed
      on boot regardless of its state before shutdown. No gate state file, no
      "remember last gate state" option. This is a safety invariant — the operator
      must always explicitly open the gate after boot.

- [ ] **AC #4: Last-used venue name persisted.** Lightweight state file (e.g.,
      `/var/lib/pi4audio/last-venue`) written on venue activation via atomic write
      (write to temp file + rename). Backup copy for corruption resilience (e.g.,
      `/var/lib/pi4audio/last-venue.bak` written before overwrite). Contains only
      the venue name string — no gain values, no gate state, no mode.

- [ ] **AC #5: One-tap venue restore on boot.** When persisted venue exists, web UI
      Dashboard shows a prominent "Restore: [venue-name]" button. Tapping it loads
      the venue config (coefficients, gain staging values from venue YAML) but does
      NOT open the gate. Operator must separately open the gate (US-125).

- [ ] **AC #6: Quantum matches initial mode.** GM sets quantum on startup to match
      Standby mode (quantum 256). This fixes F-249 (quantum not set on boot).
      Subsequent mode changes (DJ -> 1024, Live -> 256) update quantum as part of
      mode transition.

- [ ] **AC #7: GM exposes boot state via RPC.** `get_state` RPC response includes:
      `mode` (standby), `gate_open` (false), `venue_loaded` (bool),
      `persisted_venue` (string or null), `quantum` (256). Web UI uses this to
      render the correct boot-time UI (restore button, first-boot banner, etc.).

- [ ] **AC #8: Measurement mode never persists.** If the system was in measurement
      mode before shutdown, it boots to Standby. The persisted state file stores
      only venue name, not mode. Measurement is always a transient session state.

### Risks

1. **NixOS mutable state.** `/var/lib/pi4audio/` must survive NixOS rebuilds. Ensure
   it's declared as persistent state in the NixOS config (not in `/etc/` which is
   regenerated).
2. **Corrupt state file.** Atomic write + backup mitigates partial writes from crash
   during save. If both files are corrupt, fall back to first-boot experience (US-124).
3. **USBStreamer transients on boot.** AE confirmed the audio gate cannot protect
   against DAC hardware transients. Safety manual must document "amps off during boot."
