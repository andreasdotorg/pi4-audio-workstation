## US-133: Web UI Default Port 443

**As** the system operator,
**I want** the web UI to listen on the standard HTTPS port 443,
**so that** I can access it at `https://mugge.local` without specifying a port
number — cleaner URLs, easier to remember, and standard browser behavior.

**Status:** draft
**Priority:** Low (quality-of-life, no functional change)
**Effort:** 1-2 worker days
**Origin:** Owner directive (session 14)
**Depends on:** D-060 (local CA for TLS)
**Blocks:** none
**Cross-references:** US-000a (firewall), US-110 (passkey auth — Relying Party
ID uses hostname, not port; no impact), D-032 (HTTPS)

### Background

The web UI currently listens on port 8080 (HTTPS with self-signed cert). Port
443 is the standard HTTPS port — browsers default to it, so the URL simplifies
from `https://mugge.local:8080` to `https://mugge.local`. This is a small
quality-of-life improvement.

Port 443 requires elevated privileges. The uvicorn process currently runs as a
systemd user service under `ela`. Two approaches were considered:

| Approach | Pros | Cons |
|----------|------|------|
| System service + `AmbientCapabilities=CAP_NET_BIND_SERVICE` | Simple, no new processes, direct TLS | User service → system service migration; needs `XDG_RUNTIME_DIR` for PipeWire access |
| nginx reverse proxy (443 → localhost:8080) | Defense-in-depth, TLS offload, static file serving, connection limits | Additional process, additional NixOS config, adds latency |

**Architecture decision deferred** to implementation time. Architect recommends
system service; Security recommends nginx for defense-in-depth. Both are
viable — the decision will be made when this story is selected for
development, informed by the state of the system at that time.

### Acceptance Criteria

- [ ] AC #1: **Port 443 listening.** The web UI serves HTTPS on port 443 on
      all interfaces. Existing TLS certificate (D-060 local CA or self-signed)
      is used.
- [ ] AC #2: **Port 8080 closed.** Port 8080 is no longer open. No redirect,
      no listener. Clean break — single operator, no fleet migration concern.
- [ ] AC #3: **Firewall update.** `network.nix` updated:
      `allowedTCPPorts = [ 22 443 5900 ]` (replaces 8080 with 443).
- [ ] AC #4: **All existing functionality preserved.** All HTTP endpoints, all
      6 WebSocket endpoints, static file serving, and mock mode continue to
      work identically on the new port.
- [ ] AC #5: **Test infrastructure updated.** All test scripts, E2E harnesses,
      `local-demo.sh`, and any hardcoded port references updated from 8080 to
      443 (or made configurable via environment variable).
- [ ] AC #6: **Documentation updated.** SETUP-MANUAL.md, architecture docs,
      and any references to port 8080 updated to 443.

### Definition of Done

- [ ] All acceptance criteria met
- [ ] Architect review: system service conversion (if chosen) or nginx config
- [ ] Security review: capability or proxy configuration, firewall rules
- [ ] QE review: E2E tests pass on port 443, no regressions
- [ ] No port 8080 references remain in codebase (grep verification)

### Technical Notes

- If system service approach: convert `pi4-audio-webui` from user service to
  system service with `User=ela`, `AmbientCapabilities=CAP_NET_BIND_SERVICE`.
  Set `Environment=XDG_RUNTIME_DIR=/run/user/1000` for PipeWire access.
- If nginx approach: NixOS `services.nginx` with upstream to
  `127.0.0.1:8080`. uvicorn stays as user service, binds localhost only.
- Port 8080 is referenced in: `network.nix`, `web-ui/app/main.py` docstring,
  `web-ui/README.md`, E2E test files, `local-demo.sh`, CLAUDE.md,
  architecture docs, defect docs, memory files, CM role prompt.

### Risks

1. **Test breakage.** Many test files reference port 8080. Shotgun surgery
   change — use grep to find all references.
2. **PipeWire access from system service.** The web UI connects to PipeWire
   via the user session. A system service needs explicit environment variables
   to reach the user's PipeWire instance.
