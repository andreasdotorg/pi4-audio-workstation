## US-129: NixOS VM Smoke Tests in CI

**As** the project team,
**I want** automated NixOS VM boot and service health smoke tests running in CI,
**so that** configuration regressions (broken services, failed activations, missing
dependencies) are caught before they reach the Pi, closing the gap between
unit/integration tests (T1) and the SD card image build (T3).

**Status:** BACKLOG
**Depends on:** US-070 (CI pipeline), CI restructure commit `96cf138` (4-tier pipeline)
**Blocks:** none
**Related:** D-064 (NixOS VM Test Framework — Approach 1: `pkgs.nixosTest` selected)

### Problem

The current CI pipeline has a coverage gap between T1 (unit + integration tests
on x86_64) and T3 (aarch64 SD card image build):

- **T1** tests application code but not NixOS service configuration, systemd
  unit ordering, or module interactions.
- **T3** proves the image builds but does not boot it or verify services start.

A NixOS config change that builds successfully (T0 eval passes, T3 image builds)
but fails at runtime (service crash, missing PATH, wrong group, socket activation
failure) is not caught until manual deployment to the Pi. Examples from this
project: F-020 (PipeWire RT priority), cert service Group=users fix, web UI
PATH for pw-dump/pw-cli, tmpfiles Group=users — all were runtime failures that
built fine.

### Solution

Add a CI tier that boots the NixOS configuration in a VM and verifies that
critical services reach a healthy state. Per D-064, use the NixOS VM test framework (`pkgs.nixosTest`) — declarative
Python test scripts that boot a QEMU VM from the NixOS config and assert
service states, network endpoints, file existence, etc. This is the standard
NixOS upstream approach used by nixpkgs itself. The alternative (booting the
actual SD image in QEMU) was rejected — see D-064 for rationale.

### Acceptance Criteria

**1. VM boot and activation:**
- [ ] CI job boots the NixOS configuration (or a test-profile subset) in a VM
- [ ] VM reaches multi-user.target without failed units
- [ ] Boot completes within a reasonable timeout (configurable, default 120s)

**2. Service health checks:**
- [ ] PipeWire service is active and running at correct RT priority
- [ ] WirePlumber service is active
- [ ] GraphManager service is active and listening on port 4002
- [ ] signal-gen service is active and listening on port 4001
- [ ] pcm-bridge service(s) active and listening on port 9100
- [ ] Web UI service is active, HTTPS on port 8080, `/api/v1/status` returns 200
- [ ] SSL certificate generation service completed successfully

**3. Configuration validation:**
- [ ] No failed systemd units (`systemctl --failed` returns empty)
- [ ] No kernel WARNINGs in dmesg (or only known/expected ones)
- [ ] Firewall rules loaded (nftables active, expected ports open)

**4. CI integration:**
- [ ] Runs as a new tier in the CI pipeline (between T1 and T3, or parallel
  with T3 — placement per ADR recommendation)
- [ ] Gated on T0 + T1 passing (does not run if unit tests fail)
- [ ] Runs on: push to main, PRs targeting main, workflow_dispatch
- [ ] Timeout appropriate for VM boot + checks (suggested: 30 min)
- [ ] Failures produce clear output identifying which service/check failed

**5. Architecture handling:**
- [ ] VM tests run on x86_64 CI runner using the NixOS test framework's
  QEMU-based approach, OR on aarch64 native runner — per ADR recommendation
- [ ] Test assertions are architecture-independent where possible
- [ ] Any aarch64-specific checks (V3D, kernel config) documented as
  "Pi-only, not covered by VM smoke test"

**6. ADR alignment:**
- [ ] Implementation follows the approach selected in the ADR
- [ ] ADR is committed before or with this story's implementation

### Definition of Done

- [ ] VM smoke test implemented and passing in CI
- [ ] At least the 7 service health checks from AC #2 verified
- [ ] CI workflow updated with new job/tier
- [ ] Test failure produces actionable output (which service failed, journal excerpt)
- [ ] ADR committed documenting approach selection rationale
- [ ] QE sign-off on smoke test coverage

### Risks

1. **VM boot time on CI runners:** QEMU boot of a full NixOS system may take
   2-5 minutes on GitHub-hosted runners. The NixOS test framework adds Python
   test driver overhead. Mitigation: timeout tuning, minimal test profile.
2. **x86_64 vs aarch64 gap:** VM tests on x86_64 runners cannot test
   aarch64-specific kernel modules (V3D, BCM2835 watchdog) or hardware
   interactions. Mitigation: document what is and isn't covered; these checks
   remain manual on the Pi.
3. **NixOS test framework complexity:** The testing-python framework has a
   learning curve and its own failure modes (QEMU startup issues, serial
   console hangs). Mitigation: start with minimal checks, expand incrementally.
4. **Closure size:** Building a full NixOS VM closure in CI adds significant
   build time and cache pressure. Mitigation: Cachix binary cache, test-only
   profile that excludes heavy packages (Mesa, Mixxx) if not needed for
   service smoke tests.
5. **Mock audio hardware:** PipeWire and audio services expect audio devices.
   In a VM, there are no real USB audio devices. Mitigation: use PipeWire's
   null/dummy sink or the existing `PI_AUDIO_MOCK` pattern; verify services
   start even without hardware.

### Notes

- This story does NOT replace Gate 2 (deployment to real Pi). VM smoke tests
  catch configuration regressions; hardware integration testing remains manual.
- The 4 runtime failures cited in the Problem section (F-020, cert Group,
  web UI PATH, tmpfiles Group) would all have been caught by AC #2/#3 checks.
- Story scope is intentionally incremental: start with service health checks,
  expand to more sophisticated tests (e.g., GraphManager mode transitions,
  web UI E2E in VM) in future stories.
