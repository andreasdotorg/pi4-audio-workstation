## US-139: Driver Database Quality Model — Raw vs. Curated

**As** the system operator relying on driver data for crossover design,
thermal protection, and room correction,
**I want** the driver database to only ship curated (validated) driver data in
releases, keeping raw (unverified) data as a development/staging resource,
**so that** I can trust every driver entry available in the product — no
unverified data reaches safety-critical calculations like driver protection
limits, crossover points, and thermal ceilings.

**Status:** draft
**Priority:** Medium-High (data quality directly affects safety calculations)
**Effort:** 3-5 worker days
**Origin:** Owner feedback on PR #19 / F-254 (session 14). PR #19 accepted
(tolerance widened to 7% as interim fix). Owner wants structural quality
improvement as the long-term solution — raw vs. curated separation makes
tolerance widening unnecessary for curated drivers.
**Depends on:** US-039 (driver database schema — selected)
**Blocks:** none directly (but improves reliability of US-046, US-092, US-089)
**Cross-references:** F-254 (Qts consistency failures), F-255 (transient
fidelity numerical instability), US-046 (thermal ceiling computation),
US-092 (per-driver thermal protection)

### Background

The driver database (`configs/drivers/*/driver.yml`) currently contains 100+
driver entries sourced from web scrapers (US-040, US-041, US-042) and manual
entry. Many entries have:
- Missing values that could be found in manufacturer spec sheets
- Unverified T/S parameters (transcription errors, rounding, source ambiguity)
- Inconsistencies caught by validation (F-254: Qts deviations of 5.6-6.6%)

The initial approach to F-254 widened the Qts tolerance from 5% to 7% (PR #19,
accepted as interim fix). The owner's long-term direction: **raw data never
ships — only curated drivers are available in releases.** Raw data is a
development/staging resource for identifying candidates that can be promoted
to curated. The "initial curation pass" is not manual data entry — it's
identifying which existing drivers already meet quality criteria and promoting
them. Then figure out how to improve the rest of the data so more drivers
can be promoted over time.

### Acceptance Criteria

**1. Two-tier database structure:**
- [ ] AC #1: Driver entries are classified as either `quality: raw` or
      `quality: curated` in the driver YAML metadata. Default for existing
      entries: `raw`.
- [ ] AC #2: The `quality` field is an enum: `raw` (unverified, scraped or
      manually entered — development/staging only) or `curated` (verified
      against authoritative source, all required fields present, consistency
      checks pass — shipped in releases).

**2. Curated driver requirements:**
- [ ] AC #3: A curated driver entry MUST have:
      - All required T/S parameters present (no nulls): `fs_hz`, `qts`,
        `qes`, `qms`, `vas_liters`, `re_ohm`, `z_nom_ohm`, `sensitivity_db_1w1m`
        or `sensitivity_db_2v83_1m`, `pe_max_watts`, `xmax_mm`, `sd_cm2`,
        `mms_g`, `bl_tm`
      - `ts_parameter_source` field declaring the data source (manufacturer
        datasheet, measured-added-mass, measured-impedance-jig)
      - `datasheet_url` pointing to the manufacturer's published spec sheet
      - Qts consistency: `Qts = (Qes * Qms) / (Qes + Qms)` within 5%
      - Vd consistency: `Vd = Sd * Xmax` within 10% (if Vd provided)
      - EBP consistency: `EBP = Fs / Qes` within 5% (if EBP provided)
- [ ] AC #4: Curated entries MUST pass ALL consistency checks at the 5%
      threshold. The tolerance is NOT widened — if the data doesn't pass,
      it stays `raw` until corrected.

**3. Release gating — raw drivers excluded:**
- [ ] AC #5: The SD card image and release artifacts include ONLY curated
      drivers. Raw driver entries are excluded from the Nix build output.
      The build system filters `configs/drivers/` to include only entries
      with `quality: curated`. Raw data remains in the repo for development
      and staging but is never installed on the Pi.
- [ ] AC #6: A dedicated CI check validates ALL curated drivers meet quality
      standards. This runs as a path-filtered job triggered by changes to
      `configs/drivers/**`. The job:
      - Enumerates all drivers with `quality: curated`
      - Runs full consistency checks (Qts, Vd, EBP)
      - Verifies all required fields are present and non-null
      - Verifies `datasheet_url` is set
      - Fails if ANY curated driver fails validation
- [ ] AC #7: The main test suite (`test-and-build` / T1) validates schema
      compliance for ALL drivers (raw + curated). Consistency checks run
      only on curated entries. Raw entries are validated for schema only
      (they may have incomplete or inconsistent data).

**4. Safety system integration:**
- [ ] AC #8: Since only curated drivers ship in releases, all driver data
      available to safety-critical calculations (thermal ceiling from US-046,
      driver protection from US-092, D-031) is verified. No "unverified data"
      warnings needed in production — every available driver has passed
      curation checks.
- [ ] AC #9: The speaker identity's `sensitivity_db_spl` field (required
      per US-089 AC) uses the driver's sensitivity value. All curated
      drivers have sensitivity present (AC #3 requirement), so speaker
      identity creation never fails due to missing driver data.

**5. Initial curation pass:**
- [ ] AC #10: Run an automated scan of all existing drivers to identify
      which already meet curated quality criteria (all required fields
      present, consistency checks pass within 5%). Report: (a) drivers that
      pass — promote to `curated`, (b) drivers that nearly pass — list what's
      missing or failing, (c) drivers that are far from passing. The owner's
      actively-used drivers (at minimum: the Bose speakers and any drivers in
      pre-configured speaker templates from US-089) are priority candidates
      for manual correction if they don't auto-qualify.
- [ ] AC #11: Document the curation process: which fields to verify, where
      to find authoritative values, how to promote a driver from `raw` to
      `curated`, and guidelines for improving raw data (finding missing
      values, resolving inconsistencies).

**6. F-254 resolution:**
- [ ] AC #12: The two F-254 drivers (morel-tsct1104, prv-audio-6mr500-ndy-4)
      are investigated: verify Qts/Qes/Qms against manufacturer datasheets.
      If data is correct but exceeds 5%, they remain `raw` (the interim 7%
      tolerance from PR #19 accommodates them in the main suite). If data
      has transcription errors, correct and promote to `curated` if
      complete. Curated drivers use the strict 5% threshold.

### Technical Notes

- Schema change: add `quality: raw|curated` to
  `configs/drivers/{id}/driver.yml` metadata section. Default: `raw`.
- Nix build filtering: the driver package derivation filters
  `configs/drivers/` to include only entries where `quality: curated`. This
  can be a simple script that reads each `driver.yml`, checks the `quality`
  field, and copies qualifying entries to the output. Raw drivers stay in
  the repo but never reach `$out`.
- Validation module changes in `src/room-correction/validate_driver.py`:
  - `validate_curated(driver)` — full checks, strict tolerances
  - `validate_raw(driver)` — schema only, no consistency checks
  - `validate_driver(driver)` — dispatches based on `quality` field
- CI job: `.github/workflows/ci.yml` gains a `validate-drivers` job with
  `paths: ['configs/drivers/**']` filter. Runs
  `python -m validate_driver --curated-only`.
- Initial curation scan script: iterate all drivers, run curated validation,
  report pass/near-miss/fail. This is a one-time development tool, not a
  production feature.
- The curated set starts small (auto-qualifying drivers + owner's active
  drivers after manual correction) and grows as data is improved.

### Risks

1. **Small initial curated set.** Most of the 100+ drivers will remain `raw`
   after the initial scan. Only drivers that auto-qualify or are manually
   corrected will ship. This may mean only a handful of drivers are available
   in early releases — this is intentional. Better to ship fewer verified
   drivers than many unverified ones.
2. **Curation effort.** Improving raw drivers to curated quality requires
   finding authoritative datasheet values. The initial pass is automated
   (find auto-qualifying drivers); manual correction focuses on the owner's
   actively-used drivers only. Growing the curated set is ongoing work.
3. **Build complexity.** Filtering drivers at Nix build time adds a build
   step. Keep it simple — a shell script that reads YAML and copies
   qualifying entries.
