## US-127: Runtime Filter Coefficient Switching (D-053)

**As** the system operator,
**I want** to switch between different filter coefficient sets (e.g., FoH passthrough
Dirac vs own-speaker crossover + correction) at runtime without restarting PipeWire,
**so that** venue changes, measurement mode toggling, and boot-time venue restore all
work correctly through a single coordinated mechanism.

**Status:** draft
**Priority:** Critical — infrastructure dependency for venue switching, measurement
design/verify mode, and boot state
**Effort:** 2 worker days
**Origin:** Owner-approved architect investigation (session 9), D-053 safety flow
**Depends on:** US-113 (venue selection infrastructure), D-063 (audio gate)
**Blocks:** US-120 AC #4 (design/verify mode coefficient toggle), US-123 (boot state
with venue restore), US-116 (delay compensation — venue.yml integration)
**Decisions:** D-053 (profile switching safety), C-011 (PW filter-chain cannot
hot-reload coefficients), D-062 (symlink-based coefficient management)

### Background

PipeWire's filter-chain convolver cannot hot-reload coefficient WAV files (C-011).
The only viable approach is destroy-and-recreate: destroy the convolver node, let PW
auto-recreate it (picking up new coefficient files), then GM reconciliation relinks
all connections. This mechanism already works end-to-end via `deploy.py` +
`pw-cli destroy` + GM registry listener, but it is not formalized as a proper GM RPC
with safety coordination.

This story wraps the existing mechanism into a `reload_coefficients` RPC that
coordinates the full D-053 safety flow: close gate, deploy coefficients, destroy
convolver, wait for recreation, reconcile links, verify health, leave gate closed
for operator confirmation.

Architect investigation confirmed:
- **Option A (destroy-and-recreate):** Only viable approach. ~1-2s audio gap.
- **Option B (create-new, relink, destroy-old):** More complex AND worse audio gap.
- **Two permanent instances:** Niche (binary FoH toggle only), doesn't generalize.

### Acceptance Criteria

- [ ] **AC #1: GM `reload_coefficients` RPC command.** New RPC that coordinates the
      full D-053 safety flow:
      (a) Precondition: gate must be closed — GM auto-closes gate (cosine ramp-down)
      if currently open.
      (b) Runs `pw-cli destroy pi4audio-convolver` to destroy the convolver node.
      (c) Waits for PW to auto-recreate the node with new coefficients (registry
      listener detects new node).
      (d) GM reconciliation auto-relinks all connections per current mode's topology.
      (e) Verifies: convolver node healthy, all required links established.
      (f) Returns success/failure to caller with details (node ID, link count, any
      missing links).
      (g) Gate remains closed after reload — operator must explicitly call `open_gate`.

- [ ] **AC #2: Coefficient file deployment.** Before `reload_coefficients` executes
      the destroy step, the caller updates coefficient WAV files at canonical paths
      (`/etc/pi4audio/coeffs/*.wav`). Two mechanisms supported:
      (a) Symlink swap per D-062: update `/etc/pi4audio/coeffs` symlink to point to
      new venue's coefficient directory.
      (b) Direct file overwrite: replace WAV files in place (for measurement mode
      Dirac/correction toggle where no venue change is involved).
      The `reload_coefficients` RPC does NOT manage file deployment — it assumes
      files are already in place when called.

- [ ] **AC #3: `set_venue` enhanced with automatic coefficient reload.** `set_venue`
      RPC gains coefficient-aware logic: if the new venue's coefficient files differ
      from the currently loaded ones (path comparison via symlink target or file hash),
      `set_venue` triggers `reload_coefficients` automatically as part of venue
      activation. If coefficients are unchanged (same venue, only gain changes),
      only gains are applied via `pw-cli` Mult params (fast path, no reload, no
      audio gap).

- [ ] **AC #4: Watchdog explicit unlatch during reload.** The watchdog does NOT
      auto-unlatch when nodes reappear — latch is persistent by design (proven by
      existing test `latch_persists_after_nodes_return`). The `reload_coefficients`
      handler MUST explicitly call `watchdog.unlatch()` after confirming convolver
      node recreation + link reconciliation. Sequence: destroy convolver -> watchdog
      latches -> PW recreates node -> GM verifies node healthy + links established ->
      `watchdog.unlatch()` -> gate remains closed (operator must open explicitly).
      If convolver fails to reappear (timeout), watchdog stays latched and
      `reload_coefficients` returns error. Latch/unlatch events logged.
      Tests required:
      (a) After successful `reload_coefficients`, `watchdog_status` reports
      `latched: false`.
      (b) After successful `reload_coefficients`, `open_gate` succeeds.
      (c) If convolver fails to reappear, watchdog stays latched, reload returns
      error, `open_gate` is rejected.

- [ ] **AC #5: Audio gap budget.** Total transition time: gate close ramp-down
      (~10ms) + coefficient deploy (caller responsibility) + destroy-and-recreate
      (~1-2s) + link reconciliation (~100ms) + verification (~100ms). Gate remains
      closed after reload for operator confirmation. Audio gap is ~1-2 seconds
      during the reload phase only. Acceptable for setup operations (not intended
      for mid-performance switching).

- [ ] **AC #6: Web UI integration.** Existing `POST /api/v1/filters/reload-pw`
      endpoint and `room_correction/deploy.py:reload_convolver()` redirected to call
      GM's `reload_coefficients` RPC instead of running `pw-cli destroy` directly.
      GM is the single coordinator for all coefficient operations — no direct
      `pw-cli destroy` from other components.

- [ ] **AC #7: Built-in FoH passthrough venue.** Create `foh-passthrough.yml` as a
      built-in venue with Dirac identity coefficients and unity gains on all 8
      channels. Ships with the system image. This is the natural "no processing"
      venue for delivering to venue FoH systems. Used as default venue on first boot
      (US-123/US-124).

- [ ] **AC #8: Design/verify mode integration.** US-120 AC #4 (Dirac vs correction
      toggle) uses `reload_coefficients` under the hood: design mode deploys Dirac
      coefficient files then calls `reload_coefficients`; verify mode deploys
      correction coefficient files then calls `reload_coefficients`. Same mechanism,
      different coefficient files. Gate handling per AC #1 (auto-close before,
      remains closed after).

- [ ] **AC #9: Safety compliance.** Gate must be closed before any coefficient
      operation (AC #1a enforces this). No audio passes during the reload window.
      D-009 gain constraints apply after gate re-open. Operator confirmation required
      for any gain elevation after reload (S-012). USBStreamer transient risk
      documented (AE: convolver destruction may produce brief transient — same risk
      as PW restart, mitigated by gate-closed state).

- [ ] **AC #10: Local-demo validation.** Coefficient switching works in local-demo:
      swap from Dirac to room-sim correction coefficients, verify transfer function
      changes via spectrum display. Swap back to Dirac, verify flat response.
      Watchdog does not false-alarm during the reload window. Full cycle completes
      in under 5 seconds.

### Risks

1. **Convolver recreation timeout.** PW may not recreate the convolver node if the
   config is invalid or coefficient files are corrupt. Mitigation: timeout (5s) with
   error return from `reload_coefficients`. Watchdog remains latched (safe state).
2. **Partial reconciliation.** Some links may fail to establish after recreation
   (ports not yet available). Mitigation: retry link establishment with backoff,
   report missing links in RPC response.
3. **Watchdog unlatch fails.** If `reload_coefficients` crashes between convolver
   recreation and `watchdog.unlatch()`, the watchdog stays latched (safe state — no
   audio output). Operator can manually unlatch via watchdog RPC or restart GM.
   Mitigation: ensure unlatch is in the final step of a well-tested sequence.
4. **Concurrent reload requests.** Two `reload_coefficients` calls overlapping could
   race. Mitigation: GM holds a mutex on coefficient operations — second call returns
   "reload in progress" error.
