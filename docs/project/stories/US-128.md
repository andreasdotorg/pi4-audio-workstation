## US-128: Upgrade PipeWire to 1.6.2 (Latest Stable)

**As** the system operator,
**I want** PipeWire upgraded to the latest stable release (1.6.2),
**so that** we benefit from upstream improvements (RT priority inheritance,
control_changed framework, convolver improvements, zeroramp builtin) and can
use the clean US-112 convolver-reload patch without backporting.

**Status:** draft
**Priority:** High
**Effort:** ~2.25 worker days (patch regeneration, verification, F-020 investigation, Pi validation)
**Origin:** Owner directive (session 9, 2026-04-04), architect analysis of 1.4.9->1.7.0 changes
**Depends on:** US-112 (convolver-reload patch -- has clean 1.7.0 version ready)
**Blocks:** US-112 AC #1 (clean patch path), US-127 (coefficient switching needs working convolver reload)

### Background

PipeWire 1.4.9 was inherited from Debian Trixie backports on the original
(non-NixOS) Pi. **nixpkgs (nixos-25.11) already ships PW 1.6.2 for
aarch64-linux.** The NixOS config uses `services.pipewire.enable = true`
which pulls the nixpkgs default -- so the NixOS deployment may already be
running 1.6.2. However, the US-112 convolver-reload patch
(`nix/patches/pipewire-convolver-reload.patch`) was written against the
1.4.9 API and **will fail to apply** against 1.6.2 due to:

- **Filename:** `builtin_plugin.c` renamed to `plugin_builtin.c` (commit `0efd0258a`)
- **API:** `spa_loop_invoke()` migrated to `spa_loop_locked()` (commit `c45d66793`)
- **Struct:** `port[2]` expanded to `port[4]` (latency port added at index 2)
- **`read_closest()`:** gained `def_latency` output parameter (11 params, was 10)
- **`control_changed`:** 1.6.2 already has the framework dispatch (commit `7887c365d`)

The 1.7.0 implementation in `/home/ela/pipewire/` already has all these
changes correct. The primary task is regenerating the patch against the
1.6.2 baseline.

Upstream improvements relevant to this project:

- **RT priority inheritance:** `rlimit.<resource>` config (`9b507d321`) +
  `thread.reset-on-fork` (`1191c1864`) + PI-inherit mutex for data loop
  (`65cbbf1a0`, `cd1d9ceff`). May resolve F-020.
- **control_changed framework:** Clean callback mechanism for builtin
  control ports -- exactly what US-112 needs. Already present in 1.6.2.
- **filter-chain -> filter-graph rename:** Internal SPA plugin directory
  renamed. Module name `libpipewire-module-filter-chain` is UNCHANGED --
  **no config file changes needed.**
- **zeroramp builtin:** Gap smoother/deglitcher, NOT a gain gate (see
  architect assessment below). Not suitable for audio gate replacement.
- **Convolver latency reporting:** Now reports 0 for file IRs (correct for
  minimum-phase FIR). `SPA_AUDIO_MAX_CHANNELS` increased to 128 (no impact
  on 8ch setup).
- **JACK bridge fixes:** Port registration callbacks, timebase,
  `thread.reset-on-fork=false` for JACK clients (inherits RT priority).
  All improvements, not breaking changes.

### Architect Assessment: Zeroramp NOT Suitable for Audio Gate

The zeroramp builtin (`plugin_builtin.c` lines 3021-3115) detects **gaps
of zero samples in the input** and applies cosine fade-out/fade-in around
those gaps. It is a gap smoother/deglitcher with no external trigger
mechanism -- it reacts to audio content, not RPC commands. Our gate needs
explicit open/close RPC with cosine ramp on gain, which the current GM
`pw-cli` Mult approach provides correctly.

**Recommendation:** Keep current GM gate mechanism. Zeroramp could
potentially smooth transients during coefficient reload (separate
enhancement, not US-128 scope).

### Migration Risk Assessment (architect verified)

1. **filter-chain -> filter-graph rename.** Internal SPA plugin directory
   only. The module name `libpipewire-module-filter-chain` is **UNCHANGED**.
   Our `.conf` files and WirePlumber rules do not reference internal SPA
   paths. **No config changes needed.**

2. **Config format backward compatibility.** The filter-chain `.conf` format
   is backward compatible. New features (inline IRs, latency option, noise
   gate, zeroramp) are additive. Our `30-filter-chain-convolver.conf` will
   work unchanged.

3. **JACK bridge behavior.** Several fixes landed (port registration
   callbacks, timebase, `port_type_id`). All improvements, not breaking
   changes. Notable: `thread.reset-on-fork=false` is now default for JACK
   clients (inherits RT priority -- JACK-expected behavior). Mixxx and
   Reaper via `pw-jack` should work the same or better.

4. **Convolver latency reporting.** PW 1.6.2 convolver reports latency
   properly (0 for file IRs, len/2 for hilbert). Our minimum-phase FIR
   filters use file IRs -- reported latency = 0 (correct).

5. **Dependency cascade.** Patched PipeWire hash change cascades through
   ~40 derivations (WirePlumber, openal-soft, gst-plugins-bad, SDL, ffmpeg,
   Mixxx, etc.). Full rebuild required. Build time: ~30-60 minutes on
   remote builder.

6. **US-112 patch API differences (known, solvable).** Current patch uses
   old filename (`builtin_plugin.c`), old API (`spa_loop_invoke`), old
   struct (`port[2]`), old `read_closest` (10 params). The 1.7.0
   implementation already has all changes correct -- regenerate diff against
   1.6.2 baseline.
