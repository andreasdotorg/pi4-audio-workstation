## US-141: Parameterized Deploy and Bootstrap Flake Commands

**As** the system operator deploying NixOS updates or performing clean
installs,
**I want** top-level `nix run` commands that accept the target host as a
parameter,
**so that** I can deploy or bootstrap without remembering the full
nixos-rebuild / nixos-anywhere invocation syntax and without editing
hardcoded IPs.

**Status:** draft
**Priority:** Low (developer convenience, no functional change)
**Effort:** 1 worker day
**Origin:** Owner directive (session 14)
**Depends on:** none (flake already has `mugge-deploy` NixOS configuration)
**Blocks:** none
**Cross-references:** T-072-18 (nixos-anywhere config), T-072-19 (incremental
deploy)

### Background

The flake currently documents deployment as comments with placeholder IPs
(`flake.nix:816-819`):

```
# nixos-anywhere --flake .#mugge-deploy root@<target-host>
# nix run nixpkgs#nixos-rebuild -- switch --flake .#mugge-deploy --target-host ela@<target-host> --sudo
```

The operator must copy-paste and substitute the target host manually. This
story wraps both commands as flake apps that take the target host as a
required CLI argument.

### Two Operations

| Operation | Tool | Use case | SSH user | Destructive? |
|-----------|------|----------|----------|-------------|
| **bootstrap** | nixos-anywhere | Clean-slate NixOS install (wipes disk) | `root@<target>` | YES — erases all data |
| **deploy** | nixos-rebuild switch | Update existing NixOS installation | `ela@<target> --sudo` | No — applies config diff |

### Acceptance Criteria

**1. Deploy command:**
- [ ] AC #1: `nix run .#deploy -- <target-host>` runs
      `nixos-rebuild switch --flake .#mugge-deploy --target-host ela@<target-host> --sudo`.
- [ ] AC #2: Target host is a required positional argument. If omitted, the
      command prints usage and exits non-zero.
- [ ] AC #3: The command passes through any additional arguments after the
      target host to `nixos-rebuild` (e.g.,
      `nix run .#deploy -- mugge --dry-activate`).

**2. Bootstrap command:**
- [ ] AC #4: `nix run .#bootstrap -- <target-host>` runs
      `nixos-anywhere --flake .#mugge-deploy root@<target-host>`.
- [ ] AC #5: Target host is a required positional argument. If omitted, the
      command prints usage and exits non-zero.
- [ ] AC #6: The command prints a clear warning before proceeding:
      "WARNING: This will ERASE ALL DATA on <target-host> and perform a
      clean NixOS install. Press Enter to continue or Ctrl-C to abort."
      The operator must confirm interactively.
- [ ] AC #7: The command passes through any additional arguments after the
      target host to `nixos-anywhere`.

**3. Flake integration:**
- [ ] AC #8: Both commands are defined as flake apps in the `apps` attribute
      set, alongside the existing `test-*` apps.
- [ ] AC #9: The wrapper scripts use `nixos-rebuild` and `nixos-anywhere`
      from the flake's nixpkgs pin (not the system PATH) to ensure
      reproducibility.

**4. Documentation:**
- [ ] AC #10: The comment block above `mugge-deploy` in `flake.nix` is
      updated to reference the new commands instead of the raw invocations.

### Definition of Done

- [ ] All acceptance criteria met
- [ ] Architect review: flake app structure, argument passing
- [ ] Security review: bootstrap confirmation gate (AC #6), SSH user
      selection (root vs ela)
- [ ] QE review: `--dry-activate` passthrough works, missing-argument error
      message is clear

### Technical Notes

- The wrapper scripts are `writeShellScript` derivations in the `apps`
  attrset, same pattern as the existing `test-*` apps.
- `nixos-rebuild` is available in nixpkgs. `nixos-anywhere` may need to be
  added as a flake input or referenced from nixpkgs if available.
- The bootstrap confirmation (AC #6) is a simple `read -p` in the shell
  script. It must be interactive — no `--yes` flag to skip it.
- Argument passthrough: `"$@"` after stripping the first positional arg
  (target host).

### Risks

1. **nixos-anywhere availability.** If `nixos-anywhere` is not in nixpkgs
   or the flake inputs, it needs to be added as an input. Check current
   flake inputs first.
2. **Bootstrap footgun.** The bootstrap command erases the target disk. The
   confirmation gate (AC #6) mitigates this, but the command should never
   be made non-interactive.
