## US-112: PipeWire Filter-Chain Convolver Hot-Reload Patch

**As** the system operator deploying new room correction filters at a venue,
**I want** the PipeWire convolver to reload coefficient WAV files without
destroying and recreating the filter-chain node,
**so that** filter switching is glitch-free — no audio dropout, no broken
links, no GM re-linking required.

**Status:** Draft (deferred — for later consideration)
**Priority:** Low (nice-to-have, current destroy-and-recreate is adequate)
**Effort:** Unknown (PipeWire upstream patch)
**Origin:** Owner directive (session 5, 2026-03-30)
**Depends on:** Nothing (independent PipeWire contribution)
**Blocks:** Nothing (F-221 fix provides adequate interim solution)

### Background

C-011 documents that PipeWire's `libpipewire-module-filter-chain` does
not support runtime modification of convolver coefficient WAV files. The
`config.filename` property is static — read once at node creation. The
only way to swap coefficients today is destroy-and-recreate (`pw-cli
destroy`), which breaks all links and causes a ~1-2 second audio gap.

CamillaDSP (abandoned in D-040) supported glitch-free coefficient
hot-reload via its websocket API. This was one of the implicit tradeoffs
of D-040 that was not identified at the time.

### Proposed Approach

Contribute a patch to PipeWire upstream that adds a runtime-settable
property (or SPA command) to the convolver builtin node, allowing the
`filename` to be updated without destroying the node:

1. **New control parameter** on the convolver builtin: `filename` exposed
   as a settable `SPA_PARAM_Props` parameter (similar to how `linear`
   builtins expose `Mult` and `Add`)
2. **On filename change:** The convolver reads the new WAV file, computes
   the new FFT partitions, and crossfades from old to new coefficients
   over a short window (~10ms) to avoid clicks
3. **Links preserved:** The node stays alive, ports stay alive, all PW
   links remain connected. Zero disruption to the audio graph.
4. **API:** `pw-cli s <node-id> Props '{ params = [ "convolver:filename" "/path/to/new.wav" ] }'`

### Alternative Approaches

- **SPA command** (`pw-cli send-command`): Less invasive than Props
  modification but requires a new command type in the filter-chain module
- **File watch** (inotify on the WAV file): Convolver watches its
  coefficient file and auto-reloads on change. Simplest but least
  controllable (no crossfade timing, race conditions with file writes)
- **Module-level reload signal**: The entire filter-chain module re-reads
  its config on SIGHUP or a custom PW event. Heavier than per-node
  reload but simpler to implement.

### Technical Notes

- PipeWire source: `spa/plugins/filter-graph/` (convolver implementation)
- The convolver uses FFT-based partitioned convolution (overlap-save).
  Hot-reload requires computing new FFT partitions from the new WAV file,
  then swapping the partition buffers. Double-buffering would allow
  glitch-free transition.
- The `linear` builtin already demonstrates runtime Props on filter-chain
  nodes — the convolver would follow the same pattern.
- Upstream acceptance likelihood: moderate. PipeWire maintainers have been
  receptive to well-tested contributions. The use case (room correction
  filter switching) is common in pro-audio.
