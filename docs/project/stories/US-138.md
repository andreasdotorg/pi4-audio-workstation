## US-138: CycloneDX SBOM as Release Artifact

**As** the system operator (and open-source project maintainer),
**I want** a CycloneDX Software Bill of Materials generated with each release,
**so that** I can audit the software supply chain, scan for known
vulnerabilities, and provide transparency about what's in the SD card image.

**Status:** draft
**Priority:** Low (best practice, no functional impact)
**Effort:** 2-4 worker days
**Origin:** Owner directive (session 14)
**Depends on:** Release workflow (`.github/workflows/release.yml` — already
exists)
**Blocks:** none
**Cross-references:** US-137 (About tab — optional SBOM download link)

### Background

A Software Bill of Materials (SBOM) lists every software component included in
a build. CycloneDX is one of two major SBOM formats (the other being SPDX).
SBOMs enable:
- Vulnerability scanning (feed SBOM into `grype`, `trivy`, or Dependabot)
- License compliance auditing
- Supply chain transparency

The release workflow already builds an SD card image and creates a GitHub
release. This story adds SBOM generation as an additional release artifact.

### Known Limitations (from advisory review)

1. **NixOS SBOM tooling is immature.** Dedicated tools (`cyclonedx-nix`,
   `nix-sbom`) have limited maintenance. A custom script transforming
   `nix path-info` output to CycloneDX may be more reliable.
2. **Sub-dependency visibility.** Nix-level SBOM shows top-level packages
   (e.g., `graph-manager-0.1.0`) but NOT individual Rust crate dependencies
   (50+ per binary) or Python package dependencies. For complete Rust/Python
   SBOMs, separate `cargo cyclonedx` / `pip-audit` steps would be needed at
   build time. This is out of scope for this story — document the limitation.
3. **Post-imaging drift.** The SBOM reflects the SD card image at build time.
   Any packages installed on the Pi after imaging are not tracked. This is
   industry standard — even commercial SBOMs have this caveat.
4. **The SBOM is public.** The GitHub repo is public, so release artifacts
   (including the SBOM) are publicly accessible. The SBOM reveals every
   package and version in the NixOS closure. This is acceptable — it's the
   same information available by inspecting the Nix flake.

### Acceptance Criteria

**1. SBOM generation:**
- [ ] AC #1: A CycloneDX JSON SBOM is generated from the full NixOS SD card
      image closure during the release workflow. The SBOM lists all packages
      in the runtime closure with: package name, version, Nix store path hash.
      Scope: the complete SD card image artifact — everything that ships on
      the Pi.
- [ ] AC #2: The SBOM is generated after the image build step and before the
      release creation step in `.github/workflows/release.yml`.
- [ ] AC #3: CycloneDX JSON format (version 1.5 or 1.6). JSON preferred over
      XML for modern tooling compatibility.

**2. Release artifact:**
- [ ] AC #4: The SBOM file is uploaded as a release artifact alongside the
      `.img.zst` image file. Filename: `mugge-<tag>-sbom.cdx.json`.
- [ ] AC #5: The release notes include the SBOM filename and a brief
      description: "Software Bill of Materials (CycloneDX) — list of all
      packages in the SD card image."

**3. Non-blocking:**
- [ ] AC #6: SBOM generation failure does NOT block the release. If the SBOM
      step fails, the release proceeds with a warning annotation but without
      the SBOM artifact. Rationale: SBOM is metadata, not a functional
      requirement. Immature tooling should not block deployments.

**4. Documented limitations:**
- [ ] AC #7: The SBOM includes a note (in CycloneDX metadata or a companion
      README) documenting: (a) Rust crate and Python package sub-dependencies
      are not individually listed — only top-level Nix packages appear,
      (b) post-imaging modifications are not tracked, (c) the SBOM covers
      the NixOS runtime closure only (not build-time-only dependencies).

**5. Validation:**
- [ ] AC #8: The generated SBOM passes `cyclonedx validate` (CycloneDX CLI
      schema validation). This ensures the file is well-formed and consumable
      by standard tools.

**6. Vulnerability scanning (Security recommendation):**
- [ ] AC #9: The release workflow runs
      `nix run nixpkgs#grype -- sbom:sbom.json --fail-on critical` after SBOM
      generation. **Initial rollout:** non-blocking (`continue-on-error: true`)
      for the first few releases while a `grype.yaml` ignore file is built up
      (Nix packages often have different vulnerability status than upstream due
      to independent security patches — false positives are expected). Promote
      to blocking once the ignore list stabilizes. Medium/low findings are
      always logged as warnings but do not block.
- [ ] AC #10: (Should-have) Cosign signature on the SBOM artifact for tamper
      detection. If easy to add, include; if it delays the story, defer to
      a follow-up.

### Technical Notes

- **Scope:** Full NixOS SD card image closure — everything on the Pi.
- **Nix closure enumeration (Architect-confirmed command):**
  ```bash
  nix path-info --recursive --json \
    .#nixosConfigurations.mugge.config.system.build.toplevel
  ```
  This gives every store path in the NixOS system closure. Each JSON entry
  includes the derivation name encoding `pname-version`. Parsing `pname` and
  `version` from store path names is reliable for nixpkgs packages (they
  follow the `name-version` convention).
- **Approach A (preferred if tooling is stable):** Use `nix-sbom` to generate
  CycloneDX directly from the system toplevel closure.
- **Approach B (fallback):** Custom script: `nix path-info --recursive --json`
  on system toplevel → parse pname/version → transform to CycloneDX format.
  More work but no external tool dependency.
- **Approach C (Rust/Python enrichment, future):** Run `cargo cyclonedx` at
  Rust build time, `pip-audit --format cyclonedx` for Python deps, merge
  into the Nix-level SBOM. Out of scope for this story.
- The SBOM step runs on the same `ubuntu-24.04-arm` runner as the image
  build, so the Nix store with the built closure is available.
- SBOM signing (cosign/sigstore) is a nice-to-have for a future story.
  Not required for MVP.

### Future Enhancements (out of scope)

- Application-packages-only SBOM (focused view of project-specific packages)
- Weekly cron scan of latest release SBOM for newly disclosed CVEs
- Rust crate-level and Python package-level SBOM enrichment
- SBOM download link in web UI About section (US-137)
- SPDX format as alternative output

### Risks

1. **Tooling instability.** NixOS SBOM tools are young. AC #6 (non-blocking)
   mitigates this — a broken tool doesn't break releases.
2. **Closure size.** The full NixOS system closure has 1000+ packages. The
   SBOM file could be large (several MB). This is normal for full-system
   SBOMs and acceptable as a release artifact.
3. **Grype false positives.** Nix packages often have different vulnerability
   status than upstream because nixpkgs applies security patches independently.
   Grype's CPE matching will flag vulnerabilities that nixpkgs has already
   patched. A `grype.yaml` ignore file is needed from day one (AC #9 starts
   non-blocking for this reason). With full-closure scope, expect a HIGH
   initial false positive rate.
4. **Package identity accuracy.** Nix store paths encode `pname-version` but
   not CPE identifiers. SBOM tooling infers CPEs from package names — works
   well for common packages (openssl, glibc) but may miss or mismatch
   obscure ones. The SBOM is accurate for inventory; imperfect for automated
   vulnerability assessment.
5. **Reproducibility.** Closure contents change with every nixpkgs pin
   update, even if project code doesn't change. The SBOM is tied to a
   specific release tag — accurate for that build, not "the project in
   general." This is correct behavior.
6. **Build vs runtime leakage.** `nix path-info --recursive` on the system
   profile may include some build-time dependencies not loaded at runtime.
   AC #7 documents this limitation.
