## US-134: Hardware I/O Configuration

**As** the system operator using different audio interfaces at different
venues,
**I want** to discover connected audio devices, assign their channels to
the system's logical I/O roles, and have the entire signal chain adapt,
**so that** I can use mugge with any USB audio interface — not just the
minidsp USBStreamer.

**Status:** draft (epic — decompose before implementation)
**Priority:** Medium-High (extends mugge to arbitrary hardware)
**Effort:** 18-27 worker days total (5 sub-stories)
**Origin:** Owner directive (session 14)
**Depends on:** US-115 (8ch convolver — current baseline), US-089 (speaker
config — channel assignment in profiles)
**Blocks:** none directly (but enables broader hardware support)
**Cross-references:** D-054 (configurable channel assignment per profile),
US-113 (venue config), US-136 (UI reorganization)

### Background

The entire system currently assumes a fixed hardware configuration:
- minidsp USBStreamer B: 8-in/8-out via ADAT
- Fixed channel assignment: ch 1-2 mains, ch 3-4 subs, ch 5-6 engineer HP,
  ch 7-8 singer IEM
- Hardcoded in: `CHANNEL_NAMES`, `GAIN_PARAM_NAMES`, filter-chain convolver
  config, GraphManager reconciler, pcm-bridge, level-bridge, venue configs,
  speaker profiles

Making this configurable is the single most architecturally impactful change
in the project. It touches every layer of the stack.

### Architecture (from Architect consultation)

**New config layer: Device Profile.** Sits between hardware discovery and
speaker profile. Maps physical device ports to logical channel numbers.

```
Hardware Discovery    Device Profile    Speaker Profile    Venue Config
(pw-dump enumeration) (logical mapping)  (crossover/FIR)    (gains/delays)
         |                  |                  |                 |
   USBStreamer ch 1  ->  Main L (logical 1)  -> HPF+correction -> venue gain
   USBStreamer ch 2  ->  Main R (logical 2)  -> HPF+correction -> venue gain
   ...
```

Stored in `/var/lib/pi4audio/device-profile.json` (runtime) with a default
in `configs/devices/usb-streamer-8ch.json`.

**Convolver adaptation:** The filter-chain `.conf` is generated from device
profile + speaker profile. Convolver node count and channel mapping become
parameters, not constants.

### Safety Requirements (from AE consultation, revised per owner feedback)

Safety enforcement is bound to the **profile type**, not applied as blanket
per-channel-role rules. mugge enforces driver protection only when mugge is
responsible for the signal chain that could cause the damage.

| Profile type | mugge's responsibility | Safety enforcement |
|---|---|---|
| `crossover` (2-way + sub, default) | Frequency splitting + room correction | D-031 HPF on mains, LPF on subs — mandatory, enforced by filter generation pipeline |
| `fullrange` (correction only) | Room correction only, no crossover | Subsonic HPF offered (default ON, operator can disable — see US-135) |
| `passthrough` (FOH, US-113) | Unity gain passthrough | No filter enforcement — operator's external system handles protection |

**Universal rules (all profile types):**

| Rule | Rationale |
|------|-----------|
| D-009 gain cap (Mult <= 1.0) | Digital clipping prevention — not speaker protection |
| D-063 gate (all gains start at 0.0, cosine ramp-up) | Startup safety |
| Channel assignment changes require gate close + reopen cycle | Prevent mid-performance routing changes |
| Changing the I/O device (not just channel map) requires a full venue reload | New device = new channel properties, old FIR filters may not be valid |
| **Known safe default** for USBStreamer MUST exist as a named preset | Prevents misconfiguration on standard hardware |
| Per-channel test tone for I/O verification | Prevents miscabling |

**Per-channel test tone:** Device discovery UI must include an "Identify"
button that plays a short beep on a specific channel for physical wiring
verification.

### Decomposition (all advisors agree — ESSENTIAL)

This epic MUST be decomposed into 5 sub-stories:

#### US-134a: Device Discovery + Enumeration UI (~2-3 days)
- Read-only: enumerate PipeWire audio devices in web UI via `pw-dump`
- Display: device name, channel count, sample rate, connected/disconnected
- "Identify" button: play test tone on a specific channel
- Whitelist devices by node name prefix (e.g., `pi4audio-*`, `USBStreamer`,
  `ADA8200`) — don't expose all PipeWire nodes
- No config changes — discovery only
- **Depends on:** none

#### US-134b: Device Profile Data Model (~2-3 days)
- Define JSON schema for device-to-logical channel mapping
- Default profile for USBStreamer 8ch (current hardcoded assignment)
- Persistence in `/var/lib/pi4audio/device-profile.json`
- API endpoint: `GET/PUT /api/device-profile`
- Validation: channel count vs device capabilities, role uniqueness
- No topology changes — just the data model
- **Depends on:** US-134a

#### US-134c: Dynamic Convolver Config Generation (~5-8 days)
- Generate filter-chain `.conf` from device profile + speaker profile
- Variable convolver node count (2, 4, 6, or 8 channels)
- Variable gain node count matching convolver channels
- Handle mismatched channel counts (device has fewer channels than profile
  expects): error with clear message, not silent degradation
- **This is the critical path and the hardest sub-story.**
- **Depends on:** US-134b

#### US-134d: Channel Assignment UI (~3-5 days)
- Two-column logical-to-physical mapping table (UX recommendation)
- Left column: logical roles (Main L, Main R, Sub 1, Sub 2, Eng HP L/R,
  IEM L/R) — fixed, defined by system
- Right column: dropdown per row selecting physical device + channel
- Staged changes: pending state shown, Apply/Revert buttons
- Inline safety warnings for dangerous assignments (sub -> headphone)
  — non-blocking but impossible to miss, require explicit confirmation
  for cross-boundary assignments
- **Depends on:** US-134a, US-134b

#### US-134e: Dynamic GraphManager + Watchdog (~5-8 days)
- GraphManager link topology adapts to channel count from device profile
- Watchdog gain monitoring adapts to variable gain node count
- pcm-bridge and level-bridge variable channel counts
- `CHANNEL_NAMES` and `GAIN_PARAM_NAMES` become dynamic (loaded from
  device profile)
- **Depends on:** US-134b, US-134c

### Acceptance Criteria (epic-level)

- [ ] AC #1: **Device discovery.** Web UI shows all connected USB audio
      devices with name, channel count, sample rate, and connection status.
- [ ] AC #2: **Channel assignment.** Operator can assign physical device
      channels to logical I/O roles via the web UI. Assignment is persisted
      in the device profile.
- [ ] AC #3: **Known safe default.** The USBStreamer 8ch assignment
      (ch 1=Main L, 2=Main R, 3=Sub1, 4=Sub2, 5=Eng HP L, 6=Eng HP R,
      7=IEM L, 8=IEM R) is a named preset that can be restored at any time.
- [ ] AC #4: **Dynamic signal chain.** The filter-chain convolver,
      GraphManager link topology, gain nodes, and watchdog all adapt to the
      configured channel count and assignment.
- [ ] AC #5: **Profile-type-bound safety.** When profile type is `crossover`,
      the system validates that sub-assigned channels have LPF protection and
      main-assigned channels have HPF protection in their FIR chain (missing
      protection is a blocking error). When profile type is `fullrange` or
      `passthrough`, no crossover-derived filter validation is performed.
      D-009 gain cap and D-063 gate apply to all profile types.
- [ ] AC #6: **Channel identification.** "Identify" button per channel plays
      a test tone for physical wiring verification.
- [ ] AC #7: **Gate interaction.** Channel assignment changes require audio
      gate close. Device changes require full venue reload.
- [ ] AC #8: **Variable channel count.** The system supports 2, 4, 6, or 8
      output channels depending on the connected device and profile.

### Known Limitations (from AD challenge)

1. **Multi-device clock sync.** Two USB audio devices on Pi 4 have
   independent clocks with 1-50 ppm drift. PipeWire's adaptive resampling
   handles this but adds latency and CPU load. Multi-device support may
   not be feasible on Pi 4B's limited CPU budget.
2. **Hot-plug.** If the configured audio device is disconnected during a
   performance, the system must degrade gracefully (mute, gate close) rather
   than crash. This is a future resilience story.
3. **Blast radius.** This is the deepest architectural change in the project.
   Every component that references channel count, device names, or gain
   node names is affected.
