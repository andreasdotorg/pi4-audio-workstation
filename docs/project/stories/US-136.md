## US-136: Web UI Information Architecture Redesign (Design Spike)

**As** the system operator,
**I want** the web UI and its underlying configuration model redesigned
holistically — covering config object relationships, management flows, and
navigation structure,
**so that** the system has a coherent, designed config architecture instead of
one that has grown organically, and I can find what I need in the UI without
scrolling through an increasingly crowded configuration page.

**Status:** draft
**Priority:** Medium (UX debt — becomes more urgent as US-132, US-134, US-110
add new UI sections)
**Effort:** 5-8 worker days (design spike — no implementation)
**Origin:** Owner directive (session 14): "conceptual and UX work" + "the
structure has grown, not designed, needs a fresh holistic view"
**Depends on:** none (design work, no code dependencies)
**Blocks:** Implementation stories derived from this spike
**Cross-references:** US-132 (adds Network config), US-134 (adds Hardware I/O
config), US-110 (adds Auth pages), US-137 (About section in System tab)

### Background

The web UI has 8 top-level tabs: Dashboard, System, Graph, Config, Measure,
TF, Test, MIDI. The Config tab has become a dumping ground with 12+
sub-sections in a single scrollable view: Channel Gains, Quantum, Filter
Chain, Venue, Audio Gate, FIR Filter Generation, Generation Results, Deploy
Filters, Filter Versions, Driver Protection, Speaker Profiles, Speaker
Identities, Amplifiers, DACs.

Upcoming stories will add: Network config (US-132), Hardware I/O config
(US-134), Auth/device management (US-110), and About info (US-137). Without
reorganization, the UI becomes unusable.

**But the problem is deeper than navigation.** The underlying config structure
has grown organically: some config lives in YAML files, some in environment
variables, some in GM in-memory state. Config changes propagate through
different paths (REST API, direct file writes, GM RPC). There is no unified
config object model or consistent management flow. The UI layout reflects
this ad-hoc structure. Reorganizing the UI without first designing the config
model would just be moving deck chairs.

The owner explicitly requested "conceptual and UX work" covering both the
config model and the UI. This is a design spike — no implementation. All
advisors agree: design first, implement in follow-up stories.

### Scope

This story produces DESIGN ARTIFACTS ONLY. No code changes. The deliverables
inform implementation stories that follow.

### Acceptance Criteria

**1. Config object model:**
- [ ] AC #1: Entity inventory diagram listing all config entities: venue,
      speaker profile, device profile (hardware I/O), driver, filter
      coefficients, safety constraints, runtime parameters (quantum, gains).
      For each: storage location, format, mutability model (immutable after
      activation / runtime-changeable / requires service restart).
- [ ] AC #2: Entity relationship diagram showing: venue -> speaker profiles
      -> drivers -> filter coefficients; device profile -> channel assignments
      -> speaker profiles. Include cardinality and optionality.
- [ ] AC #3: Lifecycle diagram per entity: create, validate, activate,
      deploy, archive. Which entities have distinct draft/active states?
      Which are immutable once deployed?

**2. Config flow diagram:**
- [ ] AC #4: Map ALL current config change propagation paths: Web UI -> REST
      API -> GM in-memory state -> file write (if persistent) -> service
      reload (if needed). Include env vars, direct file writes, GM RPC calls.
      Identify which paths are intentional design and which are ad-hoc.
- [ ] AC #5: Propose a unified config change model: a consistent flow for
      how config changes propagate, replacing the ad-hoc paths. Include
      safety gate interaction: which config changes require gate
      re-evaluation, which bypass it.

**3. UI information architecture (grounded in config model):**
- [ ] AC #6: Navigation hierarchy document: all views/pages, their grouping,
      navigation hierarchy (top nav, sub-tabs, sections within tabs), and
      which views are new vs. reorganized. The navigation structure reflects
      the config entity hierarchy from AC #1-2.
- [ ] AC #7: Evaluate and present the UX-proposed two-tier nav structure:

      **OPERATE (gig time):** Dashboard, Measure, Transfer Fn, Test, MIDI
      **SETUP (pre/post gig):** Config, Hardware, Network, System

      Config sub-tabs: Venue, Gains, Filters, Speakers, Safety
      Hardware sub-tabs: Devices, Channel Map, Amplifiers, DACs
      Network sub-tabs: Status, WiFi, Ethernet, AP Fallback
      System sub-tabs: Health, Graph, About

      Include rationale for each grouping decision.

**4. CRUD and management flows:**
- [ ] AC #8: Flow diagrams for config entity management: create new speaker
      profile, edit existing venue, clone and modify, delete/archive. Show
      validation points, confirmation steps, and rollback paths.

**5. Wireframes:**
- [ ] AC #9: Low-fidelity wireframes for:
      - (a) New top-level navigation layout (two tab groups with visual
        separator)
      - (b) Config tab with sub-tabs (Venue, Gains, Filters, Speakers,
        Safety)
      - (c) At least one new section (Network or Hardware) with sub-tabs
      - (d) Mobile wireframes for the 4 phone-priority views: Dashboard,
        Config > Venue, Config > Gains, Network > Status
      - (e) A CRUD flow wireframe (e.g., "create new speaker profile")

**6. User flow diagrams:**
- [ ] AC #10: Flow diagrams for the 3 primary operator journeys:
      - (a) Arrive at venue -> set up -> perform (setup flow)
      - (b) Mid-gig monitoring (operational flow)
      - (c) Troubleshooting (diagnostic flow)

      Each flow shows which views the operator visits in sequence and
      validates that the proposed IA supports the workflow without
      unnecessary navigation.

**7. Migration path:**
- [ ] AC #11: Document how to get from the current state to the target model:
      - What can be refactored incrementally vs. what requires a breaking
        change
      - Which implementation stories depend on this model being settled first
      - Tabs renamed, moved, or removed
      - Config sections redistributed to sub-tabs
      - Impact on existing E2E tests (which selectors change)
      - Impact on URL structure and API route organization

**8. Mobile priority classification:**
- [ ] AC #12: Classify every view as:
      - **Phone-priority:** must work well at 375px width (Dashboard,
        Config > Venue, Config > Gains, Network > Status)
      - **Tablet-acceptable:** functional at 768px, simplified at 375px
      - **Desktop-only:** requires 1024px+ (Measure, TF, Test — canvas-heavy)

### Design Principles (from advisory input)

1. **Config model drives UI, not the other way around.** The UI navigation
   and layout reflect the config entity hierarchy. Getting the config model
   right is the foundation — the UI is a projection of it.
2. **OPERATE vs. SETUP distinction.** The operator's mental model has two
   phases: during a gig (monitoring, adjustment) and before/after a gig
   (configuration, admin). The nav should reflect this.
3. **Setup workflow adjacency.** During venue setup, the operator needs in
   sequence: venue config -> measurement -> TF review -> gains/delays -> gate.
   These should be reachable in a natural flow.
4. **Config is not a dumping ground.** Sub-tabs within Config prevent the
   current scrolling problem while keeping related controls grouped.
5. **Unified change propagation.** All config changes should flow through a
   consistent path. No ad-hoc mix of REST, RPC, env vars, and direct file
   writes.
6. **Mutability model explicit.** Every config entity has a clearly defined
   mutability model: what's immutable after activation (coefficients), what's
   runtime-changeable (gains, quantum), what requires service restart.
7. **Hardware/Network/System are "set once" sections.** They're accessed
   during initial configuration but not during per-venue setup. Separate
   them from the performance-critical views.
8. **Navigation scaling.** The two-tier approach scales to ~12-15 top-level
   entries and ~5-7 sub-tabs per section before becoming unwieldy. Enough
   headroom for planned features.
9. **Design route structure for future RBAC.** Group API routes by
   sensitivity even if roles aren't implemented now: `/api/v1/network/`
   (future admin), `/api/v1/config/` (future admin), `/api/v1/status/`
   (future read-only). Costs nothing now, enables role-based access later.

### Risks

1. **Owner may prefer a different model or layout.** This is why it's a
   design spike — get approval before investing in implementation.
2. **E2E test breakage.** Any navigation change breaks tests that use tab
   selectors. AC #11 documents the impact; implementation stories handle the
   test migration.
3. **Scope creep.** The expanded spike (config model + UI) is significantly
   larger than a UI-only spike. Strictly enforce the "no code" boundary —
   the deliverable is a design document, not a prototype.
4. **Downstream dependency.** US-134 (hardware I/O), US-135 (room correction
   only), and venue management features all depend on the config model this
   spike defines. Getting it wrong means rework across multiple stories.
