## US-120: Real-Time Transfer Function Measurement with Coherence

**As** the audio system operator tuning a venue,
**I want** a continuously updating frequency response and coherence display using dual-FFT analysis of pink noise excitation,
**so that** I can verify and iteratively adjust room correction in real time without running sequential sweep measurements.

**Status:** draft
**Priority:** Medium — complements existing sweep/IR pipeline; does not replace it
**Effort:** 2-3 worker days
**Origin:** Owner-approved architect analysis (session 9)
**Depends on:** US-059 (GraphManager — DONE), US-115 (8ch convolver — Phase 0 DONE), US-052 (signal-gen — pink noise ready)
**Blocks:** US-122 (unified measurement display)
**Decisions:** D-040 (PW filter-chain), D-009 (cut-only correction / gain safety)
**Theory:** `docs/theory/realtime-transfer-function.md`

### Background

The existing measurement pipeline uses sequential sweep/deconvolution — high SNR but
slow (5-15 seconds per channel, room must be quiet, full re-measurement cycle for
each adjustment). SMAART-style dual-FFT transfer function analysis provides continuous
real-time feedback: play pink noise, capture reference and measurement signals, compute
cross-spectrum with Welch's method. Magnitude-squared coherence (MSC) shows per-frequency
measurement confidence (0-1), letting the operator know which frequencies are trustworthy
and which are contaminated by ambient noise.

Existing infrastructure ready for reuse:
- signal-gen: PinkNoiseGenerator (Voss-McCartney, +/-0.5 dB slope accuracy) — no changes
- pcm-bridge: monitor instance (port 9100) for reference, capture instance (port 9101)
  for UMIK-1 measurement — no changes
- Web UI: 4096-point FFT pipeline with Hann window and spectrum renderer — needs extension
- GraphManager: link topology management — needs new measurement-with-noise mode

### Acceptance Criteria

- [ ] **AC #1: Transfer function WebSocket endpoint.** New `/ws/transfer-function`
      WebSocket endpoint in the web-ui backend. Reads both pcm-bridge TCP streams
      (monitor port 9100 for reference, capture port 9101 for measurement). Computes
      time-aligned cross-spectrum using Welch's method. Outputs JSON frames:
      `{magnitude_db[], phase_deg[], coherence[], freq_axis[], channel}`.
      Parameters: FFT size (default 4096), averaging count (default 16), overlap (50%).

- [ ] **AC #2: Time alignment (automatic delay finder).** On measurement start,
      cross-correlate reference and measurement signals to find propagation delay.
      Apply delay to reference before computing cross-spectrum. Re-compute periodically
      (every ~5 seconds) to track speaker movement. Without time alignment, phase of
      Gxy is meaningless and Cxy drops — this is a prerequisite for valid results.

- [ ] **AC #3: Coherence computation.** Magnitude-squared coherence per frequency bin:
      `Cxy(f) = |Gxy(f)|^2 / (Gxx(f) * Gyy(f))`. Uses exponential averaging with
      configurable alpha parameter (controls display memory / update speed).
      Coherence only meaningful with multiple averaged blocks (minimum 8).

- [ ] **AC #4: Design vs verify mode (Dirac / correction filter toggle).** The
      measurement supports two modes, both using the **same post-convolver reference
      tap** (pcm-bridge-monitor, port 9100). No pre-convolver reference needed.
      - **Design mode:** GM loads **Dirac (identity) room correction coefficients**
        into the convolver while preserving crossover slopes and HPF (D-031 driver
        protection stays active). The transfer function shows the raw room response
        — speaker + crossover + room, no room correction applied. Used for initial
        measurement and filter design.
      - **Verify mode:** GM loads the **real room correction filters** into the
        convolver (crossover/HPF unchanged). The transfer function shows the
        corrected response — should be approximately flat if correction is working.
        Used to verify deployed filters achieve the target.
      The UI provides a simple toggle: "Dirac (design)" / "Correction (verify)".
      The UI clearly indicates which mode is active.

- [ ] **AC #5: Measurement gain profile (S-012).** GM enters a predefined
      "measurement" mode that: (a) establishes measurement link topology —
      signal-gen -> convolver input, convolver output -> speakers + pcm-bridge-monitor,
      USBStreamer capture -> pcm-bridge-capture; (b) loads Dirac or correction filters
      per AC #4; (c) elevates output gain to a measurement-appropriate level within
      D-009 safety bounds; (d) requires explicit operator confirmation before
      activating (S-012 — no unattended gain changes). Measurement mode coexists
      with audio gate — measurement can run while gate is open (live tuning during
      soundcheck).

- [ ] **AC #6: Web UI transfer function display.** New component showing:
      magnitude (dB) vs frequency (log scale, 20Hz-20kHz), phase (degrees) vs
      frequency, coherence overlay color-coded (green > 0.85, yellow 0.5-0.85,
      red < 0.5). Controls: start/stop, channel select, averaging speed slider.
      Optional target curve overlay for visual comparison.

- [ ] **AC #7: Safety compliance.** signal-gen -20 dBFS cap applies to continuous
      noise (S-2). D-009 gain constraints unchanged — real-time TF is read-only
      measurement, no automated correction in this story. Measurement gain profile
      (AC #5) must be explicitly confirmed by the operator before any gain elevation.

- [ ] **AC #8: Local-demo validation.** Transfer function works in local-demo
      with room-sim convolver. Design mode (Dirac): pink noise through identity
      room filters + crossover + room-sim produces expected raw room response shape.
      Verify mode (correction): pink noise through correction filters + crossover +
      room-sim produces approximately flat response. Both modes use post-convolver
      reference. Coherence converges to > 0.9 in local-demo (no ambient noise
      contamination).

### Intended Iterative Workflow

The design and verify modes support an iterative correction cycle. Both modes use
the same post-convolver reference tap — only the convolver coefficients change.

1. **Design phase:** Operator toggles to "Dirac (design)." GM loads identity room
   correction coefficients (crossover/HPF preserved, D-031 protection active).
   Pink noise plays through speakers. Real-time TF shows raw room response —
   peaks, nulls, room modes visible through the crossover.
2. **Filter design:** Operator captures a snapshot of the raw room response. The
   existing sweep/IR pipeline (or future automated correction) generates FIR
   correction filters from this data.
3. **Verify phase:** Operator toggles to "Correction (verify)." GM loads the new
   correction filters into the convolver (replacing the Dirac coefficients,
   crossover/HPF unchanged). Real-time TF now shows the corrected response —
   should be approximately flat. Deviations from target indicate where correction
   is insufficient or over-correcting.
4. **Iterate:** If correction is inadequate, toggle back to "Dirac (design)" and
   refine. Each cycle takes seconds instead of minutes.

This cycle replaces the previous "measure -> compute -> deploy -> re-measure" batch
workflow with a continuous, interactive loop. The simplified architecture (always
post-convolver, toggle coefficients) avoids any need for pre-convolver reference
tapping or link topology changes between modes.

### Implementation Approach

**Prototype: Option B (server-side Python)** — web-ui backend reads both pcm-bridge
TCP streams, computes cross-spectrum with numpy. Browser renders pre-computed data.
If CPU cost exceeds ~5-10% of one core on Pi 4, migrate to Option C (dedicated Rust
service `transfer-function-bridge`).

### Future Work (Phase 2)

- **Program-material-as-reference:** Use live music (DJ output or backing track) as
  the reference signal instead of pink noise. Enables real-time transfer function
  measurement during live performance without audible test signals. Requires the
  dual-FFT engine to accept an arbitrary reference stream — the math is identical
  (cross-spectrum of program material vs mic). Coherence will be lower (program
  material is not spectrally flat), but sufficient for broadband verification.
  This is a Phase 2 enhancement — Phase 1 uses dedicated pink noise excitation.

### Risks

1. **CPU cost on Pi 4.** Python numpy cross-spectrum computation may be too expensive
   for real-time updates. Mitigation: measure CPU cost early (T-120-01), migrate to
   Rust if > 5-10% of one core.
2. **Two concurrent pcm-bridge WebSocket streams.** ~384 KB/s each from Pi to browser.
   Server-side computation (Option B) avoids this — browser only receives pre-computed
   JSON (~1 KB/frame).
3. **Time alignment bootstrap.** If propagation delay is large (>50ms in reverberant
   venue), initial cross-correlation window must be wide enough to capture the peak.
