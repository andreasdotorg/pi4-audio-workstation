## US-110: Web UI Passkey Authentication

**As** the system operator performing at a venue,
**I want** the web UI protected by passkey authentication with invite-based
onboarding,
**so that** unauthenticated devices on the venue network cannot control the
signal generator, trigger measurement sweeps, or access live audio streams
during a performance.

**Status:** IMPLEMENT (session 4 — architect decomposed 17 tasks)
**Priority:** High (security — live performance availability risk)
**Effort:** 3-5 worker days
**Origin:** F-037 (web UI no auth — converted to story)
**Depends on:** D-060 (local CA for TLS — approved 2026-03-29)
**Blocks:** US-018 (guest device access requires auth first)
**Decisions:** D-032 (HTTPS), D-060 (local CA replaces self-signed certs)

### Background

F-037 identified that the web UI binds on all interfaces with no
authentication. Any device on the venue LAN can control the signal generator,
trigger measurement sweeps, and access live PCM audio streams. During a live
performance this is a direct availability threat — an attacker could play
test tones through the PA.

All advisors (security, architect, UX) aligned on passkey-based auth
(2026-03-29). Passkeys provide phishing-resistant, passwordless
authentication suitable for the venue threat model. The owner explicitly
chose passkeys over simpler schemes (Basic Auth, API keys).

### Acceptance Criteria

#### Registration and Invite Flow

- [ ] AC #1: **CLI bootstrap command** — `mugge-auth invite` generates a
      one-time registration URL with a 256-bit cryptographically random,
      URL-safe base64 token. Displays the URL and a terminal-rendered QR
      code. Token is single-use with 30-minute expiry. The first invite
      auto-grants admin role. Subsequent invites (via CLI) require no
      existing auth — CLI access implies SSH access implies admin.
- [ ] AC #2: **Web UI invite generation** — authenticated users can
      generate invite QR codes from a "Invite device" action in the web UI.
      QR code rendered client-side (e.g., `qrcode.js`) with a "Copy link"
      button and visible expiry countdown. In-UI invite tokens have a
      10-minute expiry (shorter than CLI because the inviting user is
      present). A "Register this browser" button is also available for the
      admin to register their own device without scanning.
- [ ] AC #3: **Passkey registration** — the registration page triggers the
      WebAuthn `navigator.credentials.create()` ceremony. The server stores
      the credential public key, credential ID, and a human-readable device
      name. No passwords are stored or accepted at any point. After
      registration the invite token is consumed and cannot be reused.
      Expired or consumed tokens show a clear error page with recovery
      instructions (not a redirect to login).
- [ ] AC #4: **No open registration window** — every passkey registration
      requires a valid invite token, including the very first one. There is
      no time window where the system accepts unauthenticated registrations.
      `mugge-auth invite` is the only bootstrap path.
- [ ] AC #5: **Device management** — authenticated users can list registered
      devices and revoke individual credentials via the web UI. Revoking a
      credential immediately invalidates all active sessions for that
      credential.

#### Login Flow

- [ ] AC #6: **Passkey login** — unauthenticated access to any protected
      page redirects to `/auth/login` with `?next=` for post-login redirect.
      The login page shows a single "Sign in" button (no username field —
      passkeys self-identify). The button triggers the WebAuthn
      `navigator.credentials.get()` ceremony. On successful assertion the
      server issues a session cookie and redirects to the original page.
      No password fallback exists.
- [ ] AC #7: **Session cookie** — cookie name: `__Host-pi4audio_session`.
      Flags: `HttpOnly`, `Secure`, `SameSite=Strict`, `Path=/`. Session
      token is cryptographically random, stored server-side in SQLite.
      Sessions expire after **12 hours of inactivity** (sliding window —
      activity resets the timer). Covers a full gig day. A new session ID
      is generated after each successful authentication (prevents session
      fixation).
- [ ] AC #8: **Sign out** — a non-prominent "Sign out" option in the
      settings area (not on the dashboard) revokes the session server-side.

#### WebSocket Authentication

- [ ] AC #9: **WS auth on handshake** — all 6 WebSocket endpoints
      (`/ws/monitoring`, `/ws/system`, `/ws/pcm`, `/ws/pcm/{source}`,
      `/ws/siggen`, `/ws/measurement`) validate the session cookie in the
      HTTP upgrade request BEFORE calling `ws.accept()`. Connections
      without a valid session are rejected with WebSocket close code 4003.
      No per-message auth — the handshake is the auth boundary.
- [ ] AC #10: **Session expiry during WS** — if a session expires while a
      WebSocket is open, the client displays an inline overlay banner
      "Session expired — please sign in" with a clickable sign-in link
      (same tab, not a jarring full-page redirect). After re-authentication,
      WebSocket connections auto-reconnect without manual page refresh.

#### Localhost Bypass

- [ ] AC #11: **Kiosk mode bypass** — requests originating from `127.0.0.1`
      or `::1` bypass authentication entirely. This enables the local kiosk
      display (labwc/Chromium on the Pi itself) to show the dashboard without
      credentials. The bypass applies to both HTTP and WebSocket endpoints.
      Localhost check uses the actual TCP peer address (`request.client.host`),
      NEVER `X-Forwarded-For` or other proxy headers.
- [ ] AC #12: **Bypass is not configurable** — localhost is always trusted.
      Deliberate design choice: physical access to the Pi already implies
      full system access. Consistent with existing localhost-only services
      (GraphManager :4002, signal-gen :4001, pcm-bridge :9100).

#### Endpoint Protection

- [ ] AC #13: **All endpoints require auth** — every HTTP and WebSocket
      endpoint requires a valid session, including read-only monitoring
      (`/`, `/api/status`, `/ws/pcm/*`). The owner chose full protection
      over convenience.
- [ ] AC #14: **Static assets and login exempted** — CSS, JS, images, fonts,
      and the login/registration pages are served without auth. These are the
      only resources accessible without a session.

#### Storage and Security

- [ ] AC #15: **Credential storage** — WebAuthn credentials, sessions, and
      invite tokens stored in a SQLite database via `aiosqlite` (async — must
      not block the asyncio event loop per F-064 lessons). Database path:
      `/var/lib/pi4audio/auth.db`. File permissions 0600, owned by the
      service user. Path provisioned via `systemd.tmpfiles.rules`.
- [ ] AC #16: **No secrets in config files** — no shared passwords, API
      keys, or tokens in configuration files or environment variables. All
      authentication material is in the SQLite database, generated at
      runtime.
- [ ] AC #17: **No account lockouts** — failed authentication attempts are
      rate-limited (1-second delay between attempts) but never lock out a
      credential. Account lockouts during a live gig are unacceptable.

### Future Enhancements (out of scope)

- Better admin bootstrap than CLI (e.g., physical button, OLED prompt) —
  owner noted `mugge-auth invite` is adequate for now
- Role-based access (read-only vs admin, engineer vs singer) — all
  authenticated users are equal for MVP
- Remote invite management (send invite link via network) — QR code at
  the Pi is sufficient for venue setup
- Audit logging of auth events
- Relying Party ID configuration (test `mugge.local` vs IP across browsers
  during implementation)

### Technical Notes

- Library: `py_webauthn` (BSD-3, high-level WebAuthn API, FastAPI-friendly)
- Storage: `aiosqlite` for async SQLite access
- QR rendering: `qrcode` (Python, for CLI) + `qrcode.js` (browser, for
  in-UI invites)
- New module: `src/web-ui/app/auth/` (webauthn.py, middleware.py,
  models.py, routes.py)
- New pages: `static/auth/login.html`, `static/auth/register.html`
- CLI tool: `scripts/mugge-auth` (invite subcommand)
- Nix: add `py-webauthn`, `aiosqlite`, `qrcode` to web-ui Python env
- HTTPS/TLS with a valid certificate chain is a prerequisite for passkeys —
  WebAuthn requires `SecurityLevel::SECURE`. D-060 (local CA) replaces D-032's
  self-signed approach, which was blocked in Chrome and Safari.
- WebSocket auth via cookie in the upgrade request is natively supported
  by browsers — no custom token protocol needed.
- SQLite is appropriate for the expected scale (1-5 registered devices).
