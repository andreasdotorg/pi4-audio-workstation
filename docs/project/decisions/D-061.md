## D-061: GM manages PipeWire and WirePlumber lifecycle — target architecture amendment (2026-03-30)

**Context:** D-058 (2026-03-28) established GM as the process supervisor for
signal-chain services (pcm-bridge, level-bridge, signal-gen) as internal
threads. However, D-058's "unchanged" table explicitly listed PipeWire and
WirePlumber as systemd-managed singletons. Session 5 discussion with the
owner challenged this: there is no fundamental reason PipeWire must be
managed by systemd when GM is already the audio session owner (D-050) and
service supervisor (D-058).

The immediate trigger was F-220 / F-221: the filter deploy panel uses
`reload_pipewire()` which does `systemctl --user restart pipewire.service`
(or `local-pw-test-env.sh stop && start` in local-demo), killing all PW
clients. The measurement session's `_reload_convolver()` approach (`pw-cli
destroy` + auto-recreate) is non-disruptive but still requires GM to
re-link after node recreation. If GM owned the PW lifecycle, it could
coordinate the entire restart/reload sequence internally.

**Decision:** Amend D-058 — the **target architecture** moves PipeWire and
WirePlumber from "systemd-managed" to "GM-managed." GM spawns PW (and WP
if needed) as child processes, gaining full lifecycle control.

**What this enables:**

1. **Deterministic startup ordering:** GM → PW → WP → filter-chain ports
   activate → GM spawns bridges → GM establishes links. No race conditions,
   no systemd dependency ordering.
2. **Coordinated restart:** GM can restart PW, wait for ports, re-link —
   all internally. No shell hacks, no systemd, no killed-client problem.
3. **Coordinated coefficient reload:** GM destroys convolver node via
   `pw-cli destroy` (or libpipewire API), waits for PW to recreate from
   `.conf.d/`, re-establishes all links. Single coordinator, no external
   scripts.
4. **Clean shutdown:** One SIGTERM to GM tears down everything in order.
   PW → WP → bridges all stop cleanly. No orphans.
5. **Single systemd unit:** `graph-manager.service`. That's it. Everything
   else is internal to GM.
6. **Local-demo simplification:** `local-demo.sh` reduces to "start GM."
   No process juggling, no trap handlers, no `PW_TEST_ENV` scripts.

**Updated D-058 table — target architecture:**

| Service | Management | Notes |
|---------|-----------|-------|
| PipeWire | GM child process | GM spawns, monitors, restarts |
| WirePlumber | GM child process | If needed for port activation (PW 1.6.x constraint) |
| pcm-bridge | GM thread | Dynamic instances per topology |
| level-bridge | GM thread | Dynamic instances per topology |
| signal-gen | GM thread | Dynamic instances per topology |
| GraphManager | systemd unit | The sole systemd-managed service |
| Web UI | systemd unit | HTTP server, no PW dependency |

**PipeWire filter-chain limitation (C-011):** PW filter-chain does NOT
support hot-reloading coefficient WAV files. The `config.filename` property
is static — read once at node creation. The only path to swap coefficients
is destroy-and-recreate (`pw-cli destroy`), which breaks all links. GM must
re-link after recreation. This is a D-040 tradeoff — CamillaDSP had
glitch-free hot-reload via its websocket API; PW filter-chain does not.
See C-011 for full documentation.

**Implementation path:**

This is a significant architectural change. Implementation order:
1. GM spawns PW as a child process (with `PIPEWIRE_RUNTIME_DIR` control)
2. GM spawns WP as a child process (if PW 1.6.x port activation requires it)
3. GM detects PW/WP readiness (port activation, node presence)
4. GM spawns signal-chain threads (existing D-058 plan)
5. GM handles PW restart (kill PW child, respawn, wait for readiness,
   re-establish links)
6. `local-demo.sh` simplified to start GM only
7. Static systemd units for PW/WP removed from NixOS config

**Amends:** D-058 table (PW/WP moved from "unchanged/systemd" to
"GM-managed"). Does NOT amend D-043 (WP retained for port activation —
still required on PW 1.6.x).
