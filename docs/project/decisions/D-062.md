## D-062: First-Boot / Active Config Architecture — Symlink-Based Coefficient Management (2026-03-30)

**Context:** The system needs a clear architecture for how speaker profiles,
venue measurements, and coefficient files relate to each other. Previously,
coefficient files were generated and placed directly in `/etc/pi4audio/coeffs/`.
This created ambiguity about which profile/venue was active, risked desync
between config and coefficients, and had no clear first-boot or FoH passthrough
story. The local-demo also generated Dirac impulses at startup, violating the
mock boundary (US-075 audit).

**Decision:** Two-layer config model with symlink-based activation. Three
authoritative sources produce two derived artifacts. No file copying.

### Two Config Layers

1. **Speaker Profile** (hardware layer) — defines what speakers are connected:
   crossover frequencies, driver count, gain staging limits, thermal/mechanical
   protection parameters. Stored as YAML. Generates crossover FIR coefficients.

2. **Venue Config** (measurement layer) — room correction, time alignment,
   per-channel gain trims. Derived from measurement data at a specific venue
   with a specific speaker profile. Combined with speaker profile crossover
   to produce final FIR coefficients.

### Three Authoritative Sources

1. Speaker profile YAML (defines hardware)
2. Venue measurement data (defines room)
3. `active.yml` pointer (declares which profile + venue is active)

### Two Derived Artifacts

1. `/etc/pi4audio/coeffs` symlink → active venue's coefficient directory
2. PW filter-chain `.conf` file (generated from active profile topology)

**Activation = `ln -sfn` + regenerate `.conf`.** Nothing is copied. The symlink
ensures coefficients and config cannot desync — they always point to the same
source of truth.

### FoH Passthrough as Baseline

The simplest use case is "output channels cleanly to front-of-house" — no
crossover, no room correction, unity gain, immediate audio. This is implemented
as a **built-in speaker profile** (not a separate mode). The FoH passthrough
profile uses Dirac (identity) coefficients with unity gain. A freshly deployed
system defaults to this profile.

### "Own Speakers" Mode Adds Safety

When the operator selects a speaker profile for their own speakers:
- Default gain is mute (`Mult=0.0`) — safety-first
- Speaker profile activation generates crossover coefficients
- Explicit gain staging gate before audio flows
- Thermal limiter active (US-092)

### State Machine

```
Fresh deploy → FoH passthrough active (Dirac + unity gain)
            → Select speaker profile (own speakers)
            → Optional: venue measurement (room correction)
            → Auto-restore on reboot (active.yml persists)
```

### Local-Demo Impact

Dirac generation removed from local-demo startup — violated mock boundary.
Local-demo should behave identically to a freshly deployed production system
(FoH passthrough profile active by default).

**Rationale:** (1) Symlink activation eliminates an entire class of desync
bugs (copied files diverging from source). (2) FoH passthrough as a built-in
profile means the system produces audio immediately after deployment without
any configuration — critical for the "power on and go" gig workflow. (3)
Mute-default for own speakers prevents accidental speaker damage during
profile setup. (4) The two-layer model cleanly separates "what speakers do
I have" from "what room am I in" — speakers are reused across venues, venue
measurements are per-location.

**Impact:** Amends D-010 (speaker profiles now include FoH passthrough as
built-in), D-051 (config generator consumes profile YAML + venue data via
active.yml pointer), D-053 (profile switching safety flow incorporates the
mute-default for own speakers). US-089 (speaker config management) and
US-011b (config generator) must implement this architecture.

**Owner directives captured:**
- "System should work without any room correction or speaker setup, just outputting channels"
- "Safety mechanism to prevent blowing out speakers, e.g. forcing gain staging setup"
- "Active config drives the coefficients"
- "Bad pattern: just copy files. Good pattern: active config and coefficients in sync"
- "Simplest case is linear passthrough to FoH"
