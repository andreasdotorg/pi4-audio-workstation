## D-060: Local CA for TLS — replacing self-signed certificates (2026-03-29)

**Context:** D-032 established HTTPS for the web UI using a self-signed
certificate. This was sufficient for AudioWorklet secure context requirements
but creates a critical blocker for US-110 (passkey authentication).

WebAuthn requires `SecurityLevel::SECURE` in the browser. Self-signed
certificates get `DANGEROUS` security level even after the user clicks
"Proceed" through the browser warning. This was confirmed from Chromium
source code (`IsSecurityLevelAcceptableForWebAuthn()`) during session 2
(2026-03-29):

- **Chrome:** BLOCKED — source-confirmed. Self-signed cert gets
  `CERTIFICATE_ERROR`, WebAuthn returns `NotAllowedError`.
- **Safari:** BLOCKED — strict certificate enforcement (likely same behavior).
- **Firefox:** May work (historically permissive), but not reliable enough
  to depend on.
- **Localhost:** Always allowed (irrelevant for remote device access).

Without a valid certificate chain, passkey registration and login ceremonies
fail on the two most common browsers. US-110 cannot proceed with D-032's
self-signed approach.

**Decision:** Replace self-signed certificates with a **local Certificate
Authority (CA)** generated on the Pi. The CA signs the web UI server
certificate, and the owner installs the CA certificate on their devices once.

**Owner directive:** Approved 2026-03-29.

**Implementation:**

1. **CA generation:** `mugge-auth init` generates a local CA key pair and
   self-signs the CA certificate. CA key stored at
   `/var/lib/pi4audio/ca/ca.key` (0600, service user). CA cert at
   `/var/lib/pi4audio/ca/ca.crt`.
2. **Server cert signing:** `mugge-auth init` also generates the web UI
   server key and certificate, signed by the local CA. Server cert includes
   SANs for the Pi's hostname (`mugge.local`) and IP addresses. Stored at
   the existing cert paths (`/etc/pi4audio/certs/`).
3. **CA cert distribution:** The CA certificate is made available for
   download via a dedicated endpoint or the invite flow. The owner installs
   it once per device (iOS Settings, macOS Keychain, Chrome cert store,
   Firefox cert manager).
4. **Certificate renewal:** CA cert has a long validity (10 years). Server
   cert has 1-year validity. `mugge-auth renew` regenerates the server cert
   without requiring CA reinstallation on devices.
5. **Per-platform install docs:** SETUP-MANUAL.md updated with CA
   installation instructions for iOS, macOS, Android, Windows, and Linux.

**What this enables:**

- Browsers see a valid certificate chain (CA cert trusted locally).
- `SecurityLevel::SECURE` achieved — WebAuthn ceremonies succeed.
- No browser security warnings on trusted devices.
- HTTPS remains mandatory (D-032 still applies for AudioWorklet).

**What this does NOT change:**

- The CA is local-only. It provides no protection against MITM by someone
  who compromises the CA key on the Pi. This is acceptable for the venue
  threat model — physical access to the Pi already implies full control.
- Devices that have not installed the CA cert will still see browser
  warnings (same as current D-032 behavior). They cannot use passkeys.
- The `mugge-auth` CLI tool is the sole management interface for
  certificates and authentication (also handles invite tokens per US-110).

**Standard pattern:** This is the standard approach for LAN appliances
(Home Assistant, Synology, UniFi). Local CA + device trust is the only way
to get valid TLS on a `.local` hostname without a public domain and
external CA (Let's Encrypt requires port 80/443 reachable from the
Internet).

**Amends:** D-032 (HTTPS for AudioWorklet). D-032's self-signed certificate
approach is replaced by local CA-signed certificates. The HTTPS requirement
itself remains unchanged.

**Related:** D-032 (HTTPS), US-110 (passkey auth — unblocked by D-060),
F-037 (web UI no auth — origin of US-110), US-000a (platform security).
