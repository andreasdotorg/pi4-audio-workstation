## D-064: NixOS VM Test Framework for CI Smoke Testing (2026-04-04)

**Status:** DRAFT

**Context:** The current CI pipeline has a gap between T2 (E2E tests that validate
software behavior in a local-demo environment) and T3 (SD card image build that
produces the deployment closure). The missing verification is: "do the NixOS modules
wire services correctly via systemd?" This is currently only caught by manual smoke
testing on real Pi hardware (Gate 3), which is slow, requires physical access, and
cannot run in CI.

Two approaches were evaluated:

### Approach 1: NixOS VM Test Framework (`pkgs.nixosTest`)

Write a `smoke.nix` test using the NixOS VM test framework (`pkgs.nixosTest`). This
boots a QEMU VM with our shared NixOS module set, using hardware overrides appropriate
for a virtual machine (no Pi-specific firmware, kernel, or device tree).

**What it tests:**
- systemd unit wiring (service dependencies, ordering, socket activation)
- Service health: PipeWire, WirePlumber, GraphManager, signal-gen, web UI all start
- Web API responses (health endpoint, config endpoint, graph topology)
- User session setup (greetd, labwc, Wayland environment variables)
- File permissions, tmpfiles, SSL certificate generation

**What it cannot test:**
- Pi 4 boot chain (U-Boot, firmware blobs, `config.txt`, device tree overlays)
- PREEMPT_RT kernel behavior (VM boots a standard NixOS kernel)
- Real hardware: V3D GPU, USB audio (USBStreamer), ALSA device nodes
- Actual SD card image boot sequence

**Characteristics:**
- KVM required (GitHub ARM runners provide this)
- Runtime: ~3-5 minutes
- Implementation: ~1 day
- Runs as T1.5 in CI (after Nix eval, before E2E tests)
- Low maintenance burden — reuses the same NixOS modules as the SD image build

### Approach 2: Boot Actual SD Image in QEMU

Decompress the `.img.zst` SD card image, extract the kernel and initrd from the
NixOS generation, and boot with `qemu-system-aarch64 -machine virt`.

**Problems identified:**
- The Pi 4 PREEMPT_RT kernel is built with `bcm2711_defconfig` and lacks virtio
  drivers (`virtio_blk`, `virtio_net`, `virtio_console`) needed for QEMU `-machine
  virt`
- Device tree mismatch: the SD image ships `bcm2711-rpi-4-b.dtb`; QEMU virt expects
  its own generated DTB
- initrd modules are Pi-specific (SD card, USB, DWC2) — no overlap with QEMU virt
  requirements
- Would require building a QEMU-compatible NixOS configuration variant with different
  kernel, initrd, and DTB — effectively becoming Approach 1 with extra steps
- U-Boot does not run in QEMU virt, so the actual Pi 4 boot chain is untested
  regardless

**Characteristics:**
- Runtime: ~10-15 minutes
- Disk: ~15 GB (decompressed image + QEMU overhead)
- Implementation: ~3-5 days
- High maintenance burden — must track kernel config changes, DTB changes, and
  QEMU compatibility independently
- Still cannot test the actual Pi 4 boot sequence

**Decision:** Adopt Approach 1 — NixOS VM Test Framework (`pkgs.nixosTest`).

Approach 2 provides no additional coverage over Approach 1. Both approaches are
unable to test the Pi 4 hardware boot chain (firmware, U-Boot, RT kernel, device
tree). The service wiring verification — systemd units, service health, API
responses — is identical in both approaches. Approach 2 adds significant
implementation complexity (3-5x), runtime cost (3-5x), and maintenance burden
for zero additional test coverage.

The remaining gap (Pi 4 boot chain validation) is inherently untestable in any
VM or emulator. It requires real hardware and remains a Gate 3 manual test.

### Test Scope

The `smoke.nix` test MUST verify:

| Category | Checks |
|----------|--------|
| systemd wiring | All audio-stack units reach `active` state: PipeWire, WirePlumber, GraphManager, signal-gen, pcm-bridge (both instances), web-ui, SSL cert generator |
| Service health | GraphManager RPC responds on port 4002, signal-gen on port 4001, pcm-bridge on port 9100 |
| Web UI | HTTPS on port 8080: `/api/v1/status` returns 200, `/api/v1/config` returns valid JSON, `/api/v1/graph/topology` returns nodes and links |
| User session | greetd auto-login succeeds, labwc Wayland compositor starts, `XDG_RUNTIME_DIR` and `WAYLAND_DISPLAY` set |
| PipeWire | `pw-cli info 0` succeeds, filter-chain convolver node exists, 8 gain nodes present |
| File system | Coefficient directory exists with expected structure, `active.yml` present, SSL certificates generated |

### CI Integration

- **Stage:** T1.5 (after `nix flake check` / eval, before E2E test suite)
- **Runner:** ARM with KVM (`runs-on: [self-hosted, ARM64]` or GitHub ARM runner)
- **Failure mode:** Hard gate — T1.5 failure blocks the pipeline. No point running
  E2E tests if NixOS modules don't wire correctly.
- **Flake output:** `checks.aarch64-linux.smoke` (standard NixOS test integration)

### Hardware Override Pattern

The smoke test imports the production NixOS modules but overrides hardware-specific
settings:

```nix
# Pseudocode — actual implementation may vary
pkgs.nixosTest {
  nodes.machine = { pkgs, ... }: {
    imports = [ ../../nix/nixos/configuration.nix ];

    # Override Pi-specific hardware
    boot.loader.grub.enable = false;  # no U-Boot in VM
    hardware.raspberry-pi = lib.mkForce {};
    # Use VM-compatible kernel instead of Pi RT kernel
    boot.kernelPackages = lib.mkForce pkgs.linuxPackages;
    # Disable hardware-specific services
    services.pipewire.extraConfig = { /* null sink instead of USBStreamer */ };
  };

  testScript = ''
    machine.wait_for_unit("multi-user.target")
    machine.wait_for_unit("pipewire.service", "ela")
    machine.succeed("curl -sk https://localhost:8080/api/v1/status")
    # ... further checks per scope table above
  '';
}
```

**Rationale:** (1) The NixOS VM test framework is battle-tested infrastructure
used by nixpkgs itself for thousands of integration tests. (2) Reusing production
NixOS modules with hardware overrides ensures the test validates the same systemd
wiring that runs on the Pi. (3) ~3-5 minute runtime is fast enough for CI without
being a bottleneck. (4) KVM is available on GitHub ARM runners, making this viable
without self-hosted infrastructure. (5) The ~1 day implementation cost is
proportional to the value: catching systemd wiring regressions before they reach
Gate 3 saves hours of manual Pi smoke testing per iteration.

**Consequences:**
- New flake output: `checks.aarch64-linux.smoke`
- CI pipeline gains T1.5 stage between eval and E2E
- Gate 3 manual Pi smoke test remains required for hardware boot chain validation
- NixOS module changes that break service wiring are caught automatically in CI
- D-023 (reproducible test protocol) is strengthened: the smoke test is an
  executable, version-controlled test that runs on every commit
