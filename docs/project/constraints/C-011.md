## C-011: PipeWire filter-chain does not support hot-reloading convolver coefficients (2026-03-30)

**Type:** Technical constraint (PipeWire architecture limitation)
**Verified:** Session 5, confirmed by codebase analysis and PW API investigation
**Permanent:** Yes — this is a PipeWire filter-chain module design limitation

### Constraint

PipeWire's `libpipewire-module-filter-chain` does NOT support runtime
modification of convolver coefficient WAV files. The `config.filename`
property in the filter graph definition is a **static load-time property**
— it is read once when the filter-chain module creates the node and is
never re-read.

There is no PipeWire API mechanism to update `filename`:
- `pw-cli set-param` / `pw_node_set_param()`: Only works for
  `SPA_PARAM_Props` — specifically the `params` array holding runtime
  control values (Mult, Add, Control). The convolver `config.filename`
  is NOT in the Props params array.
- `pw-cli send-command`: No known command to reload convolver coefficients.
- `filter.graph` property: Static configuration block parsed once at module
  load. No API to modify at runtime.

### What IS Settable at Runtime

The `linear` builtin gain nodes expose `Mult` and `Add` as runtime-settable
Props. These work via:
- `pw-cli s <node-id> Props '{ params = [ "gain_name:Mult" 0.5 ] }'`
- Native PW API: `pw_node_set_param()` with `SPA_PARAM_Props`

This is used for runtime gain adjustment (muting, attenuation) but cannot
change which WAV file a convolver loads.

### Reload Mechanism

The only way to swap coefficients is **destroy-and-recreate**:

1. Deploy new WAV files to the coefficients directory
2. Update the `.conf.d/` drop-in file if filenames changed
3. `pw-cli destroy <convolver-node-name>` — destroys the entire
   filter-chain node
4. PipeWire re-reads `.conf.d/` drop-in files and recreates the node
   from scratch with the new coefficient filenames
5. The new node gets a new PW node ID and new port IDs
6. **All links to/from the destroyed node are lost** — GraphManager must
   detect the new node and re-establish all links via reconciliation

### Implications

- Brief audio dropout during node destruction/recreation (~1-2 seconds)
- GraphManager must handle node disappearance and reappearance
- No glitch-free coefficient crossfade (unlike CamillaDSP's hot-reload)
- The USBStreamer output goes silent during the gap, but no transient risk
  (node simply disappears — no USB reset occurs)

### D-040 Tradeoff

This limitation was implicitly accepted in D-040 (CamillaDSP abandoned).
CamillaDSP supported glitch-free coefficient hot-reload via its websocket
API (`config.reload()`). The D-040 decision was based on CPU efficiency
(3-5.6x better) and architectural simplicity (no external DSP process).
The hot-reload capability was not identified as a tradeoff at the time
because the measurement session's `_reload_convolver()` approach provides
an adequate (if not seamless) alternative.
