### 1.2 Test Objective

**Type:** Benchmark (hypothesis test).

**Hypothesis:** PipeWire's built-in filter-chain convolver, using non-uniform
partitioned convolution with PFFFT (ARM NEON intrinsics), can process 16,384-tap
FIR filters on 4 speaker channels at 48 kHz on the Pi 4B within an acceptable
CPU budget.

**Question this test answers:** Is PipeWire-native FIR convolution viable on Pi
4B ARM (Cortex-A72), and how does its CPU cost compare to CamillaDSP's
partitioned FFT engine under identical workloads?

**What this benchmark is NOT:**
- Not a feature-equivalence test (CamillaDSP has websocket API, config hot-swap,
  level metering -- none of that is tested here).
- Not a production deployment test (filter-chain is not replacing CamillaDSP
  today -- this informs long-term architecture direction per the unified graph
  analysis decision tree).
- Not a latency test (latency is measured by G-0 in US-056, not here).

**Context:** BM-2 is the single most important benchmark in the unified graph
analysis (Section 8.1). The question is not whether PipeWire has partitioned
convolution (confirmed: since v0.3.56, 2022) but whether its ARM performance
on Cortex-A72 with 16,384-tap filters is within budget. AE expects PASS based
on PFFFT's hand-written ARM NEON intrinsics being competitive with or faster
than CamillaDSP's rustfft (LLVM auto-vectorization). This benchmark produces
definitive data.

**Synthetic Dirac impulse methodology (matching US-001):**

The FIR filter files are single-sample impulses (value 1.0 at sample 0, zeros
thereafter) of length 16,384 samples. This ensures:
1. The convolution workload is identical to production filters (same tap count,
   same FFT sizes, same partition count).
2. The filter is a perfect passthrough -- output equals input (no frequency
   modification), making signal integrity trivially verifiable.
3. Direct comparability with US-001 CamillaDSP benchmarks which used the same
   Dirac impulse methodology.

Each of the 4 convolvers loads an identical Dirac WAV. The computational cost
of partitioned convolution depends on tap count and channel count, not on filter
coefficient values -- a Dirac impulse exercises the same code path as a real
room correction filter.

### 1.5 Pass/Fail Criteria

| # | Criterion | Measurement method | Pass condition | Fail condition | Justification |
|---|-----------|-------------------|----------------|----------------|---------------|
| C-1 | CPU% at quantum 1024 (DJ mode) | `pw-top -b` average over 5 minutes (see Section 1.6 Step 5) | Mean CPU < 20% | Mean CPU > 30% | US-058 AC from unified graph analysis: < 20% is comparable to CamillaDSP ALSA baseline (~8-9% interpolated at chunksize 1024, between T1a 5.23%@cs2048 and T1b 10.42%@cs512). > 30% means filter-chain is not viable on Pi 4B. 20-30% is MARGINAL. |
| C-2 | CPU% at quantum 256 (live mode) | `pw-top -b` average over 5 minutes (see Section 1.6 Step 7) | Mean CPU < 40% | Mean CPU > 50% | Quantum 256 has 4x more FFT overhead (64 vs 16 partitions). CamillaDSP at chunksize 256 is 19.25%. PW overhead expected higher due to graph thread overhead. < 40% keeps system viable with Reaper (~30% CPU). > 50% leaves no headroom. |
| C-3 | Xrun count at quantum 1024 | `pw-top -b` xrun field + `journalctl --user -u pipewire --since` | 0 xruns in 5 minutes | Any xrun | Zero xruns is the baseline for production viability. CamillaDSP achieves this (US-001). |
| C-4 | Xrun count at quantum 256 | `pw-top -b` xrun field + journal | 0 xruns in 5 minutes | > 2 xruns | Quantum 256 is more demanding. 0 xruns = PASS. 1-2 xruns = MARGINAL (note and investigate). > 2 = FAIL. |
| C-5 | CPU stability (variance) | Standard deviation of 1-second CPU samples over 5 minutes | StdDev < 5% of mean | StdDev > 15% of mean | High variance indicates GC pauses, thermal throttling, or scheduling interference. CamillaDSP CPU is stable (< 1% variance in US-001). |
| C-6 | Thermal ceiling | Max CPU temperature during benchmark | Max temp < 75C | Max temp > 80C | A4 threshold. Sustained convolution may heat up more than CamillaDSP due to higher CPU. > 80C triggers ARM throttling, invalidating the CPU measurement. |
| C-7 | Filter-chain module loads successfully | `pw-dump` shows filter-chain nodes with correct port count | 4 convolver nodes visible, ports linked | Module fails to load or incorrect port count | Basic sanity: if the filter-chain config is rejected by PipeWire, the benchmark cannot proceed. |

**Marginal zone (20-30% for C-1, 40-50% for C-2):**

If a result falls in the marginal zone, it is recorded as MARGINAL (not PASS,
not FAIL). The marginal result is documented with:
- Exact CPU%, variance, and temperature data
- Comparison to CamillaDSP baseline at same quantum
- Assessment of whether optimization could close the gap (AE + Architect input)
- Recommendation for next steps

### 1.6 Execution Procedure

**Pre-requisites:** BM2-1 (filter-chain config) and BM2-2 (benchmark runner
script) must be complete and reviewed before execution.

#### Phase 0: Pre-flight (5 minutes)

```
Step 0.1: Record system state
    Command:  uname -r && pipewire --version && date -Iseconds
    Expected: 6.12.62+rpt-rpi-v8-rt, PipeWire 1.4.9
    On fail:  ABORT — wrong system state.

Step 0.2: Verify PipeWire scheduling
    Command:  chrt -p $(pidof pipewire)
    Expected: pid <N>'s current scheduling policy: SCHED_FIFO
              pid <N>'s current scheduling priority: 88
    On fail:  ABORT — F-020 override not active.

Step 0.3: Stop CamillaDSP
    Command:  sudo systemctl stop camilladsp
    Verify:   systemctl is-active camilladsp → "inactive"
    On fail:  Retry once. If still active, ABORT.

Step 0.4: Verify no audio clients running
    Command:  pw-cli list-objects Node | grep -E 'mixxx|reaper|camilladsp'
    Expected: No output (no audio clients)
    On fail:  Kill the offending process. If cannot, ABORT.

Step 0.5: Set CPU governor to performance (all 4 cores)
    Command:  for i in 0 1 2 3; do
                echo performance | sudo tee /sys/devices/system/cpu/cpufreq/policy$i/scaling_governor
              done
    Verify:   cat /sys/devices/system/cpu/cpufreq/policy*/scaling_governor
    Expected: All 4 read "performance"
    On fail:  Note and continue if cpufreq not available (governor already
              locked by kernel config). Record actual governor.

Step 0.6: Record thermal state
    Command:  awk '{print $1/1000}' /sys/class/thermal/thermal_zone0/temp
    Expected: < 60.0
    On fail:  Wait 2 minutes and re-check. If still > 60C after 3 attempts,
              ABORT — system is thermally stressed.

Step 0.7: Verify filter files exist
    Command:  ls -la <path-to-dirac-filters>/*.wav
    Expected: 4 WAV files, each 65,580 bytes (16,384 samples x 4 bytes + 44
              byte WAV header) or similar
    On fail:  ABORT — BM2-1 deliverable missing.

Step 0.8: Verify FFT engine (NEON optimization)
    Command:  ldd /usr/lib/*/spa-*/filter-graph/libspa-filter-graph.so
              nm -D /usr/lib/*/spa-*/filter-graph/libspa-filter-graph.so | grep -i pffft
    Expected: PFFFT symbols present (pffft_transform, pffft_new_setup or similar).
              If FFTW3 linked instead, note but do not abort — different FFT engine
              may explain performance differences.
    On fail:  Record output. If neither PFFFT nor FFTW3 found, note as anomaly.
              Continue — CPU measurement will reveal if SIMD path is active.
```

#### Phase 1: Filter-chain loading (5 minutes)

```
Step 1.1: Load filter-chain config
    Command:  <BM2-1 load command — either pw-filter-chain or PW conf.d>
    Expected: No errors on stdout/stderr.
    On fail:  Record error. Check PipeWire journal for details. ABORT if
              module fails to load.

Step 1.2: Verify filter-chain topology
    Command:  pw-dump | grep -A5 '"filter-chain"' (or equivalent)
              pw-link -l (verify port count)
    Expected: 4 convolver processing nodes visible. Correct number of input
              and output ports. Ports linked to USBStreamer (or appropriate
              sink).
    On fail:  Record topology. If ports missing or wrong count, ABORT —
              BM2-1 config is incorrect.

Step 1.3: Record PipeWire graph state
    Command:  pw-top -b -n 1
    Expected: Filter-chain nodes visible with near-zero CPU (no audio flowing).
    On fail:  Note and continue. Idle CPU > 2% is suspicious but not blocking.
```

#### Phase 2: Quantum 1024 benchmark (10 minutes)

```
Step 2.1: Set quantum to 1024
    Command:  pw-metadata -n settings 0 clock.force-quantum 1024
    Verify:   pw-metadata -n settings | grep clock.quantum
    Expected: "1024"
    On fail:  Retry. If quantum does not change, ABORT.

Step 2.2: Start audio source
    Command:  pw-play --target=<filter-chain-input-node> <test-file> &
              (or alternative audio source per BM2-2 script)
    Expected: Audio flows through filter-chain. pw-top shows non-zero CPU on
              filter-chain nodes.
    On fail:  If no audio flows, check pw-link for missing links. Fix and
              retry once. If still no flow, ABORT.

Step 2.3: Wait for stabilization (30 seconds)
    Rationale: Allow PipeWire graph to reach steady state. Initial quantum
    transitions and buffer fills can cause transient CPU spikes.

Step 2.4: Run benchmark measurement (5 minutes)
    Command:  <BM2-2 benchmark runner script>
              — must sample pw-top at 1-second intervals for 300 seconds
              — must record: per-sample CPU%, xrun count, temperature
              — must compute: mean, median, P95, P99, max, stddev
    Expected: Script runs to completion, outputs JSON or structured data.
    On fail:  If script crashes, record partial data and note deviation.

Step 2.5: Stop audio source
    Command:  kill %1 (or equivalent)

Step 2.6: Record results
    — Mean CPU%: ___
    — Median CPU%: ___
    — P95 CPU%: ___
    — P99 CPU%: ___
    — Max CPU%: ___
    — StdDev: ___
    — Xrun count: ___
    — Start temperature: ___
    — End temperature: ___
    — Max temperature: ___
```

#### Phase 3: Quantum 256 benchmark (10 minutes)

```
Step 3.1: Set quantum to 256
    Command:  pw-metadata -n settings 0 clock.force-quantum 256
    Verify:   pw-metadata -n settings | grep clock.quantum
    Expected: "256"

Step 3.2: Wait for graph re-stabilization (30 seconds)
    Rationale: Quantum change triggers buffer reallocation across all nodes.

Step 3.3: Start audio source
    Command:  pw-play --target=<filter-chain-input-node> <test-file> &
    Expected: Audio flows. pw-top shows non-zero CPU (higher than Phase 2
              due to 4x more partitions per second).

Step 3.4: Wait for stabilization (30 seconds)

Step 3.5: Run benchmark measurement (5 minutes)
    Command:  <BM2-2 benchmark runner script>
    Expected: Same output format as Phase 2.

Step 3.6: Stop audio source

Step 3.7: Record results (same fields as Step 2.6)
```

#### Phase 4: Teardown and verification (5 minutes)

```
Step 4.1: Unload filter-chain
    Command:  <unload command per BM2-1>
    Verify:   pw-dump shows no filter-chain nodes.

Step 4.2: Restore quantum to production default (256)
    Command:  pw-metadata -n settings 0 clock.force-quantum 256

Step 4.3: Restart CamillaDSP
    Command:  sudo systemctl start camilladsp
    Verify:   systemctl is-active camilladsp → "active"

Step 4.4: Restore CPU governor (if changed)
    Command:  for i in 0 1 2 3; do
                echo ondemand | sudo tee /sys/devices/system/cpu/cpufreq/policy$i/scaling_governor
              done

Step 4.5: Verify system restored to production state
    Command:  systemctl is-active camilladsp && chrt -p $(pidof camilladsp)
    Expected: active, SCHED_FIFO/80
```

**Fair comparison rules (AD Finding 7):**
- **Primary comparison:** BM-2 quantum 1024 vs BM-1 quantum 1024. Same
  processing granularity. This is the fair comparison.
- **Secondary comparison:** BM-2 quantum 256 vs US-001 T1c chunksize 256.
  Same processing granularity. Fair comparison for live mode.
- **Informational only:** BM-2 at any quantum vs US-001 T1a (chunksize 2048).
  CamillaDSP at chunksize 2048 processes 4x larger blocks, requiring only 16
  partitions vs 64 at chunksize 256. The lower CPU is a structural advantage of
  the larger chunksize, not an engine efficiency advantage. Do not use this
  comparison to claim CamillaDSP is "3x faster."

### 1.9 Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Filter-chain module fails to load (config syntax) | Medium | Benchmark cannot start | BM2-1 author validates config locally if possible; operator inspects PW journal for error details. Record and abort. |
| PFFFT not compiled with NEON (build flag missed) | Low | CPU dramatically higher than expected (non-SIMD path) | Pre-flight: `ldd` on filter-graph SPA plugin to verify PFFFT is linked. Check PW build flags if CPU is anomalously high (> 60%). |
| Thermal throttling during benchmark | Medium (summer/flight case) | CPU measurement invalid (throttled frequency) | Monitor temperature continuously. If > 80C, note and flag. Benchmark on open desk, not in flight case. |
| Quantum 256 causes xruns unrelated to convolution (PW overhead) | Medium | False FAIL on C-4 | Run a control measurement: quantum 256 WITHOUT filter-chain loaded. If xruns occur in control, they are not caused by convolution. Record control data. |
| pw-top sampling granularity too coarse | Low | CPU% imprecise | BM2-2 script should also use `top -bn1 -p $(pidof pipewire)` as cross-reference. Both methods record. |
| BM-1 not yet available for primary comparison | High (US-056 not started) | Cannot complete comparison table | Record BM-2 results now. Comparison table completed when BM-1 data is available. Note as limitation. |
