## F-210: PipeWire crashes ~20s after local-demo startup under normal web UI usage (VERIFIED FIXED)

**Filed:** 2026-03-29
**Severity:** P1 / High (US-075 local-demo unusable for owner testing)
**Status:** VERIFIED FIXED (2026-03-29 — QE exploratory: 18/18 pass, 200+ tab switches, no crashes)
**Affects:** `nix run .#local-demo` — PipeWire test environment stability
**Found by:** Owner (2026-03-29, manual testing)
**Related story:** US-075 (Local PW integration test env)

### Description

Running `nix run .#local-demo` and opening the web UI causes the entire stack
to crash within seconds of switching tabs. The crash sequence:

1. **PipeWire dies first** — all streams simultaneously transition
   `Streaming -> Unconnected` at `16:20:49-50` (~20s after startup)
2. **GM's gain integrity pw-dump fails** — "can't connect: Host is down"
   (exit status 255) at `16:20:57`
3. **All services cascade-shutdown** — GM, signal-gen, level-bridge (x3),
   pcm-bridge all receive shutdown signals at `16:21:27`

The owner observed: Links 0/0, 502 on config tab, 502 on
`/api/v1/graph/topology`. "Things crashed just from switching tabs."

### Evidence from logs

**PipeWire stderr** (`/tmp/pw-test-stderr.log`):
Continuous `out of buffers on port 0 2` warnings from `spa.audioconvert`
every 2 seconds from startup through crash. This indicates buffer pool
exhaustion in the audio processing graph.

**Reconciler storm** (see F-211):
278 "Required link endpoint missing" warnings in the first 2 seconds of GM
startup. The reconciler fires on every node-add registry event while waiting
for port enumeration, with no backoff or deduplication. This likely
contributes to PW resource exhaustion.

**Link thrash with WirePlumber** (lines 730-753):
After GM creates measurement-mode links, WirePlumber creates competing links
(id 218/219 at `16:20:44`). GM immediately destroys them as "stale." WP
recreates them. This create/destroy cycle repeats 4 times in <1 second,
generating rapid link churn on the PW graph.

**Web UI thermal polling** (lines 782-805):
`/api/v1/thermal/status` polled ~20 times in 17 seconds (~1.2 req/sec).
While not directly causal, this adds unnecessary load to an already stressed
local-demo environment.

### Probable Cause

The PipeWire test environment (null sinks, no real hardware) exhausts its
buffer pool under the combined load of:
1. Reconciler link churn (278 retry warnings + 4 link create/destroy cycles)
2. Filter-chain convolver processing (even with dirac passthrough)
3. 7 PW client streams (signal-gen, 3x level-bridge, pcm-bridge, 2x convolver)

The `out of buffers` warning from startup suggests the test environment's
default buffer configuration is insufficient for the number of active streams
and link operations.

### Suggested Investigation

1. Check if local-demo PipeWire config has adequate buffer allocation for
   7+ concurrent streams with null sinks
2. Determine if the reconciler storm (F-211) causes pathological PW API
   call volume that triggers buffer exhaustion
3. Test with fewer concurrent services to isolate the trigger
4. Compare buffer config between local-demo and production Pi
