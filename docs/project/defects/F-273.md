## F-273: nixos-rebuild switch fails -- disko.nix and sd-image.nix produce incompatible partition layouts

**Filed:** 2026-04-06
**Severity:** Critical
**Status:** RESOLVED
**Affects:** Deployment workflow (nixos-rebuild switch on sd-image-based system)
**Found in:** `nix/nixos/disko.nix`, `nix/nixos/sd-image.nix`
**Found by:** Owner (production incident 2026-04-06)
**Related:** F-266 (neededForBoot on /boot/firmware)
**Fixed in:** defect/F-273-partition-label-sync

### Description

Running `nixos-rebuild switch --flake .#mugge-deploy` on a Pi booted from an
SD card image fails. The two deployment paths (`sd-image.nix` and `disko.nix`)
produce incompatible partition layouts:

- **sd-image.nix** creates an MBR partition table with filesystem labels
  `FIRMWARE` (vfat) and `NIXOS_SD` (ext4), referenced via `/dev/disk/by-label/`.
- **disko.nix** creates a GPT partition table with GPT partition labels
  `disk-main-firmware` and `disk-main-root`, referenced via
  `/dev/disk/by-partlabel/`. The `neededForBoot = true` flag (F-266)
  attempts the firmware mount in the initrd, which fails when the device
  path doesn't exist on an MBR-partitioned card.

### Root Cause

Two independent problems:

1. **Partition table type mismatch:** sd-image produces MBR, disko produces
   GPT. The fileSystems entries reference different device path namespaces
   (`by-label/` vs `by-partlabel/`).

2. **No shared filesystem label convention:** The two paths don't agree on
   filesystem labels, so switching from one to the other changes the fstab
   entries and triggers NixOS `switch-to-configuration` errors.

### Fix

Replaced `sd-image.nix` (upstream NixOS module) with a custom MBR image
builder in `repart-image.nix`. Both deployment paths (SD card image and
nixos-anywhere) now produce identical MBR partition layouts with the same
filesystem labels. MBR was chosen over GPT because it is proven to boot
reliably on Pi 4 — VideoCore scans for FAT partitions by MBR type
(0x0B/0x0C) without GUID issues. systemd-repart was considered but is
GPT-only, so a custom builder using sfdisk + mkfs was written instead.

1. **New `repart-image.nix`:** Custom MBR image builder using `sfdisk` for
   partition table, `mkfs.vfat` (via mtools) for firmware, and nixpkgs
   `make-ext4-fs.nix` for root. Partition layout: 256 MiB FAT32 (type 0x0C,
   W95 FAT32 LBA, label `FIRMWARE`) + rest ext4 (type 0x83, bootable, label
   `NIXOS_SD`). Firmware files populated from firmware.nix. Boot files
   (extlinux) and Nix store closure on root partition. First-boot expansion
   via `boot.postBootCommands`. ext4 created with `-O ^orphan_file` because
   U-Boot 2025.10 cannot read ext4 with the orphan_file INCOMPAT feature.

2. **Updated `disko.nix`:** Switched from `type = "gpt"` (ESP) to disko's
   legacy `type = "table"` with `format = "msdos"` for MBR. Added `extraArgs`
   to set filesystem labels (`-n FIRMWARE`, `-L NIXOS_SD`) and `-O
   ^orphan_file` matching repart-image.nix. Added `lib.mkForce` overrides on
   fileSystems to use `by-label/` instead of disko's default paths.

3. **Shared `fileSystems` in `configuration.nix`:** Both configs share the same
   fileSystems declarations (root via `by-label/NIXOS_SD`, firmware via
   `by-label/FIRMWARE` with `nofail,noauto`).

4. **Updated `firmware.nix`:** Activation script uses `mountpoint -q` instead
   of `[ -d ]` to detect mount state. Mounts on-demand when firmware partition
   isn't auto-mounted (`nofail,noauto`). Added DT overlay `disable-sd-uhs` to
   prevent U-Boot MMC voltage switch failure, plus `disable-bt` overlay.

5. **Updated `flake.nix`:** mugge config imports `repart-image.nix` instead of
   `sd-image.nix`. Image output uses `system.build.image` instead of
   `system.build.sdImage`.

### Files Changed

- `nix/nixos/repart-image.nix` — NEW: custom MBR image builder replacing sd-image.nix
- `nix/nixos/disko.nix` — MBR table type + filesystem labels + mkForce fileSystems
- `nix/nixos/configuration.nix` — shared fileSystems declarations
- `nix/nixos/firmware.nix` — mountpoint check + on-demand mount + DT overlays
- `flake.nix` — repart-image.nix import, system.build.image output

### Impact

**Critical.** Without this fix, the standard deployment workflow (image for
initial install, nixos-rebuild for updates) is broken. The Pi cannot be
updated remotely after initial SD card imaging.
