## F-290: E2E link-count assertion incorrect — DJ has more links than live

**Filed:** 2026-04-06
**Severity:** Low
**Status:** OPEN
**Affects:** E2E test `test_live_has_at_least_as_many_links_as_dj`
**Related:** F-272 (reconciler settlement), US-140 (epoch-based settlement)

### Description

The `test_live_has_at_least_as_many_links_as_dj` E2E test asserts
`live_links >= dj_links`, but DJ mode legitimately has more links than
live mode because DJ routes Mixxx (8-channel) through the convolver.
Live mode has a simpler routing topology.

Observed: DJ=43 links, live=25 links. The reconciler correctly settled
both topologies. The test assertion is wrong, not the reconciler.

This was previously misattributed to F-272 (reconciler race). The xfail
has been updated to reference F-290.

### Root Cause

The test assumes live mode has at least as many PipeWire links as DJ
mode. This is incorrect: DJ mode routes Mixxx (8 output channels)
through the convolver chain, creating significantly more links than
live mode which has a simpler routing path.

### Evidence

T2 run (2026-04-06):
- DJ topology: 43 links
- Live topology after `await_settled`: 25 links
- `await_settled` returned `ok: true` (settlement is correct)

### Resolution Criteria

1. Correct the test assertion: either reverse it (`dj >= live`), or
   assert each mode has its expected link count independently
2. `test_live_has_at_least_as_many_links_as_dj` passes without xfail
