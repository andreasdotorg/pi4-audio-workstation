## F-290: await_settled reports settled before live mode links are fully created

**Filed:** 2026-04-06
**Severity:** Medium
**Status:** OPEN
**Affects:** E2E test stability, mode-switching operations
**Related:** F-272 (reconciler settlement), US-140 (epoch-based settlement)

### Description

After switching from DJ to live mode via the test-tool API (which uses
`_gm_set_mode` with `await_settled`), the reconciler reports settled but
the live mode link count is significantly lower than DJ mode: live=25
vs dj=43.

The `test_live_has_at_least_as_many_links_as_dj` E2E test asserts
`live_links >= dj_links`, which fails because the settlement epoch
advances before all live mode links are created.

### Root Cause

The GM reconciler's `settled_epoch` advances when `desired == actual`
link counts match, but the desired count for live mode may be computed
before the full live link set is requested. This causes `await_settled`
to return prematurely with an incomplete topology.

This is distinct from F-272 (sleep-based polling race). F-272 was about
callers not waiting at all. F-290 is about `await_settled` returning
too early because the settlement condition is satisfied prematurely.

### Evidence

T2 run (2026-04-06):
- DJ topology: 43 links
- Live topology after `await_settled`: 25 links
- `await_settled` returned `ok: true`

### Resolution Criteria

1. `await_settled` does not return until the full live link set is created
2. `test_live_has_at_least_as_many_links_as_dj` passes without xfail
3. Or: the test assertion is corrected if live mode genuinely has fewer
   links than DJ (Mixxx source links are DJ-only)
