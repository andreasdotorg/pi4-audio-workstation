## F-262: E2E measurement session fails non-deterministically on filter verification

**Filed:** 2026-04-04
**Severity:** Medium
**Status:** FIX IN PROGRESS (rework per owner feedback)
**Affects:** CI T2 (E2E tests with local-demo PipeWire stack)
**Related:** CI run #335 (main at `fc40435`, PW 1.4.9)
**Origin:** CI test failure (owner directive: all CI failures tracked as defects)

### Description

`test_full_session_api` and `test_full_session_browser` in
`src/web-ui/tests/e2e/test_measurement.py` assert `state == "complete"`
but the measurement session sometimes ends in "error" because filter
verification fails on room-sim generated IRs.

The initial investigation incorrectly suspected UMIK-1 signal delivery
(silent microphone in measurement mode). **Signal delivery is confirmed
working** — gain calibration and sweep capture both succeed. The failure
occurs downstream in the `filter_gen` stage.

This is NOT a flaky test — QE confirms the failure is consistent and
needs investigation.

### Previous Fix (rejected by owner)

The first fix introduced `PI4AUDIO_FILTER_MINPHASE_FATAL=0` env var to
make min-phase verification non-fatal in local-demo. Owner rejected this
approach: masking the min-phase failure prevents diagnosis of a potential
real computational bug. The room-sim should not affect the *computed
correction filter's* minimum-phase property — the filter generation
pipeline takes an IR as input and produces a minimum-phase FIR regardless
of the IR's origin.

### Current Fix (rework)

Remove the `PI4AUDIO_FILTER_MINPHASE_FATAL` env var entirely. Min-phase
verification is always enforced. Changes:

- `session.py`: Remove `_FILTER_MINPHASE_FATAL` variable and all
  conditional logic. Min-phase failure is always fatal.
- `local-demo.sh`: Remove `PI4AUDIO_FILTER_MINPHASE_FATAL=0` export.
- `test_measurement.py`: Update docstrings to reflect always-enforced
  min-phase check.

After the F-265 fix (gate open with correct venue instead of silence),
re-run T2 to determine whether min-phase now passes. If it still fails,
add diagnostics (log energy_ratio, peak_fraction, filter stats) and file
a new defect for the computational issue.

### Partial Resolution (D-249)

D-249 (GM quantum init fix) resolved the audio flow symptom. The
`test_level_bridge_sw_has_signal` test in `test_audio_flow.py` now passes
consistently — the GM's explicit quantum initialization stabilized signal
delivery through the pipeline. xfail marker removed from that test.

The filter verification xfail in `test_measurement.py` (2 tests) remains
OPEN — that is a separate issue where the E2E measurement filter
verification step fails. This defect's status stays OPEN for those
remaining symptoms.

### Impact

T2 gate `test_full_session_api` and `test_full_session_browser` must pass
with min-phase always enforced. If they fail after the F-265 fix, a new
defect will be filed with diagnostic data to identify the root cause.
