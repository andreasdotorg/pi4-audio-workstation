## F-275: Spectrum display gone after US-131 merge -- PCM bridge monitor connection refused

**Filed:** 2026-04-06
**Severity:** High (user-visible regression)
**Status:** RESOLVED
**Affects:** Dashboard spectrum display, PCM bridge monitor connection
**Found by:** Owner (Gabriela Bogk, 2026-04-06)
**Introduced by:** PR #20 (US-131, dynamic port allocation)
**Related:** US-131 (parallel local-demo), pcm-bridge

### Description

After PR #20 (US-131, dynamic port allocation) merged, the spectrum display on
the web UI dashboard is gone. The PCM bridge cannot connect to its monitor
source. Console log:

```
2026-04-06 16:16:11,795 WARNING PCM relay: giving up after 15 attempts (source=monitor): [Errno 111] Connection refused
```

This is a post-merge regression -- spectrum display was working before PR #20.

### Root Cause

US-131 commit `7626b16f` changed pcm-bridge in local-demo from hardcoded port
9090 to dynamic port 0 with `--port-file`. The web-ui startup in
`scripts/local-demo.sh` did not set `PI4AUDIO_PCM_SOURCES`, so `main.py` fell
back to `_DEFAULT_PCM_SOURCES = {"monitor": "tcp:127.0.0.1:9090"}` -- the old
hardcoded port that no longer matched the dynamically assigned one.

Production (NixOS) was unaffected because `production.nix` uses fixed port 9090.

### Fix

Added `PI4AUDIO_PCM_SOURCES` to `scripts/local-demo.sh` web-ui startup with
the dynamic `PORT_PCM` value. Added regression E2E test
(`TestPcmWebSocketRelay`) that connects to `/ws/pcm` and verifies binary PCM
frames arrive -- this test would have caught the regression.

### Verification

1. Dashboard spectrum display shows live data
2. No "Connection refused" warnings in PCM relay logs
3. PCM bridge monitor source connects successfully
4. `TestPcmWebSocketRelay::test_ws_pcm_receives_binary_data` passes in E2E
