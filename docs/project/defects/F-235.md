## F-235: Measurement mode still doesn't work in local-demo

**Filed:** 2026-03-31 (session 6)
**Severity:** High (measurement pipeline non-functional locally)
**Status:** RESOLVED (session 7, commits `94fbf2a`, `1bb85ec`)
**Affects:** Local-demo measurement mode, measurement session workflow
**Found by:** Owner (session 6, morning testing)

### Description

Starting a measurement session in local-demo fails. pw-record captures
zero audio because its Stream/Input/Audio node never activates when
WirePlumber policy.standard is disabled (D-039/F-210: GM is sole link
manager).

### Root Cause

pw-record creates a PipeWire stream node with media.class
Stream/Input/Audio. In production, WP policy.standard creates session
items for stream nodes, which triggers port negotiation (suspended ->
idle -> running) and auto-linking via `--target`. In local-demo,
policy.standard is disabled — only policy.node is enabled (creates
session items for port activation but does NOT create links).

With the original `--target <node_name>`, pw-record relies on WP to
both activate the stream AND create the link. Without policy.standard,
neither happens — the node stays suspended with 0 ports.

### Fix (session 7)

Use `--target 0` (no auto-link) with explicit node activation
properties that force port creation without WP policy.standard:

```
pw-record --target 0 \
  -P "node.always-process=true,node.passive=true,node.want-driver=true" \
  --rate 48000 --channels 1 --format f32 output.wav
```

After pw-record creates ports (1s settle), manually link the source
to pw-record via `pw-link`:

```
pw-link alsa_input.usb-miniDSP_Umik-1:capture_FL pw-record:input_MONO
```

This works in both environments:
- **Local-demo** (WP policy.standard off): Properties force port
  activation, pw-link creates the connection.
- **Production** (WP policy.standard on): pw-link is idempotent —
  if the link already exists, it returns "File exists" (harmless).

### Approaches Investigated

1. **`--target <node_name>`** (original): Fails — requires WP
   policy.standard for both activation and linking.
2. **`-P "node.always-process=true"` with `--target <name>`**: Fails —
   target auto-linking still requires policy.standard.
3. **`--target 0` with activation properties**: Works — forces port
   creation without auto-linking, then pw-link creates the connection.
4. **pcm-bridge TCP fallback** (reverted `ce96012`): Owner rejected —
   bypasses pw-record instead of fixing the real problem.
5. **Re-enable WP policy.standard**: Rejected — conflicts with GM
   reconciler (D-039, F-210).
6. **Add pw-record to GM routing table**: Rejected by architect —
   pw-record is ephemeral, doesn't belong in GM's mode-based topology.

### Verification

E2E test with real pw-record capture:
- pw-record creates ports (`pw-record:input_MONO`)
- pw-link connects UMIK-1 to pw-record
- Audio captured: -31.1 dBFS RMS, -19.2 dBFS peak (191,488 frames)
- Unit tests: 36/36 measurement tests pass
- QE UI verification: Measure tab loads, wizard advances to gain cal
