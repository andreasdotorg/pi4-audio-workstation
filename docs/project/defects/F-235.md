## F-235: Measurement mode still doesn't work in local-demo (FIX IMPLEMENTED)

**Filed:** 2026-03-31 (session 6)
**Severity:** High (measurement pipeline non-functional locally)
**Status:** Fix implemented — awaiting architect review + end-to-end verification
**Affects:** Local-demo measurement mode, measurement session workflow
**Found by:** Owner (session 6, morning testing)

### Root Cause

`pw-record` relies on WirePlumber's `policy.standard` to activate its stream
node and create the link to the target (UMIK-1).  In local-demo,
`policy.standard` is disabled (F-210, D-039: GM is sole link manager).
Without WP linking policy:

1. pw-record creates a PipeWire node (media.class = Stream/Input/Audio)
2. The node stays in `suspended` state with 0 ports
3. No link is created to the UMIK-1 source
4. pw-record captures zero audio (80-byte WAV header only)
5. `gain_calibration.py` calls `np.max(np.abs(empty_array))` and crashes

This is a fundamental incompatibility: pw-record is a **stream** node that
needs WP session item management for port activation.  Device-class nodes
(filter-chain, convolver, loopback) get their ports from PipeWire directly
and don't need WP policy.

### Approaches Investigated

1. **`pw-record -P "node.always-process=true"`** — No effect.  The property
   is a hint for WP's session item, which isn't created without `policy.node`'s
   linking hooks.
2. **`pw-record -P "node.driver=true"`** — No effect.
3. **`pw-cli create-link` with auto-port** — Rejected: "unknown input port -1".
   PW link-factory requires explicit port IDs, but pw-record has none.
4. **`pw-link` by port name** — Failed: pw-record ports don't exist.
5. **Enable partial WP linking policy** — Rejected: conflicts with D-039.
6. **pw-loopback capture bridge** — Same pw-record problem on the source side.

### Fix (session 7)

Modified `src/measurement/pw_capture.py` to add pcm-bridge TCP fallback:

- `start_capture()` starts pw-record, then checks if ports appeared
  (via `pw-link -i`).
- If no ports (WP policy disabled): kills pw-record, starts
  `PcmBridgeCapture` — reads from pcm-bridge TCP (port 9090), extracts
  UMIK-1 channel (ch3), writes WAV.
- `stop_capture()` handles both pw-record (`Popen`) and `PcmBridgeCapture`.
- On production (WP policy.standard active), pw-record works normally —
  fallback never triggers.

### Verification Status

- **Mechanical capture works**: PcmBridgeCapture connects, parses v2 frames,
  extracts channel, writes WAV, load_wav() reads it. Duration and frame
  count correct.
- **End-to-end NOT verified**: pcm-bridge streams all zeros because
  signal-gen TCP handler is stuck with CLOSE-WAIT connections (separate
  issue, not F-235). Full end-to-end test requires working signal-gen.
