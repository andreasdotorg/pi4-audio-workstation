## F-208: D-029 gain staging validator maps midrange/tweeter to "satellite" — skips role-specific headroom checks (OPEN)

**Filed:** 2026-03-29
**Severity:** Medium (safety-adjacent — headroom validation silently skipped for mid/HF channels in 3-way profiles)
**Status:** OPEN
**Affects:** `speaker_routes.py:632-635`, D-029 gain staging validation
**Found by:** Audio Engineer (pre-existing inconsistency, flagged 2026-03-29)
**Related story:** US-091 (N-way crossover support), US-089 (speaker config management)

### Description

The D-029 gain staging validator at `speaker_routes.py:632-635` unconditionally
maps all non-subwoofer roles (`satellite`, `midrange`, `tweeter`, `fullrange`)
to the `"satellite"` key in the profile's `gain_staging` dict:

```python
if role in ("satellite", "midrange", "tweeter", "fullrange"):
    gs_group = gain_staging.get("satellite", {})
```

When a 3-way profile has dedicated role-specific `gain_staging` sections (e.g.,
`gain_staging.midrange` and `gain_staging.tweeter` with their own `headroom_db`
values), the validator ignores them and always reads `gain_staging.satellite`.
If no `satellite` key exists (because the profile uses role-specific keys
instead), `gs_group` becomes `{}`, `headroom_db` defaults to `0`, and the
headroom check effectively passes for any `max_boost` value — silently skipping
D-029 validation for those channels.

This is the same class of bug as F-195 (config generator) and F-197 (target
gains), but in the validation path rather than the generation or activation
paths.

### Impact

- 3-way profiles with role-specific gain_staging sections skip D-029 headroom
  validation for midrange and tweeter channels
- Operator receives no warning when headroom is insufficient for mid/HF drivers
- Safety-adjacent: D-029 headroom checks exist to prevent driver damage from
  excessive boost. Skipping them for mid/HF drivers in 3-way configs defeats
  the safety intent.

### Fix

Apply the same role-lookup pattern used in the F-197 fix (`fa94ee5`): look up
`gain_staging.get(role, gain_staging.get("satellite", {}))` so role-specific
sections are preferred over the satellite fallback. The fix at lines 632-633
should become:

```python
if role in ("satellite", "midrange", "tweeter", "fullrange"):
    gs_group = gain_staging.get(role, gain_staging.get("satellite", {}))
```

This preserves backward compatibility (profiles with only a `satellite` section
still work) while correctly using role-specific sections when present.
