## F-252: WebSocket/PCM capture timeouts in CI browser tests

**Filed:** 2026-04-04
**Severity:** High (upgraded from Medium per AD review — entire WebSocket/PCM capture test category non-functional in CI)
**Status:** OPEN
**Affects:** `tests/integration/test_capture_spectrum.py` (6 failures)
**Related:** CI run 23978434071
**Origin:** CI test failure (owner directive: all CI failures tracked as defects)

### Description

6 tests in `test_capture_spectrum.py` fail with 10-second timeout waiting for
WebSocket connection state changes. All failures follow the same pattern:
the test opens a WebSocket connection to the spectrum endpoint and waits for
data, but the connection either never establishes or never receives the
expected state transition within the 10-second timeout.

### Root Cause (suspected)

Three possible causes:

1. **Race condition:** The test starts the WebSocket connection before the
   backend server has fully initialized the PCM capture pipeline. On faster
   local machines, the server is ready before the client connects. On CI
   runners (shared, variable CPU), the server may not be ready in time.

2. **Insufficient timeout:** GitHub-hosted runners have variable performance
   (shared infrastructure). The 10-second timeout may be adequate locally
   but marginal on CI. Cold Nix builds and concurrent CI load add latency.

3. **Mock backend divergence:** Tests run with `PI_AUDIO_MOCK=1` in CI. If
   the mock does not accurately simulate WebSocket connection lifecycle
   timing, tests may pass locally (where the mock is fast) but fail in CI
   (where setup overhead exposes the timing gap). Investigate whether the
   mock's WebSocket behavior matches real PipeWire's.

### Fix

1. **Add server readiness check:** Before opening the WebSocket, poll a
   health endpoint (e.g., `/api/v1/status`) until the server reports ready.
   This eliminates the race condition regardless of machine speed.

2. **Increase timeout for CI:** Set CI-specific timeout (e.g., 30s) via
   environment variable. Keep local timeout at 10s for fast feedback.

Option 1 is recommended — the readiness check eliminates the root cause.
