## F-252: WebSocket/PCM capture timeouts in CI browser tests

**Filed:** 2026-04-04
**Severity:** High (upgraded from Medium per AD review — entire WebSocket/PCM capture test category non-functional in CI)
**Status:** RESOLVED
**Affects:** `tests/integration/test_capture_spectrum.py` (6 failures)
**Related:** CI run 23978434071
**Origin:** CI test failure (owner directive: all CI failures tracked as defects)

### Description

6 tests in `test_capture_spectrum.py` fail with 10-second timeout waiting for
WebSocket connection state changes. All failures follow the same pattern:
the test opens a WebSocket connection to the spectrum endpoint and waits for
data, but the connection either never establishes or never receives the
expected state transition within the 10-second timeout.

### Root Cause

The `/ws/pcm/{source}` and `/ws/pcm` WebSocket endpoints had no mock mode
path. In mock/CI mode (`PI_AUDIO_MOCK=1`), they attempted to TCP-connect to
pcm-bridge (a real-hardware service) which doesn't exist in the test
environment. The `_pcm_tcp_relay` function retried the TCP connection 15
times with 1-second backoff, then silently closed the WebSocket — never
reaching the "streaming" state that the tests wait for.

### Resolution

Added `_mock_pcm_stream()` to `app/main.py`: when `MOCK_MODE` is true, the
PCM WebSocket endpoints generate synthetic v2 wire-format frames (440 Hz
sine at -40 dBFS) instead of attempting TCP relay. The mock stream uses the
same binary format as real pcm-bridge, so the JS FFT pipeline processes it
normally and the spectrum canvas renders visible data.

Both `/ws/pcm/{source}` and `/ws/pcm` check `MOCK_MODE` before deciding
whether to use mock stream or TCP relay. All 6 xfail markers removed.
