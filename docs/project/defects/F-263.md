## F-263: GM reconciler timing — rapid DJ->live switch link count drops

**Filed:** 2026-04-04
**Severity:** Medium
**Status:** OPEN
**Affects:** E2E test `test_production.py::TestLinkCountsPerMode::test_live_has_at_least_as_many_links_as_dj`
**Related:** F-262 (GM reconciler not linking on CI), PipeWire link topology

### Description

When switching from DJ to live mode rapidly, the GM reconciler may not fully
stabilize the link topology before the test asserts link counts. Live mode
should have at least as many links as DJ mode, but during the transition the
link count temporarily drops as old links are removed before new ones are
created.

The test polls for stabilization (6 rounds x 2s), but the reconciler can
take longer to complete all link operations, especially under load.

### Root Cause

The GM reconciler destroys old-mode links before creating new-mode links.
During this window, `GET /api/v1/graph/topology` reports a lower link count.
The test's polling loop may not be long enough for the reconciler to finish,
or PipeWire link creation may be delayed by node registration timing.

### Failing Test

- `test_live_has_at_least_as_many_links_as_dj` — asserts
  `live_links >= dj_links` after mode switch; currently xfail'd

### Fix

Options:
1. **Increase poll timeout** in the test (simple, may mask deeper issues)
2. **Add reconciler completion signal** — GM could expose a
   `/api/v1/graph/reconcile-complete` endpoint or include a
   `reconcile_pending: false` field in topology responses
3. **Atomic mode transition** — destroy old links and create new links in a
   single reconciler pass rather than sequential destroy-then-create
