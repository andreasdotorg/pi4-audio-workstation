## F-247: pcm-bridge 4ch/8ch channel mismatch produces zero audio in local-demo (OPEN)

**Filed:** 2026-03-31 (session 7)
**Severity:** Medium
**Status:** Open
**Affects:** Spectrum analyzer display in local-demo (UMIK-1 and Monitor sources show "No capture signal")
**Found by:** worker-1 during F-236 investigation (session 7)

### Description

pcm-bridge is started with `--channels 4` but the PipeWire filter-chain
graph is 8-channel after US-115 (8ch convolver extension). PipeWire's
audioconvert adapter fails format negotiation between pcm-bridge (4ch
F32LE) and the graph (8ch F32P dsp format). Result: pcm-bridge receives
heartbeat frames with `frame_count=0` and `clock_pos=0`.

Level-bridges (8ch) work fine — only pcm-bridge is affected.

### Evidence

- PW stderr: `"out of buffers on port 0 2"` from spa.audioconvert
- PW stderr: `"no matching params"` — format negotiation failure
- pw-top: pcm-bridge QUANT=0, RATE=0 (not receiving data)
- level-bridge-sw (8ch) and level-bridge-hw-out (8ch): data flows correctly

### Root Cause

US-115 extended the convolver to 8 channels but pcm-bridge startup was
not updated to match. The channel count mismatch causes PipeWire's
format negotiation to fail silently — pcm-bridge connects but receives
no audio data.

### Fix

Update pcm-bridge startup in `local-demo.sh` to use `--channels 8` (or
match graph channel count dynamically). May also need changes in
production `pipewire.nix` pcm-bridge config.
