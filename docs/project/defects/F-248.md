## F-248: Spectrum display hiccups — event loop jitter in PCM relay (OPEN, POSTPONED)

**Filed:** 2026-04-03
**Severity:** Medium
**Status:** OPEN — analysis complete, fix postponed per owner
**Affects:** Web UI spectrum analyzer (all tabs using PCM WebSocket)
**Related:** F-105 (surface fix, marked RESOLVED but root cause persists), F-098 (original hiccup)

### Description

The real-time spectrum display has intermittent visual "hiccups" — frames
arriving in bursts rather than smoothly at the PipeWire graph rate. F-105
addressed a surface-level timing issue but the underlying root cause is
deeper: the PCM TCP-to-WebSocket relay in `src/web-ui/app/main.py`
(`_pcm_tcp_relay`) suffers from event loop contention.

### Root Cause (confirmed, session 9 single-clock analysis)

Three compounding factors:

1. **Event loop contention:** `asyncio.to_thread(tcp_sock.recv, 65536)`
   serializes through the default thread pool executor. pw-dump subprocess
   calls (every 5s), HTTP handlers, and other async tasks compete for the
   same event loop turns. During pw-dump execution, the relay stalls.

2. **Missing TCP_NODELAY:** The TCP client socket to pcm-bridge does not
   set `TCP_NODELAY`. Nagle's algorithm batches small PCM frames into
   larger TCP segments, adding up to 40ms of jitter.

3. **Clock domain mismatch:** pcm-bridge emits frames at the PipeWire
   graph rate (256 quantum @ 48kHz = ~187 frames/sec). The browser renders
   via `requestAnimationFrame` at 60Hz. Without clock-aware rendering,
   frames bunch up ~3-at-a-time then starve for 1-2 frames.

### Rejected Approaches

- **Frame decimation** (drop frames to match 60Hz): Owner rejected —
  breaks single-clock design principle. All frames carry timing information.
- **Time-based batching**: Owner rejected — same reasoning.

### Approved Fix Options (when work resumes)

| Option | Description |
|--------|-------------|
| **A: Dedicated relay thread** | Move TCP->WebSocket relay out of asyncio entirely. Dedicated thread with its own socket reads, pushes to WebSocket via `queue.Queue`. Add `TCP_NODELAY`. Eliminates server-side jitter. Does not solve browser 60/187Hz mismatch. |
| **B: Browser clock-aware rendering** | Fix `TCP_NODELAY`. Browser-side JS reconstructs timing from frame metadata (graph_pos/nsec timestamps in v2 header) and interpolates/renders at the correct rate. Honors single-clock principle end-to-end. |
| **A+B combined** | Recommended. Clean delivery (A) + correct visual timing (B). |

### Key Principle

The PipeWire graph runs on a single clock (quantum 256 @ 48kHz). Every PCM
frame carries timing information (graph_pos, nsec in v2 header). Any fix
must preserve the clock chain end-to-end — no decimation, no batching, no
frame dropping.
