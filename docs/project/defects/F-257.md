## F-257: CI flake.nix nixConfig warnings — untrusted substituters ignored

**Filed:** 2026-04-04
**Severity:** Low
**Status:** OPEN
**Affects:** `.github/workflows/ci.yml` (all jobs), `flake.nix` (`nixConfig` block)
**Related:** CI pipeline, Cachix configuration
**Origin:** CI warning (owner directive: all CI issues tracked)

### Description

CI runs emit warnings:

```
ignoring untrusted flake configuration setting 'extra-substituters'
ignoring untrusted flake configuration setting 'extra-trusted-public-keys'
```

The `nixConfig` block in `flake.nix` declares Cachix as an extra
substituter with its public key. However, the Nix daemon on GitHub-hosted
CI runners does not trust the flake's `nixConfig` settings (the invoking
user is not in `trusted-users`). The settings are silently ignored.

This is harmless — `cachix-action@v17` configures the Cachix substituter
at the daemon level, so binary cache hits still work. The `nixConfig`
block is redundant on CI and produces noisy warnings.

### Root Cause

Nix daemons only apply `nixConfig` from flakes when the invoking user is
in `trusted-users`. GitHub-hosted runners run Nix as a non-trusted user.
The `cachix-action` works because it modifies `/etc/nix/nix.conf` directly
(daemon-level configuration), bypassing the trust check.

### Fix

Remove the `nixConfig` block from `flake.nix`. It serves no purpose:
- **CI:** `cachix-action@v17` handles substituter configuration at the
  daemon level.
- **Local dev:** Owner's `/etc/nix/nix.conf` already has `trusted-users`
  and Cachix substituter configured.

The `nixConfig` block was added as a convenience but is redundant in both
environments and produces warnings in CI.
