## F-264: Mode badge/switching browser tests fail — WebSocket relay timeout

**Filed:** 2026-04-04
**Severity:** Medium
**Status:** OPEN
**Affects:** E2E tests in `test_production.py::TestModeBadgeDefault` and `TestModeSwitching`
**Related:** F-228 (mode badge default, RESOLVED), F-262 (GM reconciler linking)

### Description

Three browser-based E2E tests fail because the WebSocket relay does not
deliver GM mode state to the browser within the test timeout window. The
tests wait for DOM elements (`#sb-mode`, `#sys-mode`) to show the expected
mode text, but the WebSocket update arrives too late or not at all in the
local-demo E2E environment.

### Failing Tests

1. **`TestModeBadgeDefault::test_mode_badge_shows_standby_in_browser`** —
   waits for `#sb-mode` to show "STANDBY" after initial page load; times
   out because WebSocket has not yet delivered GM state.

2. **`TestModeBadgeDefault::test_system_view_mode_shows_standby`** — waits
   for `#sys-mode` to contain "STANDBY"; same WebSocket delivery issue.

3. **`TestModeSwitching::test_mode_badge_updates_in_browser_after_dj_switch`** —
   switches mode to DJ via API, waits for `#sb-mode` to show "DJ"; the
   WebSocket relay does not push the update within the 15s timeout.

### Root Cause (suspected)

The web UI receives GM state via a WebSocket connection to the backend.
In the local-demo E2E stack, the WebSocket connection may not be established
or may reconnect during the test window. Possible sub-causes:

1. **WebSocket connection race:** The browser opens the page, but the
   WebSocket connection is not established before the test starts waiting
   for mode data.
2. **GM state push timing:** After an API mode switch, the GM notifies the
   backend via RPC, but the backend's WebSocket broadcast to connected
   clients may be delayed.
3. **Local-demo startup ordering:** If GM reports mode before the web UI
   WebSocket handler is ready, the initial mode is missed and only the
   next push (on mode change) delivers state.

### Fix

Options:
1. **Increase WebSocket data timeout** in tests (currently 10-15s)
2. **Add explicit WebSocket readiness check** — wait for at least one
   successful WebSocket message before asserting mode state
3. **Backend: send current state on WebSocket connect** — ensure new
   WebSocket connections immediately receive the current GM state (not
   just diffs/changes)
