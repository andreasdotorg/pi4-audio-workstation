## F-209: US-044 watchdog + gain integrity assume gain builtins are separate PW nodes — both non-functional on real hardware (VERIFIED)

**Filed:** 2026-03-29
**Severity:** P1 / High (safety-critical — both safety modules silently non-functional)
**Status:** VERIFIED (2026-03-29, Worker-1)
**Affects:** `gain_integrity.rs` (entire module), `watchdog.rs` (pre-fix), `main.rs:190-194,613`
**Found by:** Architect + Audio Engineer (2026-03-29, during US-044 review)
**Related story:** US-044 (Safety Protection Suite), AC #4 (watchdog), AC #5 (gain integrity)

### Description

The US-044 safety modules (watchdog T-044-4, gain integrity T-044-5) were
implemented against a mental model where the 4 gain stages (`gain_left_hp`,
`gain_right_hp`, `gain_sub1_lp`, `gain_sub2_lp`) are separate top-level
PipeWire nodes visible in `pw-dump` output. In reality, these are `linear`
builtin Mult parameters **internal to the `pi4audio-convolver` filter-chain
node**. They are not separate `PipeWire:Interface:Node` objects — they are
accessed as params on the convolver node via `pw-cli set-param` or
`pw_node_set_param()`.

**Watchdog (T-044-4):** Originally checked for 6 "nodes" including the 4 gain
names. On real hardware, `pw-dump` only returns the 2 actual PW nodes
(`pi4audio-convolver`, `pi4audio-convolver-out`), so the watchdog always
detected 4 "missing nodes" and would trigger on every check cycle. Depending
on latch state, this either produced continuous false alarms or, if the
watchdog auto-latched, prevented legitimate alarm clearing.

**Gain integrity (T-044-5):** The `parse_pw_dump_gains()` function at
`gain_integrity.rs:207` iterates pw-dump JSON looking for top-level Node
objects with `node.name` matching the gain names (line 230-237). Since the
gain builtins never appear as top-level nodes, the parser always returns an
empty vec, causing `GainIntegrityCheck::check()` to always return
`MissingNodes` (line 91-101). The Mult <= 1.0 safety check is **never
actually evaluated** on real hardware.

Both modules pass their unit tests because the tests construct synthetic
pw-dump JSON with the gain names as top-level nodes — matching the incorrect
mental model rather than real PW behavior.

### Impact

- **Watchdog:** Cannot correctly detect convolver disappearance — false
  positives mask real failures. Safety muting unreliable.
- **Gain integrity:** Mult > 1.0 violations (accidental amplification) are
  never detected. The D-029 gain staging safety net is absent at runtime.
- **Combined:** Both US-044 safety modules are non-functional on real
  hardware. The 54 passing tests provide false confidence.
- **Safety-critical:** These modules exist to prevent driver damage from
  excessive gain or pipeline failures. Their silent non-functionality defeats
  the safety intent of US-044.

### Root Cause

The gain builtins (`linear` type, Mult param) are created by the PipeWire
filter-chain module as internal processing elements within the convolver
node's DSP graph. They share the convolver's `PipeWire:Interface:Node`
identity and are accessed via the node's param interface, not as independent
graph objects. The implementation incorrectly assumed each builtin would
appear as a separate node in `pw-dump` output.

### Fix

**Watchdog (partially applied):**
- `MONITORED_NODES` now tracks only the 2 real PW nodes (`pi4audio-convolver`,
  `pi4audio-convolver-out`) — node-level monitoring is correct.
- `GAIN_PARAM_NAMES` replaces `GAIN_NODE_NAMES` — acknowledges these are
  params, not nodes.
- Safety mute action changed to `pw_node_set_param()` on the convolver node
  with prefixed param names.

**Gain integrity (applied):**
- `gain_integrity.rs:24` imports `GAIN_PARAM_NAMES` (renamed from `GAIN_NODE_NAMES`).
- `parse_pw_dump_gains()` (line 220-255) locates the convolver node by
  `node.name == "pi4audio-convolver"` using `CONVOLVER_NODE_NAME` constant,
  then extracts Mult values from `info.params.Props[].params` flat array.
- `extract_gain_params_from_convolver()` (line 262-302) parses the alternating
  key-value array for `<gain_name>:Mult` entries matching `GAIN_PARAM_NAMES`.
- `main.rs:194` references `watchdog::GAIN_PARAM_NAMES` (correct).
- Tests use realistic pw-dump JSON with the convolver as a single node
  containing internal gain params (not synthetic top-level gain nodes).

### Verification (2026-03-29, Worker-1)

**Test command:** `nix run .#test-graph-manager` (after clearing stale cargo
cache: `rm -rf ~/.cargo-target/pi4audio-gm`)

**Result:** 248 tests passed, 0 failed, 0 ignored.

**Key gain_integrity tests (all pass):**
- `parse_pw_dump_gains_convolver_with_gain_params` — realistic pw-dump JSON
  with convolver node and 4 prefixed Mult params in Props flat array
- `parse_pw_dump_gains_no_convolver_returns_empty` — graceful empty vec when
  convolver absent (watchdog handles missing nodes)
- `parse_pw_dump_gains_ignores_non_gain_prefixed_params` — skips non-gain
  entries like `convolver_left:blocksize` in the same params array
- `parse_pw_dump_gains_convolver_without_gain_params` — convolver present but
  no gain params returns empty
- `all_ok_with_production_values` — production Mult values (0.001/0.000631)
  pass the <= 1.0 check
- `violation_detected_when_mult_exceeds_one` — Mult > 1.0 correctly flagged

**Stale cache note:** Initial test run used old compiled binary (245 tests,
old test names like `parse_pw_dump_gains_with_gain_nodes`). Clearing
`~/.cargo-target/pi4audio-gm` and rebuilding from scratch confirmed 248
tests with correct names. The cargo incremental build cache was not
detecting source changes because the old binary's fingerprint hash
(`de799ad70e0a0609`) was still considered valid.
