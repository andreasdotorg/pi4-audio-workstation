## F-218: E2E test_capture_spectrum.py — 6 WebSocket timeout failures (OPEN)

**Filed:** 2026-03-30 (session 5)
**Severity:** Medium (test infrastructure — not production code)
**Status:** Open
**Affects:** `src/web-ui/tests/integration/test_capture_spectrum.py`
**Found by:** QE (session 5 baseline audit)

### Description

Six tests in `test_capture_spectrum.py` fail with WebSocket timeout waiting
for `/ws/pcm` streaming status. The mock test server does not provide a
WebSocket endpoint for `/ws/pcm`, so tests that depend on capture spectrum
data time out waiting for a connection that will never succeed.

### Root Cause

The test server's mock mode has no pcm-bridge WebSocket endpoint. Tests
expecting `/ws/pcm` streaming data hang until the WebSocket connection
timeout expires, then fail.

This is DISTINCT from F-217 (which addresses `/ws/pcm` 403 errors in
teardown console error assertions). F-217 fixes teardown noise; F-218
is about the 6 tests that functionally depend on pcm-bridge streaming
data that doesn't exist in mock mode.

### Impact

- 6 of 7 E2E test failures in session 5 baseline
- Present since session 4 (environment gap, not a regression)
- Blocks clean E2E test suite pass
- Does not affect production functionality

### Fix

Add a mock pcm-bridge WebSocket endpoint to the test server that provides
minimal simulated PCM streaming data, sufficient for the capture spectrum
tests to exercise the UI rendering path without a real pcm-bridge process.

### Related

F-217 (conftest `/ws/pcm` 403 filter — different symptom, same missing
endpoint), US-089 (blocked by test failures), US-111 (local-demo redesign
may provide real pcm-bridge in test env)
