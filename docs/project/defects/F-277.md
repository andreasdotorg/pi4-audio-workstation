## F-277: Thermal status should use WebSocket instead of HTTP polling

**Filed:** 2026-04-06
**Severity:** Low
**Status:** OPEN
**Affects:** Browser JS (`GET /api/v1/thermal/status`), web UI dashboard
**Found by:** Owner (Gabriela Bogk, 2026-04-06)
**Related:** US-092 (thermal monitoring)

### Description

The browser fetches thermal status via HTTP polling (`GET /api/v1/thermal/status`
every ~2 seconds). All other real-time data feeds in the web UI use WebSockets:
levels, pcm, status, siggen. Thermal status should be consistent with this
pattern and use a WebSocket instead of HTTP polling.

The 2-second update interval is fine -- the issue is the transport mechanism,
not the frequency.

### Fix Needed

Replace HTTP polling of `/api/v1/thermal/status` with a WebSocket feed
(e.g., `/ws/thermal`), consistent with the existing real-time data feeds.
Server pushes thermal updates at the current ~2s interval over the WebSocket.

### Verification

1. Dashboard thermal display updates via WebSocket (no HTTP polling visible
   in browser network tab for thermal data)
2. Thermal update rate remains ~2 seconds
3. Server logs no longer show repeated `GET /api/v1/thermal/status` requests
4. Consistent with other real-time feeds (levels, pcm, status, siggen)
