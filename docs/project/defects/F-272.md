## F-272: GM reconciler settlement relies on sleep-based polling

**Filed:** 2026-04-06
**Severity:** Medium
**Status:** RESOLVED (core mechanism)
**Affects:** Mode-switching operations, E2E test stability
**Related:** US-140 (deterministic settlement), F-262 (E2E CI failures),
F-263 (rapid mode switch link count drops)
**Implementing PR:** #27

### Description

All six call sites that perform GM mode switches relied on `time.sleep()`
and polling loops to wait for the reconciler to stabilize the PipeWire link
topology. This caused:

- Flaky E2E test failures when the reconciler took longer than the hardcoded
  sleep duration (F-262)
- Temporary link count drops during rapid DJ-to-live switches that the
  polling loop missed (F-263)
- Non-deterministic test timing that varied with system load

### Root Cause

The GM `set_mode` RPC returned immediately after requesting a mode change,
before the reconciler had finished creating/destroying links. Callers had
no way to know when the reconciler was done, so they used fixed-duration
sleeps or poll-and-retry loops.

### Fix

PR #27 (US-140) adds epoch-based `await_settled` to the GM:

- The server tracks `reconcile_epoch` (incremented each pass) and
  `settled_epoch` (set when desired == actual link counts stabilize)
- `set_mode` returns the current epoch
- `await_settled` blocks until `settled_epoch >= since_epoch` or timeout
- Python `FilterChainCollector.notify_mode_change()` and
  `invalidate_pw_dump_cache()` push confirmed mode into caches immediately

4 of 6 call sites converted. Site 5 (unmute) and Site 6 (web UI JS) remain
on sleep-based waits and are tracked as follow-up work in US-140.

### Verification

- T1: 278 Rust tests pass
- T2: 899 Python tests pass
- E2E mode-switch tests no longer use `MODE_SWITCH_SETTLE_S` sleeps
- Topology API returns correct mode immediately after `set_mode` + settlement
