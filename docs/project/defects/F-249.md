## F-249: GM does not set force-quantum on initial startup (quantum test failures)

**Filed:** 2026-04-03
**Severity:** Medium
**Status:** FIXED
**Affects:** E2E quantum browser tests (`src/web-ui/tests/e2e/test_production.py::TestQuantumOnModeSwitch`)
**Related:** F-230 (quantum per mode), local-demo PipeWire config

### Description

Four E2E tests fail because the browser status bar (`#sb-quantum`) shows "1024"
instead of "256" in standby/live modes. The mock-mode path was also broken
(fixed separately: `_mock_quantum` init and mode-switch sync).

### Root Cause

`set_quantum_for_mode()` (`src/graph-manager/src/main.rs:932`) is only called
during mode **transitions** (line 435), not during initial startup. When the GM
starts with `--mode standby`:

1. The initial mode is set to `standby` but `set_quantum_for_mode(standby)` is
   never called.
2. The `_ensure_standby` test fixture calls `_set_mode(standby)` via the
   test-tool API, but the `/restore-mode` endpoint detects `current == target`
   and returns without calling `_gm_set_mode` ŌĆö so `set_quantum_for_mode`
   never fires.
3. PipeWire's runtime metadata `clock.quantum` reports 1024 in the local-demo
   environment (likely the negotiated quantum from `default.clock.max-quantum`).
   The `clock.force-quantum` config default (256) is set in the PW config but
   may not propagate to the metadata key that the GM parses, or it is cleared
   to 0 by PW internals.
4. The PipeWireCollector computes `effective_quantum = force_q if force_q > 0
   else base_q`. With `force_q=0` and `base_q=1024`, effective is 1024.

### Failing Tests

- `test_standby_quantum_256` ŌĆö expects "256", gets "1024"
- `test_live_quantum_256` ŌĆö expects "256" after live switch, gets "1024"
- `test_dj_then_live_quantum_changes` ŌĆö DJ "1024" passes, live "256" fails
- `test_dj_then_standby_quantum_changes` ŌĆö DJ "1024" passes, standby "256" fails

`test_dj_quantum_1024` passes because 1024 happens to be the stale value.

### Fix

Two changes applied:

1. **Startup call (already on main, US-123):** `set_quantum_for_mode(initial_mode)`
   called at GM startup (line 238) before the PW main loop starts.

2. **Explicit 256 for non-DJ modes (this fix):** Changed non-DJ branch from
   `_ => 0` (clear force-quantum) to `_ => 256` (explicit). GM is now the
   single authority for quantum in all modes ŌĆö no reliance on PW config
   defaults or negotiation behavior. Quantum map:
   - DJ: force-quantum=1024 (~21ms PA path)
   - Live/Standby/Measurement: force-quantum=256 (~5.3ms PA path)

   This eliminates the root cause: PW negotiating quantum=1024 when
   force-quantum=0, regardless of the config file's `force-quantum=256`.

### Key Files

- `src/graph-manager/src/main.rs` ŌĆö `set_quantum_for_mode()` (line 932),
  mode transition handler (line 435)
- `src/web-ui/app/collectors/pipewire_collector.py` ŌĆö effective quantum logic
- `scripts/local-demo.sh` ŌĆö PW config with `default.clock.force-quantum = 256`
- `src/web-ui/tests/e2e/test_production.py` ŌĆö `TestQuantumOnModeSwitch` class
