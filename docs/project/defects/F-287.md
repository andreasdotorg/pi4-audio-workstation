## F-287: Local-demo capture-usb pcm-bridge violates mock boundary (D-057)

**Severity:** high
**Status:** open
**Found in:** `scripts/local-demo.sh`, `src/web-ui/static/js/test.js`
**Affects:** D-057 addendum (mock boundary), testing-principles.md Principle 2
**Found by:** owner (session 14 review)

### Description

Local-demo's `capture-usb` pcm-bridge instance is configured differently from
production, violating the D-057 mock boundary rule ("only physical hardware may
be mocked; everything else must be real").

**Production** (`nix/nixos/production.nix:91-99`):
```
capture-usb = {
  mode = "capture";
  target = "alsa_input.usb-MiniDSP_USBStreamer-00.pro-input-0";
  channels = 8;
  ...
};
```
Reads from the USBStreamer ALSA input — 8 channels of ADA8200 ADC (vocal mic,
spare, etc.).

**Local-demo** (`scripts/local-demo.sh:591-597`):
```
--mode capture
--node-name alsa_input.usb-miniDSP_Umik-1
--channels 1
```
Reads from the UMIK-1 node — 1 channel, a completely different USB device.

This is not a hardware substitution (permitted by D-057). It changes the
*software configuration* of a real component (pcm-bridge) to point at a
different source with a different channel count. The `capture-usb` instance in
local-demo represents a fundamentally different thing than in production.

### Secondary violation: virtual `umik1` source in JS

`src/web-ui/static/js/test.js` contains a virtual `umik1` source that extracts
channel 3 from the `monitor` pcm-bridge stream. This source:
- Does not exist in production
- Is application-level logic that only exists in the test/UI code
- Was introduced to work around the `capture-usb` misconfiguration
- Caused the F-286 duplicate dropdown bug

The F-286 fix (suppressing the duplicate) papers over the mock boundary
violation instead of fixing the root cause.

### Root cause

The mock boundary principle (D-057 addendum) was documented in a decisions
file that workers don't read during implementation. The principle was not
surfaced in any file that workers or reviewers encounter during normal
development. This has been addressed by the creation of
`docs/project/testing-principles.md` and references from CLAUDE.md, worker
prompt, QE prompt, and development HOWTO.

### Fix required

1. **Local-demo `capture-usb`** must match production: read from the simulated
   USBStreamer input node (the null-source adapter), 8 channels. The only
   difference from production is the PW node is a null-source instead of real
   ALSA hardware — that's a permitted hardware substitution.

2. **Virtual `umik1` source in test.js** must be removed. If measurement needs
   UMIK-1 PCM data in the UI, it should arrive through the same path it uses
   in production — not through a channel-extraction hack that only exists in
   local-demo.

3. **Verify the production UMIK-1 data path** actually works. Given the state
   of local-demo, it may be broken on the real Pi too.
