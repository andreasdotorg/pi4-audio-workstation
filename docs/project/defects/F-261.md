## F-261: CM protocol gap — destructive operations can lose uncommitted changes

**Filed:** 2026-04-04
**Severity:** Medium
**Status:** OPEN
**Affects:** CM protocol, `.claude/team/roles/change-manager.md`
**Origin:** Session incident — uncommitted kernel-rt.nix changes lost during destructive operation

### Description

The CM protocol does not require stashing or committing uncommitted
changes before destructive operations (`git checkout`, `git reset`,
worktree cleanup). During this session, uncommitted changes to
`nix/nixos/kernel-rt.nix` were lost when a destructive operation
was performed without preserving the working tree state.

### Root Cause

The CM prompt does not include a mandatory pre-destructive-operation
checklist. The CM has no protocol obligation to check for uncommitted
changes before running commands that modify the working tree.

### Fix

Add to the CM prompt (`.claude/team/roles/change-manager.md`):

**Before any destructive operation (checkout, reset, worktree remove,
branch delete with unmerged changes), the CM MUST:**

1. Run `git status` to check for uncommitted changes.
2. If uncommitted changes exist: `git stash push -m "<context>"` and
   report to the orchestrator what was stashed.
3. Only then proceed with the destructive operation.
4. After the operation, report the stash reference so it can be
   applied or reviewed later.

This is a process improvement for the CM prompt rewrite (worker-4
assignment).
