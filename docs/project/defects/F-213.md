## F-213: Driver protection doesn't engage after profile ACTIVATE (2026-03-29)

**Severity:** Medium
**Status:** Open
**Found in:** `speaker_routes.py:_activate_profile_impl()` -> `_configure_thermal_protection()` integration
**Affects:** US-092 (Per-driver thermal/mechanical protection), activation flow
**Found by:** QE exploratory testing (Phase 2, US-092 state sync test)

**Description:**

After activating a speaker profile via POST `/api/v1/speakers/profiles/{name}/activate`,
the thermal protection subsystem (thermal monitor + thermal limiter) does not receive
the activation. The `_configure_thermal_protection()` function at line 1082 of
`_activate_profile_impl()` is called, but the protection subsystem does not reflect the
activated profile's thermal parameters.

Observed during local-demo exploratory testing: after activating a profile, the thermal
status endpoint returns no channel data, and the thermal limiter has no configured
channels. The protection subsystem remains in its initial unconfigured state.

**Expected:** After profile activation, `thermal_monitor.configure_from_ceilings()` and
`thermal_limiter.configure_channels()` should populate per-channel thermal state from
the activated profile's speaker identities (pe_max_watts, impedance, sensitivity).

**Actual:** Protection subsystem has zero configured channels after activation. The
`_configure_thermal_protection()` call either silently fails (e.g. `thermal_ceiling`
import path issue in local-demo, or missing identity files) or the monitor/limiter
objects are None in the app state.

**Root cause:** Investigation needed. Likely causes:
1. In local-demo mode, the `thermal_ceiling` module import may fail silently (line 910:
   `ImportError` caught and logged as warning, then returns without configuring)
2. The `thermal_monitor` or `thermal_limiter` may not be initialized on `app.state`
   in local-demo startup
3. Speaker identity files referenced by the profile may not exist in the local-demo
   data directory

**Note:** This is distinct from F-209 (watchdog/gain integrity node model bug). F-209
is about the Rust safety modules; this defect is about the Python thermal protection
subsystem not receiving profile data during activation.

**Fix:** TBD — needs investigation to determine which of the three likely causes applies.

**Verified:** Not yet.
