## F-220: Local-demo PW reload kills all clients — no reconnect (OPEN — root cause: F-221)

**Filed:** 2026-03-30 (session 5)
**Severity:** Medium (development workflow — blocks local-demo E2E filter reload testing)
**Status:** Open — **root cause identified as F-221** (filter deploy panel uses full PW restart instead of convolver node reload). Fixing F-221 makes F-220 moot.
**Affects:** `local-demo.sh`, `reload_pipewire()`, GraphManager, signal-gen, pcm-bridge
**Found by:** Owner (session 5, during US-111 AC #9 verification)

### Description

After `reload_pipewire()` runs `local-pw-test-env.sh stop && start` to
reload PipeWire with new filter-chain configuration, all PW client
processes (GraphManager, signal-gen, pcm-bridge) die and do not
reconnect. The PipeWire server restarts successfully, but the dependent
services that were connected to the previous PW instance are terminated
and not re-launched.

This means filter switching (e.g., applying new FIR coefficients via
config reload) cannot be tested end-to-end in the local-demo environment.
The operator must manually restart all services after a PW reload.

### Root Cause

`local-pw-test-env.sh stop` kills the PipeWire process, which disconnects
all PW clients. When PW restarts via `start`, the clients (GM, signal-gen,
pcm-bridge) are already dead — they were spawned by local-demo.sh as child
processes of the original PW session and have no reconnection logic or
supervisor to restart them.

### Impact

- Blocks E2E testing of filter reload/switching in local-demo (US-111 AC #9)
- Operator must manually restart GM + signal-gen + pcm-bridge after every
  PW reload
- Limits the usefulness of local-demo for iterative filter development

### Fix Options

1. **Service supervisor in local-demo.sh** — after PW reload, re-launch
   GM, signal-gen, and pcm-bridge automatically
2. **PW client reconnection logic** — services detect PW disconnect and
   reconnect when PW returns (more robust, but larger change)
3. **Avoid full PW restart** — use `pw-cli` to reload filter-chain config
   without restarting the PW daemon (may not be possible for all config
   changes)

### Resolution Path

F-221 is the root cause. Once the filter deploy panel switches from
`reload_pipewire()` to `_reload_convolver()` (pw-cli destroy), there is
no full PW restart and therefore no killed clients. F-220 closes when
F-221 is fixed.

Long-term: D-061 (GM manages PW lifecycle) eliminates the entire class
of problems by giving GM coordinated control over PW restarts.
