## F-221: Filter deploy panel uses full PW restart instead of convolver node reload (FIXED)

**Filed:** 2026-03-30 (session 5)
**Severity:** Medium (unnecessary destructive reload — causes audio dropout + kills clients)
**Status:** FIXED (session 5)
**Affects:** `src/web-ui/app/filter_routes.py` (`POST /api/v1/filters/reload-pw`),
`src/room-correction/room_correction/deploy.py` (`reload_pipewire()`)
**Found by:** Owner (session 5)

### Description

The filter deploy panel's "Reload PipeWire" endpoint (`POST
/api/v1/filters/reload-pw`) calls `reload_pipewire()` which does a full
PipeWire restart (`systemctl --user restart pipewire.service` in
production, `local-pw-test-env.sh stop && start` in local-demo). This:

1. Kills ALL PipeWire clients (GraphManager, signal-gen, pcm-bridge,
   level-bridge, Mixxx, Reaper)
2. Resets the USBStreamer, producing transients through the amplifier chain
   (speaker damage risk — requires amps-off safety gate)
3. In local-demo, clients don't reconnect (F-220)

Meanwhile, the measurement session's deploy phase uses
`_reload_convolver()` (`session.py:1280`) which does `pw-cli destroy
<node>` — PipeWire auto-recreates the node from updated `.conf.d/`
drop-ins. This approach:

1. Only affects the convolver node — all other clients stay connected
2. No USBStreamer reset — no transient risk (node disappears cleanly)
3. Brief audio gap while GM re-links (~1-2 seconds)
4. Works in both production and local-demo

### Root Cause

Historical: `reload_pipewire()` was written before the `pw-cli destroy`
approach was discovered. The filter deploy panel was never updated to use
the better mechanism.

### Fix

Replace the `reload_pipewire()` call in `filter_routes.py:reload_pw_endpoint()`
with the `_reload_convolver()` approach:

1. Extract `_reload_convolver()` from `measurement/session.py` into a
   shared utility (e.g., `deploy.py` or `pw_helpers.py`)
2. Update `reload_pw_endpoint()` to call the extracted function
3. Remove or deprecate `reload_pipewire()` — it should never be used
4. Update the safety confirmation message — "Reload" no longer resets
   the USBStreamer, so the scary transient warning can be softened to
   "Brief audio gap during filter reload"
5. Remove `PI4AUDIO_PW_RELOAD_CMD` from `local-demo.sh` (no longer needed)

### Impact of Fix

- F-220 becomes moot (no full PW restart = no killed clients)
- Eliminates the amps-off safety gate for filter reload (no transient risk)
- Filter switching becomes testable E2E in local-demo
- Simpler, safer, consistent with measurement session behavior
